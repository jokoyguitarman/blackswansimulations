Engineering Plan · Social Media Crisis · Trainer Debrief

Social Session Review Mode

Give a concluded social-media session its own trainer review screen by reusing the live TrainerSimDashboard in a read-only "review mode" - instead of dropping the trainer onto the generic field-ops After-Action screen that was built for bombing/evacuation exercises.

Scope confirmed: reuse the existing dashboard (no separate AAR component); trainer-only - the participant device experience is unchanged.

On this page

1. Why this is needed 2. Root cause (one condition) 3. Target flow 4. What the output looks like 5. The changes 6. File-by-file index 7. Verification 8. Out of scope

1. Why this is needed

There are two completely separate trainer surfaces in the app:

SurfaceRouteComponentDatasets
Live social-media backend view /sim/:id/trainer TrainerSimDashboard.tsx Sentiment gauges, crisis-comms standards, graded replies, consequences, orchestration injects, disputes, SOP/action tiers
Field-ops session + generic AAR /sessions/:id SessionView.tsx Incidents, map elements, team assignments, impact matrix, escalation log

When a social session is concluded, the trainer lands on the field-ops AAR - which carries the wrong datasets and measures entirely. The fix is to keep them on the dashboard they were already using during the exercise, frozen for joint review of injects, player responses, and actions taken.

2. Root cause (one condition)

The redirect in SessionView.tsx only sends a social session to the dashboard while it is in progress. Once status flips to completed, the redirect stops firing and the trainer is left on the field-ops view.

useEffect(() => {
  if (
    session &&
    (session as ...).sim_mode === 'social_media' &&
    session.status === 'in_progress' &&   // <-- only while live
    id
  ) {
    if (isTrainer) navigate(`/sim/${id}/trainer`, { replace: true });
    else navigate(`/sim/${id}/device`, { replace: true });
  }
}, [session, id, navigate, isTrainer]);

The dashboard's "Conclude Session" button also explicitly navigates to /sessions/:id after marking the session complete, compounding the same fall-through.

Why this is low-risk: every panel in the dashboard is fed by persistent, DB-backed loaders (loadSocialState, loadPosts, loadGradedReplies, loadConsequences, loadOrchestration, loadDisputes). The data already survives the session ending - the only "live" pieces are the 12s polling, the LIVE badge, the ticking timer, and the Conclude button.

3. Target flow

Trainer clicks Conclude → PATCH status = completed → Stay on /sim/:id/trainer (review mode)
Trainer opens a completed social session from /sessions → Redirect to /sim/:id/trainer → Review mode

Field-ops sessions and the participant device flow are untouched.

5. The changes

1Route completed social sessions to the dashboard

In frontend/src/pages/SessionView.tsx, broaden the redirect so a trainer viewing a social_media session is sent to /sim/:id/trainer when status is in_progress or completed. The player branch stays gated to in_progress only.

const isSocial = (session as ...).sim_mode === 'social_media';
const reviewable = session.status === 'in_progress' || session.status === 'completed';
if (session && isSocial && id) {
  if (isTrainer && reviewable) navigate(`/sim/${id}/trainer`, { replace: true });
  else if (!isTrainer && session.status === 'in_progress') navigate(`/sim/${id}/device`, { replace: true });
}

This also covers the dashboard's own Conclude button (it can simply stay on the page) and the handleCompleteSession path in SessionView, which refreshes status to completed.

2Add an isReview mode to the dashboard

In frontend/src/components/SimDevice/TrainerSimDashboard.tsx:

ConcernLiveReview (isReview = sessionInfo?.status === 'completed')
PollingsetInterval(loadAll, 12000)Load once; skip the interval
Elapsed timer30s minutesAgo(now)Freeze at final duration (start → completed)
WebSocketSubscribedDisabled (no events post-session)
Status badgeLIVECOMPLETED · REVIEW
Conclude buttonShownHidden

3Re-frame live-only panels

The "Unattended Posts" panel is a live decision-prompt (derived from unattendedPosts; harmful unhandled posts get animate-pulse + red border via postBorderStyle). In review mode, relabel to "Posts Left Unaddressed" and drop the pulse/alert styling so it reads as a debrief statistic. All scoring/debrief panels (gauges, standards, graded replies, consequences, OrchestrationTicker, disputes, SOP, tiers) stay as-is.

4Verify completed-session data availability

Confirm the loaders return rows for a completed session (they query persisted tables, so expected to work): social/state, social/posts, social/replies, social/consequences, social/orchestration, social/disputes. The orchestration route in server/routes/socialMedia.ts already accepts any status. No backend changes anticipated - flag any endpoint that filters on in_progress.

7. Verification

  • Conclude a social session from the dashboard → trainer remains in review mode (no polling, COMPLETED badge, no Conclude button, frozen timer).
  • Open a previously completed social session from /sessions → trainer is redirected into review mode.
  • Open a completed field-ops session → still shows the existing SessionView AAR (no regression).
  • Open an in-progress social session → unchanged live behavior.

8. Out of scope

  • Participant / device post-session experience (trainer-only by request).
  • A separate dedicated AAR component or new metrics datasets.
  • Field-ops SessionView AAR behavior (unchanged for non-social sessions).
