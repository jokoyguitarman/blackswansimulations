Context

The Problem

The SpatialEffectsEngine already computes per-stud fire/gas/smoke spread at time intervals and stores it as environmentalTimeline in rts_scene_configs.enrichment_result. But this data is never consumed by the compile/persist pipeline. Live session deterioration spawns hazard pins at random zone positions, completely disconnected from the physics simulation. There is also a format mismatch bug where trainer hazards receive an events[] array instead of the expected at_10min / at_20min / at_30min object.

BUG
Format Mismatch
Trainer hazards get events[] into deterioration_timeline, but runtime expects at_10min/at_20min/at_30min. Session deterioration silently does nothing.
GAP
Timeline Not Consumed
environmentalTimeline has per-stud fire data at 10+ time intervals, but compile never reads it. Only the frontend preview uses it.
GAP
Zone-Based Spawning
Live deterioration spawns new hazard pins via placeInsideZoneType('hot') -- random point in hot zone, no stud or fire-spread awareness.
Architecture

Solution Data Flow

At compile time, read the pre-computed fire spread timeline, cluster affected studs into hazard groups, and create stud-snapped delayed spawn pins. At runtime, the existing deterioration service reveals them automatically.

2
New Function: generateFireSpreadSpawnPins()
warroomService.ts
▶

New function in warroomService.ts that reads the pre-computed environmental timeline and converts stud-level fire spread into gameplay-ready spawn pins.

Function Signature

ts function generateFireSpreadSpawnPins( environmentalTimeline: EnvironmentalSnapshot[], studGrids: StudGrid[], existingHazards: Array<{ label: string; location_lat: number; location_lng: number; hazard_type: string }>, buildingPolygon: [number, number][], ): Array<Record<string, unknown>>

Algorithm

  1. Diff snapshots -- For each snapshot at time T, find studs newly on fire (fire_intensity ≥ 0.3) that were NOT on fire in the previous snapshot
  2. Cluster -- Group nearby newly-affected studs using 15m spatial proximity to avoid creating one pin per stud
  3. Pick representative -- For each cluster, select the stud with highest fire_intensity as the pin location
  4. Resolve coordinates -- Look up the stud ID in studGrids (spacing=5 to match enrichment) to get { lat, lng }
  5. Find parent hazard -- Match each cluster to the nearest existing hazard by distance to set parent_pin_label
  6. Emit spawn pin -- Return pin with status: 'delayed', spawn_condition, and stud metadata

Output Pin Shape

ts { hazard_type: 'fire', // or parent hazard's type label: 'Fire spread — east corridor T+10min', location_lat: 1.340521, // from stud grid lookup location_lng: 103.749832, floor_level: 'G', status: 'delayed', appears_at_minutes: 10, properties: { spawned_from_stud: 'bldg-0-G-12-8', fire_intensity: 1.0, cluster_stud_count: 7, description: 'Fire has spread to 7 studs in the east corridor', }, _parent_pin_label: 'Chemical storage fire', _spawn_condition: { trigger: 'parent_unresolved', at_minutes: 10, unless_status: ['contained', 'resolved'], }, }

Thresholds and Guards

⏱ Snapshots: 5, 10, 15, 20, 30 min
🔥 Fire threshold: intensity ≥ 0.3
🌏 Cluster min: 3+ studs
📋 Max pins: ~12 total
📏 Cluster radius: 15m proximity
