Payment Portal: Every Moving Piece

Trainers sign up free, enroll client organisations, and invoice them a fixed S$10,000 through Stripe. Payment converts into 1 scenario credit + 2 session credits (two games per training — e.g. a pre-test and a post-test on the same scenario). Producing the AAR after the training flips the trainer's 30% payout (S$3,000) to "pending release", which an admin approves before Stripe wires the money to the trainer's connected account.

Companion to the confirmed plan file payment_portal_implementation.plan.md. Every backend hook cites the exact file and line it touches. Business rules confirmed: 30% trainer / 70% Black Swan · 1 scenario + 2 session credits per invoice · admin-released payouts · ADMIN accounts keep full unlimited access exactly as today.

Where each S$10,000 goes

S$3,000 — trainer (30%), after AAR + approval

The full S$10,000 lands in the Black Swan platform Stripe account first. The trainer's share sits in the platform balance until the admin releases it — there is never a moment where the money is outside our control.

2. The five actors

Trainer

Signs up free (new: self-serve trainer accounts). Enrolls clients, clicks Generate Invoice, spends credits in the Warroom and on session creation, produces the AAR, receives the S$3,000 payout. Must complete a one-time Stripe Connect onboarding (identity + bank details) before any payout can be released.

Client organisation

The company being trained. Never needs a Black Swan account. Receives an email from Stripe with a hosted invoice page, pays by card or bank transfer. That's their entire interaction with the system.

Black Swan server

Creates Stripe customers/invoices, listens to webhooks, keeps the credit ledger, enforces the gates on Warroom/session/AAR endpoints, and executes the transfer when an admin approves. All billing logic is server-side; the frontend can never mint credits or move money.

Stripe

Three products: Invoicing (client-facing payment), Connect Express (trainer KYC + bank account), Transfers (the payout). Stripe also emails the invoice and hosts the payment page, so we build no payment UI for clients.

Admin (Black Swan staff)

Reviews payouts in a new /admin/payouts page. Sees which trainer, which client, which session, when the AAR was produced. One click to Release (executes the Stripe transfer) or Hold (freezes it with a reason, e.g. suspected abuse or client dispute). This human step is the anti-gaming valve for the AAR trigger.

3. The full journey, step by step

  1. 1Trainer signs up free. New "I'm a trainer" option on the signup page. After account creation the app calls POST /api/profile/become-trainer, which upgrades the account from participant to trainer server-side. (See section 11 for why this is now safe.)
  2. 2Trainer enrolls a client organisation on the new /clients page: name, contact person, contact email. Stored in the new client_organisations table. Each org gets a small profile card.
  3. 3Trainer clicks GENERATE INVOICE on the org card. The server creates a Stripe Customer for the org (first time only), then a Stripe Invoice for S$10,000 with collection_method: send_invoice and 30 days to pay, finalizes it, and Stripe emails the client a hosted payment page. We store the invoice row with status sent and the hosted URL (trainer can also copy the link and send it themselves).
  4. 4Client opens the email, clicks pay. Card or bank transfer, on Stripe's page. Nothing to build on our side.
  5. 5Stripe calls our webhook with invoice.paid. The server marks the invoice paid, writes +1 scenario credit and +2 session credits into the ledger, and creates a payout row for S$3,000 in state awaiting_completion. The trainer gets an in-app notification: "Client payment received — credits added."
  6. 6Trainer builds the scenario in the Warroom. Entering the wizard requires a scenario balance of at least 1 (this also blocks free accounts from burning our OpenAI spend on the research steps). The credit is only consumed at the final "Compile scenario" click. If compilation fails, the credit is refunded automatically.
  7. 7Trainer creates the session as usual (invite players, teams, briefing — all unchanged). With 2 session credits per invoice the trainer can create two sessions per engagement — e.g. a pre-training test game and a post-training test game. Each session creation consumes one credit and stamps the session with funding_invoice_id — the permanent link between this training and the client's payment.
  8. 8Training day runs exactly as today. No billing logic touches live gameplay.
  9. 9Trainer clicks Produce AAR. After the report generates successfully, the server follows session.funding_invoice_id to the payout row and flips it awaiting_completion → pending_release. Admins are notified. Regenerating the AAR later does nothing (the flip is one-time by design).
  10. 10Admin reviews and releases. On /admin/payouts, one click executes a Stripe Transfer of S$3,000 from the platform balance to the trainer's connected account. Row becomes released with the Stripe transfer id stored for the audit trail. Trainer is notified.

4. The three Stripe products used

ProductWhat it does for usWhat we store
Stripe Invoicing Creates and emails the S$10,000 invoice; hosts the payment page; handles card/bank payment, receipts, reminders. The client never needs an account with us. stripe_customer_id on the org, stripe_invoice_id + hosted_invoice_url on the invoice row.
Stripe Connect (Express) Turns each trainer into a "connected account". Stripe runs the KYC (identity verification) and collects bank details through its own hosted onboarding flow — we just generate the onboarding link. stripe_connect_account_id + onboarding_status in trainer_billing.
Separate charges & transfers The payment lands 100% in our platform account; the S$3,000 payout is a separate Transfer we execute whenever we choose (i.e. on admin release). This is what gives us escrow-like control without Stripe offering literal escrow. stripe_transfer_id on the payout row.

Why not Stripe's automatic split payments (destination charges)? Those push the trainer's share at payment time. Our business rule is the opposite: the trainer earns the payout only after the training concludes (AAR produced) and an admin approves. Separate transfers are the only Stripe pattern that matches.

5. The credit economy

Credits are the bridge between "client paid" and "trainer may use expensive features". They are stored as an append-only ledger, never as a mutable counter — every grant and every spend is a row, the balance is the sum.

Scenario credit — unlocks the Warroom; consumed by the final compile Session credit — consumed when a session is created
EventLedger rows writtenReason code
Invoice paid (webhook)+1 scenario +2 sessioninvoice_paid
Warroom compile succeeds−1 scenarioscenario_generated
Compile fails after consuming+1 scenario (automatic refund)admin_adjustment
Session created−1 session (records which invoice funded it)session_created
Manual correction by adminanyadmin_adjustment

Two subtleties worth knowing

Race safety. Two browser tabs both clicking "Compile" with one credit left must not both succeed. Spending goes through a Postgres function consume_credit() that takes an advisory lock per (trainer, credit type), checks the balance, and inserts the −1 row atomically. One tab wins, the other gets a clean "no credits" error.

Consume upfront, refund on failure. Scenario compilation takes 2–5 minutes and costs real OpenAI money. We consume the credit before starting (so a failing balance never triggers the expensive work) and write a +1 refund row if the compile throws. The trainer never loses a credit to our error.

Why entry to the Warroom is also gated (not just the compile)

The wizard's intermediate steps — doctrine research, geocoding validation, storyline generation, NPC generation — each call OpenAI before any credit is spent. If only the compile were gated, a free trainer account could burn unlimited AI spend without ever paying. So every generator step requires a balance ≥ 1 (admins bypass). The credit is still only consumed once, at compile.

6. Payout lifecycle (state machine)

Exactly one payout row per paid invoice (invoice_id is UNIQUE). It moves through these states:

awaiting_completion →first successful AAR on the funded session→ pending_release →admin clicks Release→ released
pending_release ↔admin Hold / Unhold (with reason)↔ held
pending_release →Stripe transfer error (e.g. onboarding incomplete)→ failed →retry after fixing→ released
PropertyGuarantee
CreatedAt invoice.paid webhook time, amount = 30% of the invoice, state awaiting_completion.
AAR flip is one-timeThe update is WHERE status = 'awaiting_completion' — regenerating the AAR can never re-trigger or double-pay.
Two sessions, one payoutBoth funded sessions carry the same funding_invoice_id. The first AAR produced flips the payout to pending release; the second session's AAR is a no-op. The admin review page shows how many of the invoice's session credits were used, so the reviewer can wait for the post-training game before releasing.
Release preconditionsState must be pending_release AND the trainer's Connect onboarding must be complete. Otherwise the admin sees a clear "trainer hasn't finished payout setup" error instead of a failed transfer.
Audit trailRow stores aar_generated_at, released_at, released_by (admin id), stripe_transfer_id, and hold reasons.

Why the human step exists

The AAR click is trainer-controlled, so it is gameable in both directions: a trainer could produce an AAR after a 5-minute sham session, or never click and strand the payout. The admin review window catches the first case (session length, participants and AAR content are all visible on the review page); an admin can also manually release for the second. Card payments can additionally be disputed for months — the Hold state is the lever if a chargeback lands after the AAR.

One mounting-order landmine

Stripe signatures are computed over the raw request bytes. The app applies express.json() globally at server/index.ts:177, which would consume and re-parse the body and break verification. The webhook route must therefore be mounted with express.raw() before that line:

app.use('/api/billing/webhook', express.raw({ type: 'application/json' }), billingWebhookRouter);
// ...then the existing...
app.use(express.json({ limit: '10mb' }));

CORS already allows origin-less requests (Stripe sends none), so no CORS change is needed.

11. Trainer self-signup (a deliberate security change)

Today, migration 189_lockdown_user_profile_role.sql makes trainer accounts operator-provisioned only — signups default to participant and a DB trigger blocks self-escalation. The new business model needs trainers to sign up free, so we reopen exactly one path:

  • POST /api/profile/become-trainer upgrades participant → trainer using the service-role client (trusted context; the 189 trigger intentionally lets service-role writes through). It can never grant admin, and refuses if the account is anything but a plain participant.
  • The signup page gets an "I'm a trainer setting up trainings" option that calls this after account creation. Invitation-based participant signups are untouched.

Why this is safe now: the trainer role used to be the only thing standing between an account and cost-incurring AI endpoints. After this project, every expensive action (Warroom AI steps, scenario compile, session creation) is gated on credits, which only exist after a client actually pays S$10,000. A free trainer account with zero credits can browse its dashboard and enroll clients — nothing more.

17. Security audit: can a hacker change someone's access level?

Audited July 2026 against the two attack classes you asked about: role/access escalation via API calls and SQL injection. Verdict: the existing defenses are layered and solid; the audit found no exploitable path. The items to carry forward are hardening rules for the new billing code.

Role escalation — four independent layers, all verified in code

LayerWhereWhat it blocks
1. Signup can't pick a role migrations/189 handle_new_user() + SignUp.tsx The signup trigger ignores all client-supplied metadata for roles. Every new account defaults to participant; elevated in-session roles come only from server-created invitation rows, and both the invitation enum (sessions.ts:1463, 1671) and the DB CHECK constraint (migrations/006) exclude trainer and admin — those can never arrive via an invitation.
2. API refuses role changes server/routes/profile.ts:57 PATCH /api/profile rejects any role or agency_name change unless the caller is already an admin. Checked in code because the server's service-role client bypasses RLS.
3. Database trigger as backstop migrations/189 prevent_role_self_escalation() Even if someone bypasses our API entirely and talks to Supabase directly with the anon key (the RLS policy does let users update their own profile row), a BEFORE UPDATE trigger raises an exception on any role/agency change by a non-admin. There is also no INSERT policy on user_profiles, so a client cannot insert a fresh row with role='admin'.
4. Token metadata is trustworthy server/middleware/auth.ts:39 The server reads the role from app_metadata, which only the service-role key / Supabase dashboard can write — never from user_metadata, which end users can edit themselves. Verified: no code path anywhere reads a role out of user_metadata, and the only updateUserById calls (join.ts:170, 218) write a display name, nothing privileged.

SQL injection — no raw SQL surface exists

CheckFinding
Raw SQL stringsclean The server never concatenates SQL. Every query goes through the supabase-js query builder (.eq(), .select()…), which parameterizes via PostgREST.
Stored proceduresclean Only 4 .rpc() calls in the codebase (objectiveTrackingService.ts), all with named parameters — parameterized, not string-built.
Input validationclean Route bodies are zod-validated and replaced by the parsed result (server/lib/validation.ts), so unknown fields are stripped before handlers run.
PostgREST filter interpolationwatch Four .or(`...${id}`) template strings exist (npcEmailReplyService.ts:140, 278, injectSchedulerService.ts:548, 783). Today every interpolated value is either a zod-validated UUID (socialMedia.ts:1088) or a DB-sourced id, so none is exploitable — but the pattern is fragile. House rule going forward: any value that lands inside .or() must be UUID-validated first. The new billing code will not use this pattern at all.
Stored HTMLclean Player email HTML is stored unescaped (socialMedia.ts:1124) but the frontend renders only body_text through React's auto-escaping (EmailApp.tsx:676) — no script injection path into a trainer's browser.

Hardening rules baked into the new billing build

  • consume_credit() locked to the server: the new SQL function gets REVOKE EXECUTE ... FROM anon, authenticated so nobody can call it through Supabase's public RPC surface and spend (or probe) other people's credits. All new billing tables: RLS enabled, zero anon/authenticated policies — service-role access only, matching the documented pattern in server/lib/access.ts.
  • become-trainer is a one-way, one-step door: it upgrades participant → trainer only, refuses any other starting role, and can never grant admin. Admin remains operator-provisioned only, exactly as today.
  • Credit checks are server-side facts, not client claims: balances live in the append-only ledger; the frontend only displays them. A tampered client can render whatever it likes — the 402 comes from the server.
  • Money endpoints double-check roles in code: payout release/hold requires role === 'admin' verified from the authenticated token on every call.
  • Webhook can't be spoofed: Stripe signature verification over the raw body plus the stripe_webhook_events idempotency table means forged or replayed "invoice paid" calls grant nothing.
