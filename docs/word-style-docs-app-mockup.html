DraftPad replacement

A shared document editor built like Word.

DraftPad becomes Docs: one team-shared document library, a Word-mobile editing experience inside the simulated phone, and a ribbon-based Word-desktop experience inside Desktop Mode. Both views open the same server-persisted document.

Confirmed product rules
  • Documents are shared with the author's assigned team.
  • Review and approval are available, but never mandatory.
  • Players copy text into Social or Mail; Docs does not publish directly.
  • Autosave keeps work intact when switching between mobile and desktop.
01 · IDENTITY
Word-inspired, not copied Word-blue interaction patterns and ribbon composition, with a custom Docs icon and Black Swan framing.
02 · CONTINUITY
One document, two views Server persistence and 800ms debounced autosave make phone-to-desktop switching seamless.
03 · TEAMWORK
Shared with the team Teammates read and review; the author remains the only editor, avoiding simultaneous-edit conflicts.
04 · FREE USE
Copy, don't publish Approval is a training signal—not a technical gate. Players can copy their text at any lifecycle status.

Feature guide

What each feature does

The application is a writing and review workspace—not another publishing channel.

Persistence

Autosave and cross-view continuity

HTML and plain text save to the server 800ms after typing stops. The phone and desktop fetch the same `player_drafts` row, so neither refresh nor switching routes destroys work.

Editing

Rich text with controlled styles

Bold, italic, underline, strike, lists, alignment, Normal, Heading 1, Heading 2, and Title. A focused subset keeps generated HTML safe and social-copy output clean.

Teamwork

Team-shared document library

A document is stamped with the author's team at creation. Teammates see it in Recent, can read it, and can review it; only the original author edits it.

Workflow

Optional submit and approval

Authors can submit for review. A teammate can approve or request changes with a note. Approval is a training event and score signal, never a technical requirement to use the text.

AI support

Word-like Editor panel

The existing crisis-aware grader becomes “Editor.” It scores accuracy, tone, clarity, completeness, and team role fit, then provides specific improvement guidance.

Handoff

Copy text into Social or Mail

“Copy text” writes clean plain text to the clipboard and confirms: “Copied — paste it into a post or an email.” The editor never posts on the player's behalf.

Lifecycle

Draft status and review history

Draft, In review, Approved, and Changes requested appear consistently in Recent, the editor, and the desktop status bar. Editing an approved document resets it to Draft.

Scoring

Observable team behaviour

Creating, submitting, approving, and grading produce existing `player_actions` events. Legal receives task credit for meaningful draft review; trainers see the workflow in the ledger.

Safety

Author-only editing and sanitised HTML

The API checks session and team access for every operation. Stored HTML is allowlist-sanitised before teammates render it, preventing stored-script injection.

Visual specification

Colours, typefaces, and composition

The containing product keeps Black Swan's navy, orange, cream, and bordered-card language. Inside the simulated Docs app, Word-blue and Fluent neutrals establish the familiar document-editor model.

User workflow

Optional review without blocking real play

The review flow creates observable coordination and Legal-team scoring while preserving player freedom. At every stage, the author may copy the document and use it elsewhere.

1 · CreateDocument inherits the author's session and team. `draft_created` is recorded.
2 · WriteRich text autosaves. Phone and desktop always reopen the same server copy.
3 · GradeAI Editor checks crisis facts, tone, completeness, and team role fit.
4 · SubmitOptional. Status becomes In review and `draft_submitted_for_approval` is recorded.
5 · ReviewA teammate approves or requests changes with a note. Legal approval earns task credit.
6 · CopyAuthor copies clean text and pastes it into Social or Mail—regardless of review status.
Important behaviour: editing an approved or in-review document resets it to Draft and clears the old review result. Past action events remain in the trainer ledger, but nobody can mistakenly treat revised text as the version that was approved.
Engineering reference

How the mockup maps to the build

Database

`player_drafts`

Stores session, author, team, title, sanitised HTML, plain text, lifecycle status, reviewer metadata, grade, and timestamps.

API

`/api/drafts`

List, create, autosave, delete, submit, review, and grade. Every endpoint validates session access and author/team permissions.

Editor

TipTap core

One controlled rich-text engine, shared hooks, and two UI presentations: `EditorMobile` and `EditorDesktop`.

Phone

`WordAppMobile`

Mounted at `/sim/:sessionId/device/drafts`; uses compact bars, sticky formatting, and bottom sheets.

Desktop

`WordAppDesktop`

Mounted from `DesktopShell` with a 1000×680 default window; uses a ribbon, paper canvas, and docked Editor pane.

Scoring

Existing action types

`draft_created`, `draft_submitted_for_approval`, `draft_approved`, and `content_graded` feed the trainer ledger and team score.

Explicit non-goals: no real-time co-editing, track changes, images, tables, pagination, direct posting, or direct emailing. These exclusions keep the first build dependable and focused on crisis-writing, review, and cross-view continuity.
