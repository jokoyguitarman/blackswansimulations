Document-Driven Scenario Fidelity

A plan to make the Social Crisis War Room ingest a trainer's design document — whatever its format or organizing structure — extract it into a structured Scenario Blueprint, and drive both scenario generation and the live runtime engines from it. A detailed document yields a scenario ≥90% true to its core; an arbitrarily-structured document still maps via a schema-discovery pass plus a never-lose catch-all; a missing or thin document gracefully falls back to today's AI generation.

1. The core idea

Change the data flow from document → raw text → LLM context to document → structured Blueprint → drives generation AND runtime, with a per-field confidence/coverage score. Two rules govern everything:

Rule 1 — "Honor what was extracted; generate only what is missing." Produces both the rich-document and the no-document behaviors with no separate code paths.
Rule 2 — "Format-agnostic, ontology-flexible, never-lose." The blueprint must NOT be hard-wired to the one sample doc's structure (faction-centric, staged timeline). It absorbs arbitrary trainer frameworks by (a) discovering the trainer's own organizing concepts before mapping, (b) using open vocabularies instead of fixed enums, and (c) capturing anything it cannot confidently place into a visible catch-all rather than silently dropping it.

Rich document

High-confidence fields are used (near-)verbatim → high fidelity to the trainer's factions, stages, triggers, and rules.

No / thin document

Empty or low-confidence fields fall through to today's AI generators → identical to current behavior, zero regression.

1b. Format vs. ontology — the real risk

"Will it adapt to a different trainer's framework?" splits into two very different questions:

Format / wording / order / length

Different headings, prose vs. bullets, "actors" vs. "factions", British spelling, 3 vs. 30 pages → handled well, because extraction is LLM semantic mapping, not heading/regex parsing.

Structure / ontology (mental model)

The trainer thinks in a different shape than ours → the fragile part. A fixed schema would silently drop whatever does not fit, recreating today's lossiness one level up.

Concrete frameworks a rigid schema would mangle — and where each lands instead:

Trainer's frameworkHow the adaptive design absorbs it
Scripted inject timeline ("at T+15, journalist calls"), not charactersCaptured as a trainer_concept ("events") + timeline; personas thin but not required
Branching decision tree ("if team does X then Y")Branches land in unmapped_directives and surface in the Gap Report for review
Verbatim character sheets / exact messagesPreserved in raw_excerpts and flagged so generation quotes rather than paraphrases
Scoring rubric / objectives-firstMaps to objectives; nuance kept in trainer_concepts
Non-social domain (cyber, financial)Open vocabularies + catch-all keep domain terms; low structure_confidence triggers the raw-excerpt floor

2. Why fidelity is low today (~40–50%)

The document currently survives only as theme/flavor because of four structural facts:

ProblemWhere
The document is concatenated into one free-text string and sent as both crisis_type and context. frontend/src/pages/SocialCrisisWizard.tsx — crisisDescription memo (128–133)

PHASE 1

Structured extraction (ingestion) — two-pass, format-agnostic

Pass A (schema-discovery) classifies the framework and lists the trainer's own concepts before any mapping. Pass B (chunked map-reduce) extracts against both the standard fields and the discovered concepts, beating the 50k cap that currently truncates the tail. Anything unplaceable goes to the catch-all; nothing is dropped.

NEW FILE server/services/blueprint/blueprintExtractionService.ts
export async function extractBlueprint(rawText: string): Promise<ScenarioBlueprint> {
  if (!rawText.trim()) return emptyBlueprint();   // no doc -> empty -> generators fill everything

  // PASS A - schema discovery (one cheap whole-doc call): what KIND of framework is
  // this, and what are the trainer's OWN organizing concepts? Adapts to their shape.
  const discovery = await discoverFramework(rawText);   // -> { kind, concepts[] }

  // PASS B / MAP: split into ~6k-char overlapping chunks; extract against BOTH the
  // standard fields AND discovery.concepts, in parallel.
  const chunks = chunkText(rawText, 6000, 400);
  const partials = await Promise.all(chunks.map((c) => extractChunk(c, discovery)));

  // REDUCE: merge factions by name, union triggers/narratives, order the timeline.
  const merged = mergeBlueprints(partials, discovery);
  // NEVER-LOSE: anything unplaceable -> unmapped_directives; keep originals in raw_excerpts.
  merged.unmapped_directives = collectUnmapped(partials);
  merged.raw_excerpts = keyExcerpts(rawText);

  // SCORE: coverage drives honor-vs-generate; structure_confidence drives the floor.
  merged.coverage = scoreCoverage(merged);
  merged.structure_confidence = scoreStructure(merged);  // low -> raw-excerpt floor downstream
  return merged;
}

Fidelity validation loop

NEW FILE server/services/blueprint/fidelityCheckService.ts
export async function checkFidelity(
  blueprint: ScenarioBlueprint,
  generated: { personas: NPCPersona[]; injects: SocialInject[]; objectives: ObjectiveDef[] },
): Promise<FidelityReport> {
  const sections = {
    factions: scoreFactions(blueprint.factions, generated.personas),   // e.g. 7/7 present
    timeline: scoreTimeline(blueprint.timeline, generated.injects),      // e.g. 9/10 stages
    triggers: scoreTriggers(blueprint, generated.injects),            // e.g. 6/8 wired
    objectives: scoreObjectives(blueprint.objectives, generated.objectives),
    unmapped: scoreUnmappedHonored(blueprint.unmapped_directives, generated), // novel-format content kept accountable
  };
  const overall = weightedAverage(sections);
  return { overall, sections, weak: below(sections, 0.7) };  // "weak" -> re-prompt only those
}

Why the existing clones differ — and why that does not justify them

The OpenAI endpoint is hit directly in ~37 files. Four of them define a near-identical callAI helper (all gpt-5.2, all json_object). Read side by side they are byte-for-byte the same except the default argument values and the log string:

Filedefault maxTokensdefault temperaturewhy the default differs
socialCrisisGeneratorService.ts (286-323)80000.7large structured generation (NPCs, storyline)

antagonistEngineService.ts (26-60)12000.95one short rival post
extremistHiveService.ts (74-108)9000.95one short bait post
The token need is real; the duplication is not justified by it. maxTokens and temperature are already function parameters — so "this process needs fewer tokens" only changes a default value, never the HTTP block. One shared client taking those as arguments covers all four cases with zero loss. And since max_completion_tokens is a ceiling billed on actual output, a smaller limit is not even a cost win — you simply pass the value you want.

Some variation across the wider call sites is legitimate and a unified client must support it via options, not copies:

  • Model by task — a legacy layer (aiService.ts, hospitalCapacityService.ts) uses gpt-4o-mini for cheap analysis; research uses gpt-5.1; images use gpt-image-2; AAR uses gpt-4.
  • Param-name drift — the legacy layer uses max_tokens; the newer gpt-5.2 layer uses max_completion_tokens. A shared client normalizes this.
  • Response shape — some force json_object, others expect prose (AAR). Needs a responseFormat option.
The real smell (not token needs): the copy-pasted HTTP plumbing, inconsistent error handling (some return null silently, some log), and no retry/timeout/backoff anywhere — every clone is fire-once, so a transient 429/500 silently degrades the feature. Consolidating into one openAiJson() / openAiText() client (model + tokens + temperature + responseFormat as options) fixes all of that in one place.

No document means the Director is a no-op and generation is byte-for-byte the current behavior.
Cost control. The real driver is call count, not token ceilings — Pass B runs N parallel calls (one per chunk) and the Director fires each bounded tick. Control via chunk size + Director cadence + caching extraction per upload + repairing only weak sections. Raising max_completion_tokens does not add cost on its own.
