How to read this guide

Every refactor below follows the same shape so it becomes predictable:

  • The pattern — the repeated code, with the real file and line numbers from your repo.
  • Why shorten it — the practical payoff (fewer bugs, less typing, one place to change).
  • Old template vs new template — side-by-side, so the shape of the change is obvious.
  • Line-by-line — every symbol explained, assuming zero prior knowledge.
Golden rule

The single most valuable habit in programming: if you copy-paste the same block more than twice, turn it into a function. A function is just a named, reusable box of code. Almost everything in this guide is one application of that rule.

The 30-second mental model of your app

Your project has two halves that talk to each other over the internet:

HalfWhereJob
Frontendfrontend/srcWhat the user sees and clicks (React + TypeScript). It asks the backend for data.
Backendserver/Receives requests, checks permissions, talks to the database (Supabase), sends data back.

Inside the backend there are three recurring kinds of files, and knowing which is which tells you where shared code belongs:

  • Routes (server/routes/*.ts) — one file per feature (e.g. channels.ts). Each defines URLs ("endpoints") the frontend can call.
  • Services (server/services/*.ts) — reusable business logic (sending notifications, websockets, etc.).
  • Lib (server/lib/*.ts) — small shared helpers (validation, logging, the access checker).
Key insight for this whole guide

Most of your duplication lives in the routes. The fix is almost always: move the repeated logic into a lib or service helper, then call it. Your repo already started doing this with server/lib/access.ts — several routes just haven't been updated to use it yet.

Syntax primer — the alphabet you'll see everywhere

Before the refactors, here is every symbol used below, in one table. Skim it now; refer back whenever something looks alien.

You'll seeIt's calledIn plain English
const x = 5constant declaration"Make a name x that points to 5 and never gets reassigned." Use const by default.
function name() {}functionA reusable box of steps. name() with parentheses runs it.
(a, b) => { }arrow functionA shorter way to write a function with no name. => means "takes these inputs, does this."

...objspread"Copy everything from obj into here." Used to clone/merge objects and arrays.

a ?? bnullish coalescing"Use a, unless it's null/undefined, then use b as a fallback."
a || blogical OR"Use a if it's truthy, otherwise b." (Looser than ??: also falls back on 0, "", false.)

`Hi ${name}`template literalA string with holes. ${...} drops a value into the text. Note the backticks.
arr.map(fn)array map"Make a new array by running fn on every item." (.filter keeps some, .find returns the first match.)
x: stringtype annotationTypeScript label: "x must be text." Catches mistakes before the app runs. Doesn't affect what the code does.
x as Footype assertion"Trust me, treat x as a Foo." Tells TypeScript to stop worrying; does nothing at runtime.

Two confusing ones, side by side

?? vs ||: both supply a fallback. ?? only falls back on null/undefined. || also falls back on any falsy value (0, empty string, false). For "a count that might legitimately be 0," use ?? so a real 0 isn't thrown away.

1 · The repeated "is this user allowed?" check biggest win

Appears 3 times inside channels.ts alone, and the same shape appears across 13 route files. A ready-made replacement already exists in server/lib/access.ts.

The pattern (what's repeated)

In server/routes/channels.ts, this almost-identical block shows up at lines 58–81, 308–331, and a longer variant at 124–180. Here is the version at lines 58–81:

server/routes/channels.ts · lines 58–81 (current)
// Verify user has access to session
const { data: session } = await supabaseAdmin
  .from('sessions')
  .select('id, trainer_id')
  .eq('id', sessionId)
  .single();

if (!session) {
  return res.status(404).json({ error: 'Session not found' });
}

// Check if user is trainer or participant
if (session.trainer_id !== user.id && user.role !== 'admin') {
  const { data: participant } = await supabaseAdmin
    .from('session_participants')
    .select('*')
    .eq('session_id', sessionId)
    .eq('user_id', user.id)
    .single();

  if (!participant) {
    return res.status(403).json({ error: 'Access denied' });
  }
}

Line-by-line (read this once, it teaches half the codebase)

  • data: session await supabaseAdmin.from('sessions')... asks the database for a row from the sessions table. The result is an object like { data, error }; { data: session } pulls out data and renames it to session.
  • .from picks the table. .select('id, trainer_id') says "only fetch these two columns" (faster than '*' = everything).
  • .eq "equals" — a filter: .eq('id', sessionId) = "where the id column equals our sessionId."
  • .single() "I expect exactly one row." (Its safer cousin .maybeSingle() returns nothing instead of erroring when there are zero rows — that's why the longer variant in your file uses it.)
  • if (!session) ! means "not." So "if there is no session, send back HTTP 404 (not found) and return to stop."
  • !== && !== is "not equal." && is "and." Together: "if the caller is NOT the owning trainer AND is NOT an admin, then we must check further."
  • 403 A second DB call checks whether the user is a participant. If not, send HTTP 403 (forbidden).

Why shorten it

Why
  • One source of truth. Security rules copy-pasted 13 times means 13 places to get it wrong. If you ever change the rule (say, add a "co-trainer" role), you'd have to find and edit every copy.
  • The variants have already drifted. Some copies use .single(), others .maybeSingle(), some log the error, some don't. That drift is exactly how subtle bugs are born.
  • It's shorter. ~24 lines collapse to 2.

Old template → New template

OLD — inline, repeated everywhere
const { data: session } = await supabaseAdmin
  .from('sessions')
  .select('id, trainer_id')
  .eq('id', sessionId)
  .single();
if (!session)
  return res.status(404)
    .json({ error: 'Session not found' });
if (session.trainer_id !== user.id
    && user.role !== 'admin') {
  const { data: participant } = await supabaseAdmin
    .from('session_participants')
    .select('*')
    .eq('session_id', sessionId)
    .eq('user_id', user.id)
    .single();
  if (!participant)
    return res.status(403)
      .json({ error: 'Access denied' });
}
NEW — call the existing helper
// at the top of the file (you already import others):
import { assertSessionAccess } from '../lib/access.js';

// inside the route:
const access = await assertSessionAccess(sessionId, user);
if (!access.ok)
  return res.status(access.status)
    .json({ error: access.error });

// bonus: the helper hands back the session row,
// so you don't need a second fetch:
const session = access.session;

How the new version works (and what talks to what)

The helper already exists in your repo. Here's the relevant part of server/lib/access.ts (lines 40–78), so you can see there's no magic:

server/lib/access.ts · lines 40–78 (existing helper)
export async function assertSessionAccess(
  sessionId: string | null | undefined,
  user: AccessUser,
  select = 'id, trainer_id',
): Promise<AccessResult> {
  if (!sessionId)
    return { ok: false, status: 404, error: 'Session not found' };

  const { data: session, error } = await supabaseAdmin
    .from('sessions').select(columns)
    .eq('id', sessionId).maybeSingle<SessionRow>();

  if (error || !session)
    return { ok: false, status: 404, error: 'Session not found' };

  // Owning trainer or admin always allowed.
  if (user.role === 'admin' || session.trainer_id === user.id)
    return { ok: true, session };

  // Otherwise must be a participant.
  const { data: participant } = await supabaseAdmin
    .from('session_participants').select('user_id')
    .eq('session_id', sessionId)
    .eq('user_id', user.id).maybeSingle();

  if (!participant)
    return { ok: false, status: 403, error: 'Access denied' };

  return { ok: true, session };
}
  • export "Let other files use this." Your route file imports what this file exports. That import line is the "wire" connecting channels.ts → access.ts.
  • params The function takes the same two things the inline code used — sessionId and user — plus an optional select with a default value (= 'id, trainer_id'), so you can usually omit it.
  • : Promise<AccessResult> The return type. Because it's async, it returns a Promise (a value that arrives later). AccessResult is either { ok: true, session } or { ok: false, status, error }.
  • the deal It returns an object instead of sending the HTTP response itself. That's deliberate — your call site decides what to do. This is why the new template checks if (!access.ok) and then sends the response. It keeps the helper reusable.
What you actually do

In channels.ts, three blocks become three two-line calls. (Two routes — the GET /messages and POST /messages — already use assertSessionAccess, so you're matching an existing pattern, not inventing one.)

2 · The giant copy-pasted error logger high value

The errorCode: x.code shape appears 7 times in channels.ts and across many route/service files.

The pattern

Whenever a database call fails, the code logs the same five fields. Example from channels.ts lines 190–206:

server/routes/channels.ts · lines 190–206 (current)
if (channelsError) {
  logger.error(
    {
      error: channelsError,
      errorCode: channelsError.code,
      errorMessage: channelsError.message,
      errorDetails: channelsError.details,
      errorHint: channelsError.hint,
      sessionId,
      userId: user.id,
    },
    'Failed to fetch DM channels',
  );
  return res.status(500).json({
    error: 'Failed to fetch DM channels',
    details: channelsError.message,
  });
}

Line-by-line

  • logger.error Writes a message to your server log. First argument = an object of structured data; second = a human-readable summary. (logger comes from ../lib/logger.js.)
  • { error, ... } The five errorX: fields just copy properties off the error object (.code, .message, .details, .hint). They're identical every time — only the table/summary changes.
  • status(500) HTTP 500 = "server error." details: channelsError.message passes the DB's own message back to the caller.

Why shorten it

Why

It's ~13 lines of noise repeated dozens of times. The only things that vary are (a) the error object, (b) the log message, and (c) some context like sessionId. A helper captures the boring 90% and lets you pass the 10% that differs. Change the log format once → it changes everywhere.

Old template → New template

First, add a tiny helper next to your logger (a new function in server/lib/logger.ts or a new server/lib/httpError.ts):

server/lib/httpError.ts · NEW small helper
import { logger } from './logger.js';
import type { Response } from 'express';

// One place that knows how to log a DB error AND reply.
export function dbFail(
  res: Response,
  err: any,
  message: string,
  context: Record<string, unknown> = {},
) {
  logger.error(
    { error: err, errorCode: err?.code, errorMessage: err?.message,
      errorDetails: err?.details, errorHint: err?.hint, ...context },
    message,
  );
  return res.status(500).json({ error: message, details: err?.message });
}
OLD — at every failed query
if (channelsError) {
  logger.error(
    { error: channelsError,
      errorCode: channelsError.code,
      errorMessage: channelsError.message,
      errorDetails: channelsError.details,
      errorHint: channelsError.hint,
      sessionId, userId: user.id },
    'Failed to fetch DM channels',
  );
  return res.status(500).json({
    error: 'Failed to fetch DM channels',
    details: channelsError.message });
}
NEW — one line
if (channelsError)
  return dbFail(res, channelsError,
    'Failed to fetch DM channels',
    { sessionId, userId: user.id });

New syntax you haven't met yet

  • err?.code The ?. means "if err somehow isn't there, don't crash — just give undefined." Safer than the old err.code.
  • ...context Spread again: it dumps whatever extra fields you passed ({ sessionId, userId }) into the log object alongside the standard ones.
  • = {} A default: if you don't pass a context, it uses an empty object so nothing breaks.
  • Record<string, unknown> A TypeScript type meaning "an object with text keys and any kind of values." Just a label for "a plain bag of extra data."
  • import type Imports a type only (for checking), which disappears when the app runs. A small correctness/perf nicety.

3 · The "members" cast, written 24 times easy win

(channel.members as string[]) || [] appears 6 times in channels.ts and ~24 across the repo.

The pattern

server/routes/channels.ts · e.g. lines 209–212, 590, 742, 853 (current)
const members = (channel.members as string[]) || [];
return Array.isArray(members) && members.includes(user.id);

Line-by-line

  • channel.members The members column is stored as JSON, so TypeScript doesn't know it's a list of user IDs.
  • as string[] "Trust me, it's an array of strings (text)." string[] = "list of strings."
  • || [] "…but if it's missing, use an empty list []" so later code doesn't crash.
  • Array.isArray / .includes "Is it really an array?" then "does it contain this user's id?" → true/false.

Why shorten it

Why

It's small but error-prone: forget the || [] in one spot and the app can crash on a channel with no members. A one-line helper guarantees you always get a safe array, and gives the intent a name.

Old template → New template

OLD — repeated cast
const members =
  (channel.members as string[]) || [];

if (!members.includes(user.id))
  return res.status(403)
    .json({ error: 'Access denied' });
NEW — named helper
// in a shared file, once:
export const getMembers = (c: any): string[] =>
  Array.isArray(c?.members) ? c.members : [];

// at each call site:
const members = getMembers(channel);
if (!members.includes(user.id))
  return res.status(403)
    .json({ error: 'Access denied' });
  • (c) => ... An arrow function that takes a channel c and returns its members.
  • : string[] Promises callers "you'll always get a list of strings," so they never need the || [] safety net again.
  • ? : The ternary: "if it really is an array, return it; otherwise return an empty list." Same safety as before, now in one place.

4 · The try/catch wrapper around every route bigger change

The closing catch (err) { ... 'Internal server error' } appears ~200+ times across your routes. This one's optional — do it once you're comfortable.

The pattern

Every route is wrapped in the same safety net so one crash doesn't take the server down:

server/routes/channels.ts · the shape repeated in every route
router.get('/session/:sessionId', requireAuth, async (req, res) => {
  try {
    // ... the real work ...
    res.json({ data: channels });
  } catch (err) {
    logger.error({ error: err }, 'Error in GET /channels/session/:sessionId');
    res.status(500).json({ error: 'Internal server error' });
  }
});

Line-by-line

  • router.get "When someone visits this URL with a GET request, run this function." post is for sending data. '/session/:sessionId' — the :sessionId part is a placeholder filled from the URL.
  • requireAuth A middleware: a checkpoint that runs first and rejects users who aren't logged in. Multiple checkpoints can be listed in order.
  • async (req, res) The handler. req = the incoming request (URL parts, body, the logged-in user). res = your tool for replying.
  • try / catch "Try the real work; if anything throws an unexpected error, jump to catch" so the server replies cleanly instead of crashing.

Why shorten it

Why

The try/catch is pure ceremony — identical in every route. You can write it once as a wrapper function and stop repeating it. Fewer lines, and you can never forget it (forgetting it is a real crash risk).

Old template → New template

server/lib/asyncHandler.ts · NEW wrapper
import type { Request, Response, NextFunction } from 'express';

// Takes your route function, returns a "wrapped" version
// that auto-catches errors.
export const asyncHandler =
  (fn: any) =>
  (req: Request, res: Response, next: NextFunction) =>
    Promise.resolve(fn(req, res, next)).catch(next);
OLD — try/catch in every route
router.get('/x', requireAuth, async (req, res) => {
  try {
    // real work
    res.json({ data });
  } catch (err) {
    logger.error({ error: err }, 'Error in GET /x');
    res.status(500)
      .json({ error: 'Internal server error' });
  }
});
NEW — wrapper handles it
router.get('/x', requireAuth, asyncHandler(async (req, res) => {
  // real work — no try/catch needed
  res.json({ data });
}));
  • fn Your original route function. The wrapper just runs it for you.
  • Promise.resolve(...).catch(next) "Run the function; if it fails, hand the error to next." next forwards errors to Express's central error handler — one place that logs + replies 500.
  • a function returning a function Notice two => in a row. asyncHandler(myRoute) returns a new function that Express calls. This "function that makes a function" pattern is common; you only need to use it, not write it often.
Caveat

This needs a small one-time central error handler registered in server/index.ts to produce the 500 JSON. It's the most involved refactor here — totally fine to save for later. The first three give the biggest payoff for the least risk.

5 · The duplicated "look up team names" block high value

Nearly identical logic appears in channels.ts at lines 376–392 (participants) and 622–648 (messages).

The pattern

server/routes/channels.ts · lines 378–392 (current)
const { data: teamRows } = await supabaseAdmin
  .from('session_teams')
  .select('user_id, team_name')
  .eq('session_id', sessionId)
  .in('user_id', participantIds);

const userTeamMap = new Map<string, string>();
for (const row of teamRows ?? []) {
  const r = row as { user_id: string; team_name: string };
  if (!userTeamMap.has(r.user_id))
    userTeamMap.set(r.user_id, r.team_name);
}

Line-by-line

  • .in('user_id', ids) Like .eq but for a list: "where user_id is any of these ids." One query for many users instead of one query per user.
  • new Map() A Map is a lookup table of key → value. Here: user id → team name. .set stores, .get retrieves, .has checks existence.
  • for...of "Do this for each row." teamRows ?? [] = "use the rows, or an empty list if none."
  • if (!...has) "Only record the first team per user" (ignore duplicates).

Why shorten it

Why

It's the same query-and-build-a-map routine in two routes (and likely useful in more). Naming it getTeamNameMap(sessionId, userIds) makes each route read like a sentence and guarantees both routes behave identically.

Old template → New template

server/lib/teams.ts (or add to access.ts) · NEW helper
export async function getTeamNameMap(
  sessionId: string,
  userIds: string[],
): Promise<Map<string, string>> {
  const map = new Map<string, string>();
  if (userIds.length === 0) return map;

  const { data } = await supabaseAdmin
    .from('session_teams').select('user_id, team_name')
    .eq('session_id', sessionId).in('user_id', userIds);

  for (const row of data ?? []) {
    const r = row as { user_id: string; team_name: string };
    if (!map.has(r.user_id)) map.set(r.user_id, r.team_name);
  }
  return map;
}
OLD — ~12 lines, twice
const { data: teamRows } = await supabaseAdmin
  .from('session_teams')
  .select('user_id, team_name')
  .eq('session_id', sessionId)
  .in('user_id', participantIds);
const userTeamMap = new Map<string,string>();

  const r = row as {user_id:string;team_name:string};

    userTeamMap.set(r.user_id, r.team_name);
}
NEW — one line, reads like English
const userTeamMap =
  await getTeamNameMap(sessionId, participantIds);

// use exactly as before:
// userTeamMap.get(someUserId)

6 · Duplication inside the helper file itself easy win

Even good files repeat themselves. In server/lib/access.ts, assertSessionAccess (lines 49–58) and assertSessionOwner (lines 93–102) start with the exact same "load the session" block.

The pattern

server/lib/access.ts · this block exists twice
const columns = ensureColumns(select);
const { data: session, error } = await supabaseAdmin
  .from('sessions')
  .select(columns)
  .eq('id', sessionId)
  .maybeSingle<SessionRow>();

if (error || !session) {
  return { ok: false, status: 404, error: 'Session not found' };
}

Why & how

Why

Both functions need the session row, fetched identically. Extract a tiny private loadSession() and both call it. Bonus learning: this shows that helpers can themselves use smaller helpers — that layering is how clean code stays clean.

OLD — same fetch in two functions
// inside assertSessionAccess AND
// inside assertSessionOwner, identical:
const columns = ensureColumns(select);
const { data: session, error } = await supabaseAdmin
  .from('sessions').select(columns)
  .eq('id', sessionId)
  .maybeSingle<SessionRow>();
if (error || !session)
  return { ok:false, status:404,
    error:'Session not found' };
NEW — one private helper
// private (no "export") = only this file uses it
async function loadSession(sessionId: string, select: string) {
  const { data, error } = await supabaseAdmin
    .from('sessions').select(ensureColumns(select))
    .eq('id', sessionId)
    .maybeSingle<SessionRow>();
  return error ? null : data;
}

// then in each function:
const session = await loadSession(sessionId, select);
if (!session)
  return { ok: false, status: 404, error: 'Session not found' };
  • no export Leaving off export makes loadSession private to this file — other files can't (and shouldn't) call it. Good for internal plumbing.
  • return error ? null : data "If the query errored, hand back null; otherwise the row." The callers then do one simple if (!session) check.

How the pieces interact (the wiring)

Here's the journey of one request, showing where each helper plugs in. Follow the arrows — this is "what talks to what":

 Browser (frontend/src/...)
     │  user clicks "open channel"
     ▼
 HTTP request  ──►  server/index.ts        (starts the server, wires routers)
                        │
                        ▼
                    requireAuth            (middleware: are you logged in?)  ◄─ server/middleware/auth.ts
                        │
                        ▼
                    channels.ts route       (the handler for this URL)
                        │
            ┌───────────┼───────────────┬───────────────────────┐
            ▼           ▼               ▼                        ▼
    assertSessionAccess  dbFail/logger   getTeamNameMap          getMembers
    (lib/access.ts)      (lib/...)        (lib/teams.ts)          (lib helper)
            │
            ▼
       supabaseAdmin  ──►  Supabase database   (lib/supabaseAdmin.ts)
            │
            ▼
       reply: res.json({ data })  ──►  back to the Browser

Two universal facts this diagram teaches:

  • Imports are the wires. A line like import { assertSessionAccess } from '../lib/access.js' at the top of channels.ts is what lets the route "reach" the helper. No import, no connection.
  • Helpers flow downward. Routes call lib/services; lib/services call the database. Data flows back up the same path. Keeping that direction tidy is what makes a codebase navigable.

Suggested order & how to stay safe

Do these in order of "most reward, least risk":

#RefactorEffortRiskWhy first/last

Beginner safety net
  • Change one thing, then test. Do a single route, run the app (npm run dev), confirm it still works, then move on.
  • Let the type checker help. Run npm run typecheck — it flags many mistakes before you ever click anything.
  • Commit between refactors. A git commit after each working step is a save point you can return to.
  • Refactors shouldn't change behavior. If the app behaves differently after, something's off — revert and compare.
