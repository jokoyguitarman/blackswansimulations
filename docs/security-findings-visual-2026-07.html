Critical

C1

Voice recordings & transcripts readable by any authenticated user (IDOR)

GET /api/voice/calls/:sessionId · server/routes/voice.ts
CriticalOpen

What it is

An IDOR (Insecure Direct Object Reference): the endpoint takes a sessionId from the URL and returns every voice call, recording file path, and speech-to-text transcript for that session — without ever checking that the caller is the session's trainer or a participant. Any logged-in account (including a throwaway "anonymous join" participant) can read private recorded communications from any exercise just by changing the ID in the URL. The new assertSessionAccess() guard exists but was never wired into this route.

Attack scenario · session transcript theft

🎭 A participant harvests another team's private call transcripts

Attacker
Mallory
anonymous participant, valid login for her own session
Target
Session B
a different agency's live exercise
Server
/api/voice/calls
service-role DB, RLS bypassed
  1. Get a valid token. Mallory joins her own session via a public join link, which gives her a normal Supabase JWT. She is authenticated — that's all the endpoint checks.
  2. Guess/collect a target session ID. Session IDs are UUIDs but they leak through shared links, screenshots, the network tab, and WebSocket rooms. She obtains Session B's ID.
  3. Swap the ID into the calls endpoint.
    bash — attacker
    $ curl https://app.blackswan.example/api/voice/calls/B_SESSION_UUID \
     -H "Authorization: Bearer <mallory_own_token>"
  4. Server returns everything.
    200 OK — response
    { "data": [ {
  "transcript": "Command, we're pulling the cordon back to Grid 4...",
  "storage_path": "voice/B_SESSION/call_88/commander.webm",
  "started_at": "2026-07-12T08:04:11Z" } ] }
Impact: full confidentiality breach of recorded voice comms across every session in the system, by ID enumeration. Nothing stops cross-tenant reads.

The fix — apply the existing access guard

Before · server/routes/voice.ts
router.get('/calls/:sessionId', requireAuth, async (req, res) => {
  const { sessionId } = req.params;

  // straight to the DB — no ownership/membership check
  const { data: calls, error } = await supabaseAdmin
    .from('voice_calls')
    .select('*, voice_recordings(*)')
    .eq('session_id', sessionId);
  ...
});
After · gate with assertSessionAccess()
import { assertSessionAccess } from '../lib/access.js';

router.get('/calls/:sessionId', requireAuth, async (req, res) => {
  const { sessionId } = req.params;
  const user = req.user!;

  // trainer/admin or a participant of THIS session only
  const access = await assertSessionAccess(sessionId, user);
  if (!access.ok) return res.status(access.status).json({ error: access.error });

  const { data: calls, error } = await supabaseAdmin
    .from('voice_calls')
    .select('*, voice_recordings(*)')
    .eq('session_id', sessionId);
  ...
});

High

H1

Objectives update — any trainer can rewrite any session's scores (IDOR)

POST /api/objectives/session/:sessionId/update · server/routes/objectives.ts
HighOpen

What it is

The route checks that the caller is a trainer or admin, but never checks that they own the target session. Since the role check passes for every trainer, one trainer can overwrite the objective progress, status, and scores of a completely different trainer's exercise. The sibling GET routes in the same file do check ownership — this one was missed.

Attack scenario · sabotaging a rival's assessment

🎭 Trainer tampers with another trainer's graded objectives

Attacker
Trainer A
legitimate trainer account
Target
Trainer B's session
a live graded exercise A does not own
Server
/objectives/.../update
role check only
  1. A is a valid trainer. The only gate is role === 'trainer' || 'admin', which A satisfies.
  2. A targets B's session ID and pushes bogus progress.
    bash — attacker
    $ curl -X POST \
   https://app.../api/objectives/session/TRAINER_B_SESSION/update \
   -H "Authorization: Bearer <trainer_A_token>" \
   -H "Content-Type: application/json" \
   -d '{"objective_id":"obj_evac","progress_percentage":0,"status":"failed"}'
  3. Server accepts it. No ownership check runs, so updateObjectiveProgress() writes to B's session. B's AAR and participant scores are now corrupted.
    200 OK
    { "success": true }
Impact: integrity loss on graded training outcomes — a trainer can silently degrade or inflate another organisation's assessment, undetected.

The fix — require session ownership

Before
if (user.role !== 'trainer' && user.role !== 'admin') {
  return res.status(403).json({ error: 'Only trainers can update objectives' });
}
// <-- no check that THIS trainer owns sessionId
await updateObjectiveProgress(sessionId, objective_id, progress_percentage, {...});
After · assertSessionOwner()
if (user.role !== 'trainer' && user.role !== 'admin') {
  return res.status(403).json({ error: 'Only trainers can update objectives' });
}

// only the owning trainer (or admin) may mutate this session's objectives
const access = await assertSessionOwner(sessionId, user);
if (!access.ok) return res.status(access.status).json({ error: access.error });

await updateObjectiveProgress(sessionId, objective_id, progress_percentage, {...});
H3

Voice upload trusts client headers — path injection + no membership check

POST /api/voice/upload · server/routes/voice.ts
HighOpen

What it is

The upload handler builds the storage path directly from the x-session-id and x-call-id request headers, with no validation and no check that the caller belongs to the session. Two problems: (1) any authenticated user can plant recordings and DB rows in any session (spoofing "who said what"); (2) because the header values are interpolated into the object path, values containing ../ can push files to unintended locations within the storage bucket.

Attack scenario · path traversal + evidence spoofing

🎭 Attacker writes a forged recording into a victim session's namespace

Attacker
Mallory
any authenticated user
Server
/voice/upload
builds path from headers
Storage
voice-recordings bucket
service-role writes
  1. Craft headers pointing at a victim session.
    bash — attacker
    $ curl -X POST https://app.../api/voice/upload \
   -H "Authorization: Bearer <mallory_token>" \
   -H "Content-Type: audio/webm" \
   -H "x-session-id: VICTIM_SESSION/../../voice/VICTIM_SESSION" \
   -H "x-call-id: commander_call" \
   --data-binary @forged-order.webm
  2. Server interpolates the raw header into the path. voice/${sessionId}/${callId}/${userId}.webm — the ../ segments and the victim IDs are trusted verbatim.
  3. A row is inserted claiming the recording belongs to the victim session/call. Later AAR transcription picks it up as genuine evidence.
    200 OK
    { "data": { "storagePath": "voice/VICTIM_SESSION/.../mallory.webm" } }
Impact: forged audio evidence injected into another session's after-action record, plus object-path manipulation in shared storage.

The fix — validate IDs + verify membership

Before
const callId = req.headers['x-call-id'];
const sessionId = req.headers['x-session-id'];
if (!callId || !sessionId) return res.status(400)...;

const userId = req.user!.id;
// header values used directly in the storage path
const storagePath = `voice/${sessionId}/${callId}/${userId}.${ext}`;
After
const callId = req.headers['x-call-id'];
const sessionId = req.headers['x-session-id'];
if (!callId || !sessionId) return res.status(400)...;

const UUID = /^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$/i;
if (!UUID.test(sessionId) || !UUID.test(callId))
  return res.status(400).json({ error: 'Invalid session or call id' });

// caller must belong to the session ...
const access = await assertSessionAccess(sessionId, req.user!);
if (!access.ok) return res.status(access.status).json({ error: access.error });

// ... and the call must actually be part of that session
const { data: call } = await supabaseAdmin.from('voice_calls')
  .select('id').eq('id', callId).eq('session_id', sessionId).single();
if (!call) return res.status(400).json({ error: 'Call does not belong to session' });

const userId = req.user!.id;
const storagePath = `voice/${sessionId}/${callId}/${userId}.${ext}`;
H4

Backend dependencies still vulnerable (nodemailer, uuid)

package.json · nodemailer ^7.0.10, uuid via exceljs
HighPartly fixed

What it is

The frontend advisories were cleared (its npm audit is now clean). The backend still ships a vulnerable nodemailer — and the advisory list has grown since June: SMTP command injection, CRLF header injection, jsonTransport file-access bypass, OAuth2 TLS validation, and raw-message SSRF/file-read. nodemailer is used at runtime by the email service, so untrusted values that flow into email (recipient names, subjects, addresses) are the exposure surface. uuid is also pulled in transitively through exceljs.

N1

Payout release has no idempotency guard → double transfer of real money

POST /api/billing/payouts/:id/release · server/routes/billing.ts
MediumNew

What it is

Releasing a trainer payout reads the row, checks status === 'pending_release', calls Stripe transfers.create(), then flips the row to released. There is no row lock/transaction and no Stripe idempotency key, so this is a classic time-of-check/time-of-use (TOCTOU) race: two requests that arrive close together both pass the status check and both issue a Stripe transfer for the same payout. It's admin-only and low-likelihood, but the blast radius is duplicated money movement.

Attack / failure scenario · the double-click that pays twice

💸 Two concurrent releases both clear the same check

Request 1
reads status
= pending_release ✓
Request 2
reads status
= pending_release ✓ (before #1 writes)
Stripe
transfers.create
called twice, no idem key
  1. Trigger. An admin double-clicks "Release", a proxy retries a slow request, or two admins act at once.
  2. Both requests read pending_release. Neither has written released yet, so both proceed past the guard.
  3. Two Stripe transfers fire. Same payout_id in metadata, but Stripe has no idempotency key to collapse them, so it creates two.
    stripe — server logs
    tr_1Ab...  amount=90000 sgd  created  payout_id=po_42
tr_1Ac...  amount=90000 sgd  created  payout_id=po_42  ← duplicate
Impact: a trainer can be paid twice for one engagement; reconciling requires a manual Stripe reversal.

The fix — conditional status flip + Stripe idempotency key

Before
const { data: payout } = await supabaseAdmin.from('payouts')
  .select('*').eq('id', id).maybeSingle();

if (payout.status !== 'pending_release') return res.status(409)...;
// nothing stops a second request reaching here too
transferId = await createTransfer({ accountId, amountCents: payout.amount_cents, ... });
After
// 1) atomically claim the payout: only ONE request wins this UPDATE
const { data: claimed } = await supabaseAdmin.from('payouts')
  .update({ status: 'releasing' })
  .eq('id', id).eq('status', 'pending_release')  // conditional
  .select().maybeSingle();
if (!claimed) return res.status(409).json({ error: 'Payout is not releasable' });

// 2) deterministic idempotency key so a retry can never double-charge
transferId = await createTransfer({ accountId, amountCents: claimed.amount_cents,
  idempotencyKey: `payout_release_${claimed.id}`, ... });
The rest of billing is solid. The Stripe webhook verifies the signature over the raw body, dedupes on event_id, ignores unknown invoices, and won't unwind an already-paid engagement. Ownership is checked in code and admin money endpoints re-check role === 'admin' per call. N1 is the one gap.
M1

Express error handler never runs → stack traces leak outside production

server/index.ts (~line 235)
MediumOpen

What it is

Express recognises an error-handling middleware only if it declares four parameters (err, req, res, next). This handler declares three, so Express treats it as an ordinary middleware and never calls it on errors. Thrown errors fall through to Express's built-in finalhandler, which — when NODE_ENV is not production — sends the full stack trace in the HTTP response. The intended sanitisation and centralised logging silently do nothing.

🧩 What the client sees on any thrown error (non-prod)

500 — response body (leaked)
Error: relation "sessions" does not exist
    at /app/server/routes/sessions.ts:155:24
    at process.processTicksAndRejections (node:internal/...)
    ... full internal stack, file paths, and query shape ...
Impact: information disclosure (source paths, DB schema, internal structure) that helps an attacker map the system. In production it also means errors aren't going through your log/scrub path as designed.

The fix — add the fourth parameter

Before (3 params — ignored)
app.use((err, _req, res) => {
  const error = err as Error;
  logger.error({ ... });
  res.status(500).json({ ... });
});
After (4 params — registered)
app.use((err, _req, res, _next) => {
  const error = err as Error;
  logger.error({ ... });
  res.status(500).json({ ... });
});
M3

Invitation token written to logs in clear text

server/routes/invitations.ts · server/lib/logger.ts
MediumOpen

What it is

The invitations route logs { error, token }, and the pino redaction list only redacts *.token — a pattern that matches a nested key like foo.token but not a top-level token. So the raw invitation token (which grants access to the signup/session flow) is persisted to logs. Anyone who can read logs (ops tooling, a log-aggregation breach, a shared console) can replay it.

Before · invitations.ts
logger.warn({ error, token },
  'Invitation not found or invalid');
After · don't log the secret
logger.warn({ error },
  'Invitation not found or invalid');
After · also close the redaction gap in logger.ts
redact: {
  paths: [
    'req.headers.authorization',
    'req.headers.cookie',
    'token',            // top-level token key
    '*.password', '*.token', '*.apiKey', '*.secret',
  ],
  remove: true,
}
M4

Internal database error text returned to clients

server/routes/channels.ts, sessions.ts, injects.ts
MediumOpen

What it is

Multiple handlers return details: error.message — the raw Supabase/Postgres error — straight to the client. That text leaks table names, constraint names, and migration hints. One branch literally tells the client to "run migration 017". This is a roadmap for an attacker probing the data model.

Before
res.status(500).json({
  error: 'Failed to fetch channel',
  details: channelError.message,
});
After
// full detail is already logged server-side
res.status(500).json({
  error: 'Failed to fetch channel',
  ...(env.nodeEnv !== 'production' && { details: channelError.message }),
});

Occurrences: channels.ts lines 148, 177, 208, 514, 520, 578, 620, 779 · sessions.ts 155 · injects.ts 808, 936, 943, 961.

M5

Rate-limit bucket derived from an unverified JWT

server/index.ts (~line 133)
MediumOpen

What it is

The global limiter calls jwt.decode() — which does not verify the signature — to key the rate limit as user:<sub>. The token isn't verified until later, so the limiter trusts attacker-controlled input. An attacker can forge tokens with arbitrary sub values.

🎭 Two abuses of a forgeable rate-limit key

  1. Evade the limit. Send a garbage-but-decodable JWT with a fresh random sub on every request → every request lands in a brand-new bucket, so the per-user cap never bites.
  2. Deny service to a victim. Forge a token whose sub equals a known victim's user ID and hammer the API → you exhaust the user:victim bucket, and the real victim gets throttled.
    bash — attacker
    $ forge() { printf '{"sub":"%s"}' "$1" | base64; }   # unsigned, still decodable
$ curl .../api/... -H "Authorization: Bearer eyJ.$(forge VICTIM_ID).x"
Impact: rate-limit bypass (undermines abuse/DoS protection) and targeted denial-of-service against a chosen user.

The fix — key by IP; do per-user limiting after auth

Before
keyGenerator: (req) => {
  const token = req.headers.authorization?.split(' ')[1];
  const decoded = jwt.decode(token);   // NOT verified — attacker controls sub
  if (decoded?.sub) return `user:${decoded.sub}`;
  return ipKeyGenerator(req.ip);
}
After
// global limiter keys on IP only — never trust an unverified token
keyGenerator: (req) => ipKeyGenerator(req.ip ?? 'unknown'),

// optional: a second limiter mounted AFTER requireAuth, where
// req.user.id is verified and safe to key on:
// router.use(requireAuth, rateLimit({ keyGenerator: r => `user:${r.user.id}` }));

Warroom document upload (PDF/DOCX parsing) is not staff-gated

POST /api/warroom/social-crisis/upload-document · socialCrisisWarroom.ts
LowNew

What it is

The endpoint runs behind requireAuth only — no requireStaff. Any authenticated user (including anonymous join participants) can submit files that are parsed server-side by pdf-parse / mammoth. The 10 MB multer limit caps size, but crafted documents still burn CPU/memory (a light DoS vector), and the extracted text later feeds AI blueprint extraction (a prompt-injection surface, though that downstream step is credit-gated).

  requireAuth,
  upload.single('file'),
  async (req, res) => { ... });
After
router.post('/upload-document',
  requireAuth, requireStaff,
  upload.single('file'),
  async (req, res) => { ... });
L1

Unauthenticated open tile proxy

GET /api/tiles/:z/:x/:y.png · server/routes/tileProxy.ts
LowOpen

What it is

The proxy is unauthenticated and interpolates the raw z/x/y path params into an outbound fetch. The destination host is hard-pinned to OpenStreetMap, so this isn't full SSRF — but it is an open, uncapped relay (bandwidth amplification) and the coordinates aren't validated as integers.

After · require auth + bound the coordinates
router.get('/:z/:x/:y.png', requireAuth, async (req, res) => {
  const z = Number(req.params.z), x = Number(req.params.x), y = Number(req.params.y);
  const max = 2 ** z;
  if (!Number.isInteger(z) || z < 0 || z > 22 ||
      !Number.isInteger(x) || x < 0 || x >= max ||
      !Number.isInteger(y) || y < 0 || y >= max) return res.status(400).end();
  // ... existing OSM fetch ...
});
L2–L5

Remaining low-severity hardening items

server/index.ts · server/env.ts · frontend components
LowMixed
  • L2 — Invitation lookup not rate-limited. GET /api/invitations/:token is public but only under the general limiter. Apply the strict joinLimiter to slow token enumeration: app.use('/api/invitations', joinLimiter);
  • L3 — CORS allowlist hardcodes localhost (a localhost:4173 entry was even added). Include the localhost origins only when env.nodeEnv !== 'production'.
  • L4 — SESSION_SECRET default (mostly addressed). It now refuses to boot with the dev default on Vercel/Render. Residual: the value is unused in code, and a non-prod, non-Vercel/Render host could still boot with the default. Remove the unused config or fail closed whenever NODE_ENV !== 'development'.
  • L5 — dangerouslySetInnerHTML (partly addressed). WindIndicator now uses textContent. Others (MapElementResponsePanel, AssetPalette, EvacuationZone, ScenarioDetailView) still use it but only with build-time SVG/emoji — keep user/AI-supplied HTML out of these sinks; sanitize with DOMPurify if that ever changes.

Confirmed fixed since June (context)

  • Privilege-escalation chain in role assignment — closed at DB (migrations 189/190), API (PATCH /profile rejects role/agency from non-admins), and client (signup role picker removed, role resolved from server).
  • H2 — Channel messages IDOR — now gated by assertSessionAccess().
  • M2 — Debug AI endpoints — router now behind requireAuth, requireStaff (and front-end /debug routes are role-gated).
  • Frontend dependency advisories — react-router-dom / socket.io-client bumped; frontend npm audit is clean.
  • PostgREST .or() injection in socialMessenger — closed via handle denylisting.
  • Defense-in-depth — CSP report-only + X-Content-Type-Options + Referrer-Policy; shared server/lib/access.ts guards applied across many routes.

Suggested remediation order

  1. C1 — one-line assertSessionAccess on the voice-calls route (critical exposure, trivial fix).
  2. H1, H3 — wire the existing guards + UUID validation into the two remaining voice/objectives routes.
  3. M1 — fix the error-handler arity.
  4. N1 — conditional status flip + Stripe idempotency key on payout release.
  5. H4 — upgrade nodemailer/uuid; commit a backend lockfile.
  6. M3, M4, M5 — stop leaking tokens/DB errors; harden the rate-limit key.
  7. N2, L1–L5 — defense-in-depth.
