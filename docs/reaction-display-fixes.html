Page Logo + Reaction Display Fixes

Three issues seen when a player who does not control the "Marina Bay Sands" page reacted to a post made by that page. Below: what each bug looks like now, why it happens, the change, and the result.

Bug 1 - Page photo shows a letter Frontend

A non-controller sees the page's post but the profile photo is missing - just the letter "M".

Before

M
Marina Bay Sands ✓ 1m
We are aware of the incident and are cooperating with authorities.

After

MBS
Marina Bay Sands ✓ 1m
We are aware of the incident and are cooperating with authorities.
6 comments
Why: the feed renders a page post's avatar from orgPageInfo, which now loads from /api/social/my-page and is null for anyone who does not control that page. So every page post falls back to the initial letter.
Fix: render the avatar from the post's page, not the viewer's. Load a per-handle logo map from the all-pages endpoint and key it by the post author handle. 
// new: logos for every page in the session, keyed by handle
+ const [orgPageLogos, setOrgPageLogos] = useState<Record<string,string>>({});
// GET /api/social/org-page/session/:id  -> map page_handle -> page_logo_url

// post avatar render
- post.author_type === 'official_account' && orgPageInfo?.page_logo_url
+ post.author_type === 'official_account' && orgPageLogos[post.author_handle]

Bug 2 - Notification icon is a heart, text says "haha" Frontend

The page controller's notification shows a red heart, but the words say the reaction was a "haha". Icon and text disagree.

Before

❤️
Jdoe reacted haha to your post1m

After

😂
Jdoe reacted haha to your post1m
Why: the in-app notification list hardcodes a heart SVG for every social_like notification, ignoring metadata.reaction_type. The text is generated from the real reaction label, so they don't match.
Fix: choose the icon from metadata.reaction_type using the existing reaction emoji map. 
- {notif.type === 'social_like' ? (<svg>...heart...</svg>) : ...}
+ {notif.type === 'social_like'
+   ? (REACTIONS.find(r => r.type === notif.metadata?.reaction_type)?.emoji ?? '\u{1F44D}')
+   : ...}

Bug 3 - Other viewers see "like" instead of the real reaction BackendFrontend

The reactor sees their "haha" on the post, but the page controller sees a plain like - until a full page reload.

Before (controller, live)

MBS
Marina Bay Sands ✓
We are cooperating with authorities.
👍 1

After (controller, live)

MBS
Marina Bay Sands ✓
We are cooperating with authorities.
😂 1
Why: the like endpoint updates the count but never broadcasts the reaction, and the live update handler only syncs counts (not reaction type). With a baseline like-count > 0 and no reaction types loaded, the display falls back to a generic ['like']. The data is correct on a fresh reload, so it's a live-sync gap.
Fix: broadcast the reaction type when a like happens, and merge it into the post's reaction set live.
Player reacts haha → like endpoint: insert + count++ → broadcast { id, like_count, reaction_type } → feed merges reaction_type into reaction_types → post shows haha live 
// backend: socialMedia.ts (player like)
+ getWebSocketService().broadcastToSession(post.session_id, {
+   type: 'social_posts.engagement_update',
+   data: { updates: [{ id: postId, like_count: newCount, reaction_type: reactionType }] },
+ });

// frontend: engagement_update handler
+ if (up.reaction_type) reaction_types = [...new Set([...(p.reaction_types||[]), up.reaction_type])];
