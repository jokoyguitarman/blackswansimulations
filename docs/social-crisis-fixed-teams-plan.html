Design & Security Plan

Fixed Teams for the Social Media Crisis Module

Four built-in teams — Communications, Procurement, Sales, Legal — with hardcoded charters, team-aware AI scoring, a team-grouped trainer dashboard, secure team assignment, and a player-facing briefing. This document maps every module touched, the expected UI experience, how the code works, and the security invariants that keep team assignment from breaking or leaking into other functions.

01 / Teams

The four built-in teams

Team names are a closed, hardcoded set defined in one place: server/services/teamCharterService.ts. Names never change; the AI adapts each team's tasks and storyline to the specific crisis at compile time. Charters are copied into scenario_teams rows so trainers can tweak wording per scenario without touching code (semi-flexible by design).

📣

Why a closed set matters for security: every downstream function — grading, routing, scoring, dashboard grouping — validates team names against this catalog (or the scenario's persisted scenario_teams rows). An unrecognised team name can never enter the system through the assignment API, and if legacy data contains one, every consumer falls back to safe generic behaviour instead of erroring. See Security Invariants.

03 / Flow

End-to-end data flow

From authoring to after-action review. The team dimension is injected once at compile, resolved once per assignment, and then flows read-only through routing, grading, and scoring.

3 · Team assignment routes/teams.ts validated

Trainer assigns each player to exactly one team. Server validates: trainer-only · participant exists · team_name exists in this scenario's scenario_teams · move-semantics (old assignment deleted in the same operation). Start warns if anyone is unassigned or a team is unstaffed.

5.3 · Assignment is validated, single-team, move-semantics

The existing endpoint accepts any non-empty string as a team name (team_name: z.string().min(1) in routes/teams.ts) — that free-string gap is closed here. Assignment becomes: authenticate → authorise → validate participant → validate team against the scenario → atomically replace any previous assignment.

5.4 · One resolver, used by everything

All team lookups at runtime go through one helper so fallback behaviour is defined exactly once. It can return null for three distinct reasons (no assignment, unknown team name in legacy data, scenario has no teams) — and every caller treats null as "behave like today".

server/services/teamCharterService.ts 
export async function getPlayerTeamContext(sessionId, userId): Promise<TeamContext | null> {
  const rows = await fetchSessionTeams(sessionId, userId);      // session_teams
  if (rows.length === 0) return null;                          // unassigned — safe fallback
  const row = pickDeterministic(rows);  // earliest assigned_at if legacy multi-team data exists
  const charter = await fetchScenarioCharter(sessionId, row.team_name); // scenario_teams
  if (!charter) {
    logger.warn({ sessionId, userId, team: row.team_name }, 'team has no charter — generic grading');
    return { team_name: row.team_name, charter: null };       // name kept for grouping, no rubric
  }
  return { team_name: row.team_name, charter };
}

5.6 · Routing resolves teams at publish time, not read time

Rather than teaching every GET endpoint about teams, team-scoped injects are resolved into the existing per-player targeting mechanism (target_player_ids, added in migrations 176/186 for echo-chamber targeting) when the inject fires. The already-tested read-time filter then does the work. Emails get the same treatment via a new nullable recipient_user_ids column.

5.7 · Team scores are a pure recomputable rollup

server/services/teamScoreService.ts (new) 
export async function computeTeamScores(sessionId): Promise<TeamScoreReport> {
  // inputs (all read-only): scenario_teams (charters), session_teams (membership),
  // player_actions (with team_at_action), social_posts/sim_emails (grades)
  for (const team of scenarioTeams) {
    const members = membership.get(team.team_name) ?? [];
    const tasks = matchExpectedActions(team.expected_actions, actions, members, startTime);
    // each task: done | pending | overdue | unstaffed (members.length === 0)
    const quality = avgGrades(artifactsBy(members));   // incl. role_fit
    const composite = 0.5*quality.overall + 0.35*tasks.completionScore + 0.15*quality.role_fit;
    // zero members ⇒ composite is null ("unstaffed"), never 0 — avoids fake failing grades
  }
  // players not in any team → report.unassigned (visible to trainer, excluded from team math)
}

Nothing mutates on the hot path: the score is derived from immutable event history, so a crash or a double-recompute can never corrupt it. Snapshots go to team_score_snapshots for trajectory charts. The websocket event team_scores.updated carries only numbers (never content), so the session-wide broadcast leaks nothing team-scoped.

06 / Security

Security invariants

Eight invariants guarantee that team assignment cannot open loopholes in other functions and that an unexpected team value can never crash a code path. Each is enforced server-side; the UI merely reflects them.

I1

Team names are a closed set, validated at the door

POST /teams/session/:id/assign rejects any team_name not present in the session's scenario's scenario_teams rows (400). This closes an existing gap: today the endpoint accepts any non-empty string, meaning arbitrary team labels can already be written into session_teams. After this change, an unrecognised team can only exist as legacy data — and I4 covers that.

I2

Only the session's trainer (or admin) can assign — already enforced, kept

The existing chain stays: requireAuth → role must be trainer/admin → session.trainer_id === user.id unless admin → target user must be a session_participants row. Players cannot self-assign, cannot assign others, and a trainer cannot touch another trainer's session.

I3

One team per player, atomically

For social sessions, assignment is a move: any other session_teams rows for that user are deleted in the same operation as the upsert. There is never a window where scoring or routing sees two teams. If legacy multi-team rows exist, getPlayerTeamContext picks deterministically (earliest assigned_at) and logs a warning — never throws, never picks randomly between requests.

I4

Every consumer treats team as optional — unknown or missing team can never error

All lookups flow through getPlayerTeamContext(), which returns null instead of throwing for: unassigned player, team name with no charter (legacy/unknown), or a teamless scenario. Callers' contract: null means "behave exactly as the system does today" — generic grading, universal content only, "Unassigned" bucket in the dashboard, excluded from team-score math. A player assigned to a team the functions don't recognise gets a normal, fully working session; the only thing they miss is the team-specific layer.

I5

No cross-team content leakage — filtering is server-side, resolved at publish time

Team-scoped injects are resolved to concrete target_player_ids when published, then filtered by the existing, already-exercised read-time visibility check. Clients never receive rows they shouldn't see, so no client-side filtering to bypass. The email list endpoint — which today returns all session emails to every participant — gains the same per-recipient filter. Trainer bypass is an explicit role check, not an absence of filtering. Websocket broadcasts for team scores carry numbers only; team-scoped content notifications go via the existing per-user emitToUser channel, not session-wide rooms.

I6

Team is stamped at write time — reassignment can't corrupt history

player_actions gains team_at_action, filled when the action is recorded. Scoring attributes each action to the team the player was on when they acted. If the trainer moves a player mid-session, past work stays with the old team and new work counts for the new team — no retroactive score shifts, no double counting, no orphaned actions.

I7

Score and ledger endpoints stay trainer/admin-only; players see only their own charter

GET /social/team-scores/... uses the same guard as the existing ledger endpoint (403 for non-trainer/admin). GET /social/my-team/... derives identity from the JWT — it takes no user_id parameter, so a player cannot request another player's charter, and the response strips timing benchmarks and scoring weights so the hidden rubric stays hidden.

I8

The client can never define team structure

At compile, the server ignores any client-sent team list for social crisis mode and builds the four TeamDefs from TEAM_CATALOG itself. Charter fields written to scenario_teams come from the server-side catalog (plus the server-run adaptation step). Later trainer edits to charter wording go through validated endpoints that can change descriptions but not team names, detection action types, or weights — the fields other functions key off remain server-controlled.

Pre-existing issues this feature fixes in passing: (1) team_name free-string acceptance in the assign endpoint (I1); (2) GET /social/emails/session/:id returning every email to every participant with no per-player filtering (I5). Both are hardened as part of Phase 3/6 work, with the email fix feature-flagged per session mode so field-ops behaviour is untouched.

08 / Compatibility

Backward compatibility & rollout

The feature is additive. Every schema change is nullable; every code path checks for the presence of team data before using it.

Existing scenarios

Compiled scenarios have no scenario_teams rows for social crisis. All new code branches on their presence, so old scenarios run sessions exactly as before — no backfill, no re-compile needed.

Field-ops untouched

Single-team enforcement, email filtering, and charter grading are all gated on sim_mode === 'social_media'. Field-ops teams, the heat meter, and participant roles are not modified.

In-flight sessions

Sessions running during deploy keep working: new columns are nullable, team_at_action is simply null on old rows, and the team dashboard section renders only when the scenario has teams.
