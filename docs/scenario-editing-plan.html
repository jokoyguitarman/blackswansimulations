Engineering Plan · Trainer Scenario Editing — Phase 1 (Social Media Crisis)

Warroom Generates, Trainer Personalises, The Sim Obeys

The War Room stays the fast path: a trainer builds an entire crisis scenario — NPCs, storyline injects, org pages, fact sheet — in one guided wizard. This feature adds three things on top: a rich read-only review step before compile so the trainer sees exactly what was produced, unlimited free editing after compile of everything the warroom made, and a guarantee that every edit actually reaches the running simulation — no field is editable unless something at runtime provably consumes it.

1. The full scenario lifecycle

One line to keep in mind while reading everything below: the social-crisis runtime reads almost everything live, every tick. The inject scheduler re-queries scenario_injects on every tick, and the NPC engines re-read scenarios.initial_state on every tick. That architecture is what makes post-compile editing safe — most edits flow into a not-yet-started session with zero extra machinery. The plan's job is to fix the four places where that is not true.

The agreed product rules

  • All editing happens after compile. The pre-compile review step is strictly read-only — it exists so the trainer can see what the warroom produced before committing.
  • Edits are unlimited and free. No credit is consumed by editing or by image regeneration.
  • The edit button locks when the trainer has 0 session launch credits. It unlocks automatically if credits are topped up (new invoice).
  • The edit button locks while any session on the scenario is live (in_progress or paused) and unlocks when it ends — provided the trainer still has session credits.

3. What compile actually does

Compile (server/routes/socialCrisisWarroom.ts · POST /api/warroom/social-crisis/compile) is the moment wizard state becomes a real, launchable scenario. It runs as an async job the wizard polls.

  1. 1Consume 1 scenario credit (refunded automatically if the job fails).
  2. 2Generate strategy windows and assemble the full payload (assemblePayload in server/services/socialCrisisGeneratorService.ts).
  3. 3Persist (server/services/socialCrisisPersistenceService.ts) in order: scenarios → scenario_teams → scenario_objectives → sop_definitions → scenario_injects.
  4. 4Pre-generate NPC post images in the background: walks each persona's image_prompts, calls generatePostImage, uploads to the sim-media storage bucket, and writes the URL into the matching inject's delivery_config.media_urls (matched by author handle).

Why this matters for editing

Everything above is the editable surface. Because injects are template rows and everything else is a scenario row or its initial_state, editing is plain row updates and JSON merge-patches — no re-compile, no second scenario credit, and the original warroom output is simply overwritten in place.

4. The new read-only Review step

Today, Step 7 "Review & Compile" shows only counts (X injects, Y personas, Z gates) plus a truncated narrative. The trainer compiles half-blind. Since all content already sits in wizard state, we enrich this step into a full read-through — display-only, per the product decision that editing lives post-compile.

What the enriched review shows

  • Narrative — full title, description, briefing
  • Inject timeline — every inject in chronological order, grouped per team + shared, with author handle, platform, trigger time/condition, and full content
  • NPC persona cards — name, handle, personality, bias, interests
  • Fact sheet — confirmed facts vs unconfirmed claims
  • Org page & brand roster — the org's page plus allies / competitors / antagonist
  • Objectives & convergence gates — with weights and criteria

What it deliberately does not have

  • No edit fields. A banner states: "Everything here can be edited after compile from the scenario's detail page."
  • No new endpoints. Rendered entirely from React state already present in the wizard (personas, storylineInjects, sharedInjects, factSheet, orgPage, objectives, convergenceGates, narrative).
  • No change to compile. The existing counts grid, compile button, progress log, and job polling stay as they are.

Deliberately NOT exposed in the editor

initial_state.facebook_groups and initial_state.sentiment_curve have no runtime consumer — groups are AI-generated fresh at session init and sentiment is computed live from posts. Exposing them would be the exact silent failure this plan exists to prevent: the trainer edits, saves successfully, and nothing in the game changes. They stay hidden until something actually reads them.

Edited injects are stamped generation_source = 'trainer' (already permitted by the DB CHECK constraint from migration 150 — no new migration needed), so warroom output and human overrides remain distinguishable.

The net guarantee after this plan ships

Every field the editor exposes has a verified runtime consumer, and every consumer either reads live (most engines) or is refreshed from the live scenario at session start (the fixes below). Combined with the live-session edit lock, there is no window where a successful save can silently fail to reach the game.

7. The four staleness traps & their fixes

Trap 1  sessions.current_state is snapshotted at session create

When a session is created, the entire scenario.initial_state is copied into sessions.current_state (server/routes/sessions.ts, create handler). A trainer who creates a session, then edits the scenario, then starts the session, would have the inject scheduler reading stale research_guidelines from that frozen copy.

Fix: when the session first transitions to in_progress, re-fetch the scenario and merge the scenario-derived keys of initial_state into current_state, preserving runtime-computed keys (e.g. social_state). Session start becomes the single moment content is frozen — which matches the edit lock exactly.

Trap 2  Org pages are seeded into sim_org_pages at session create

seedOrgPages writes session-scoped org page rows from initial_state.org_page when the session is created. Edits to the org page between create and start would not reach the participant-facing page.

Fix: re-run seedOrgPages from the live initial_state in the same session-start block as the Trap 1 fix (social sims only). The existing upsert behavior refreshes identity fields; branded history re-seeds correctly because the one-time guard checks the DB, and the session has no posts before start.

Trap 3  Pre-generated post images go stale when their inject is edited

Compile generates images from persona image_prompts and attaches URLs to injects via delivery_config.media_urls. Nothing ever regenerates them. Edit an inject's content or author and it fires with the new text but the old image — a content/image mismatch delivered mid-simulation.

Fix: the inject editor shows the attached image thumbnail. When content or author changes on an inject that has media, the UI flags the image as possibly stale and offers two explicit actions: Regenerate image (new endpoint, rebuilds from current content + persona via generatePostImage, replaces the URL) or Remove image (clears media_urls). Free, no credit cost, synchronous with a visible success/failure result.

Trap 4  The AAR reads strategic benchmarks from a column that is never written

The AAR doctrine-compliance section (server/services/aarSocialMediaService.ts) reads session.scenario_data.scenario.initial_state.strategic_benchmarks — but a repo-wide search shows sessions.scenario_data is never populated by anything. Benchmarks have been silently empty in every AAR since the feature was written, edited or not.

Fix: change the AAR to fetch the scenario row and read initial_state.strategic_benchmarks directly. This is a pre-existing bug fix that also makes the benchmarks editor meaningful — trainer-edited benchmarks now genuinely shape the debrief.

Known non-issue: ambientContentService keeps in-memory caches (NPC registry, player bubbles) per session. These only affect already-running sessions — which the live-session lock prevents editing anyway. The registry re-upserts personas every tick, so pre-start edits load fresh.

11. No-silent-failure policy

The explicit requirement: every edit either demonstrably reaches the game or visibly fails. Four layers enforce this:

  1. 1Field allowlisting by consumer. The editor only exposes fields with a verified runtime read path (§6). Dead fields are not editable at all.
  2. 2No swallowed database errors. Every Supabase write in the new endpoints checks the error result and returns a structured 4xx/5xx — never a 200 with a failed write.
  3. 3Visible save feedback. Every save button shows saving / saved / failed states inline (the established saveMsg convention); failures render the thrown error message, and local state only updates after a confirmed 2xx.
  4. 4Snapshot elimination. The Trap 1–4 fixes remove every known point where a successful save could still be invisible at runtime; the live-session lock removes the only remaining window (mid-run edits).

13. Phase 2 preview: Field Ops

Field ops differs in one structural way: injects, hazards, and casualties are generated inside compile (stageGenerateAndPersist in server/services/warroomService.ts), so a true pre-compile review requires splitting that function into generate (payload stashed on the wizard draft) and persist. The persistence boundary is already a single call at the end of the function, so the split is clean.

Everything else carries over from Phase 1 unchanged: the canEditScenario lock, the inject CRUD endpoints (same scenario_injects table), the detail-view editing framework, and the no-silent-failure policy. Phase 2 adds the field-ops-specific surfaces (hazards, casualties, equipment, insider knowledge) and two targeted regeneration jobs (hazard deterioration timelines / zones when a hazard's type or severity changes).
