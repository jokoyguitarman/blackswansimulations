Design + Code Walkthrough

Potent NPC Thread Interaction

Goal: make the NPCs — especially extremists — argue and agree with each other far more aggressively, reply to comments on their posts and to replies within threads, and have that happen across many threads at once. This document explains what the system already does, what is limiting it, exactly which code we change and why, with before/after snippets.

0 TL;DR — the key realization

The engines for NPC-to-NPC argument and for extremists/antagonists replying inside comment threads already exist and already run every tick. We are not building new systems — we are removing the throttles that make them quiet and polite, and turning up the aggression in their prompts.

The three confirmed requirements map to three throttles: (1) "dominate every hot thread" = they currently reply in only one thread per tick; (2) "reply in threads" = already built; (3) "more potent" = prompt + length tuning. (4) is fine on cost, so we relax the time-gates too.

1 What exists today

Everything below is driven by the engagement tick that fires every 30 seconds (INJECT_SCHEDULER_INTERVAL_MS).

🗣

NPC ↔ NPC generic threads — simulateThreadActivity

server/services/ambientContentService.ts · runs 12x per tick (6 X + 6 Facebook). Picks an active thread, feeds the AI the thread + each participant's personality/bias, and generates 3-5 new replies that take sides. Already supports replying to a reply via the @handle[reply_id] threading format.

🔥

Extremist cell replies — runHiveThreadReplies

server/services/extremistHiveService.ts · scores active threads, picks one, and posts a reply from a rotating "cell" member that baits the most recent player/official comment. Wired into the tick in injectSchedulerService.ts (line 1011).

🏢

Rival/antagonist page replies — runAntagonistThreadReplies

server/services/antagonistEngineService.ts · same shape as the hive, but speaking as hostile competitor brand pages. Wired into the tick (line 1017).

Important: a previous plan doc claimed extremists/antagonists "never reply in threads." That is now out of date — both reply engines were since implemented and are live. So this work is tuning, not new construction.

2 Why it does not feel "potent" yet

Both reply engines share the exact same self-limiting shape. Here is the hive's current selection loop:

// extremistHiveService.ts — runHiveThreadReplies (current)
const viable = candidates.filter((c) => c.score >= 2.5).slice(0, 6);
if (viable.length === 0) return;

let target = null;
for (const cand of viable) {
  const rows = /* fetch replies for this thread */;
  const last = rows[rows.length - 1];
  // Skip threads where the hive already had the last word
  if (last && EXTREMIST_HANDLES.has(last.author_handle)) continue;
  target = cand;
    break;   // <-- STOPS after ONE thread. This is the main limiter.
}

And the time-gate that keeps it quiet between passes:

// Returns minutes the hive must wait between reply passes
function requiredReplyGapMinutes(escalationRisk) {
  if (escalationRisk >= 50) return 2;
  return 4;        // up to 4 min of silence between any hive replies
}

3 Change 1 — Extremists dominate every hot thread

Backend  server/services/extremistHiveService.ts → runHiveThreadReplies

What: turn the "engage one thread then stop" loop into "engage up to N hot threads this tick," rotating which cell member speaks in each.

// AFTER — loop over multiple threads instead of break
const MAX_HIVE_THREAD_REPLIES_PER_TICK = 5;            // new tunable cap
const viable = candidates.filter((c) => c.score >= 2.5).slice(0, 10); // was .slice(0,6)

let engaged = 0;
for (const cand of viable) {
  if (engaged >= MAX_HIVE_THREAD_REPLIES_PER_TICK) break;
  const rows = /* fetch replies */;
  const last = rows[rows.length - 1];
  if (last && EXTREMIST_HANDLES.has(last.author_handle)) continue; // keep: never talk to ourselves
    await postHiveReplyToThread(cand, rows);   // extracted from the old single-shot body
    engaged++;                                // rotate cell member each iteration
}

Why: the only thing stopping extremists from being everywhere was the early break. Looping (with a cap) makes them pile into every active argument per tick. The existing "skip threads where we hold the last word" guard stays, so they never reply to themselves in a loop.

Also: relax the time-gate

function requiredReplyGapMinutes(escalationRisk) {
  return escalationRisk >= 50 ? 0 : 1;   // was 2 / 4 — near-continuous when threads are hot
}

Why: since one pass now covers many threads, the gate's job changes from "rate-limit volume" to "don't spam an idle board." A 0-1 minute floor means the hive works the feed almost every tick whenever hot threads exist.

Bookkeeping unchanged: each reply still inserts one extremist_reply session event (used for cadence + after-action review), still bumps the parent reply_count, and still broadcasts social_post.created so it renders live.

4 Change 2 — Antagonist pages do the same

Backend  server/services/antagonistEngineService.ts → runAntagonistThreadReplies

This function is structurally identical to the hive's (cadence gate at requiredAntagReplyGapMinutes, score, viable.slice(0,6), then break after one). We mirror Change 1 exactly:

  • Add MAX_ANTAG_THREAD_REPLIES_PER_TICK = 5 and loop instead of break (lines 406-423), keeping the platform filter and the "skip threads we already dominate" guard (line 417).
  • Raise the viable cap (line 399) and relax requiredAntagReplyGapMinutes (lines 299-302) to escalationRisk >= 50 ? 0 : 1.

Why separate from the hive: antagonists speak as hostile brand pages (a different identity and tone), and only on platforms where they own a page. Keeping the two engines parallel-but-separate preserves those differences while giving both the same reach.

5 Change 3 — Stronger reply prompts

Backend  server/services/extremistDoctrine.ts (buildReplyPrompt) & the inline antagonist reply prompt

Coverage makes them present; the prompt makes them potent. The extremist reply text is produced by buildReplyPrompt in server/services/extremistDoctrine.ts; the antagonist reply prompt is inline in antagonistEngineService.ts (around line 456).

What we add to the instructions:

  • Allow 2-3 sentence replies (currently terser).
  • Directly quote / dunk on the specific comment being baited.
  • "Refuse to back down, escalate across turns, rally allies, and drag bystanders in."
  • When another commenter agrees, amplify and pile on together; when they disagree, attack the argument by name.
Guardrail preserved: the hive already forces incites_violence: false and clamps content flags after generation (extremistHiveService.ts ~line 656). We keep that — "potent" means rhetorically aggressive, not calls to violence.

6 Change 4 — Generic threads: extremist emphasis + guaranteed presence

Backend  server/services/ambientContentService.ts → simulateThreadActivity

This is the engine that makes every NPC (not just the cell) argue. Two tweaks:

a) Prompt emphasis (lines ~1183-1205)

The prompt already says "TAKE SIDES." We strengthen it so extremists and high-bias personas escalate harder, allow 2-3 sentence replies, and bias the existing "70% chance an existing participant replies again" rule toward the most polarized voice in the thread.

b) Target selection (lines ~1126-1132)

// CURRENT — prefer player/official threads 60% of the time, else random
const pool = responderThreads.length > 0 && Math.random() < 0.6
  ? responderThreads : candidates;
const targetPost = pool[Math.floor(Math.random() * pool.length)];
// AFTER — also bias toward threads where an extremist/high-bias persona is present,
// and if none is, seed one extremist reply (unless the hive already works this thread).
const spicy = candidates.filter(hasHighBiasParticipant);
const pool = spicy.length > 0 && Math.random() < 0.7 ? spicy
  : (responderThreads.length > 0 && Math.random() < 0.6 ? responderThreads : candidates);
const targetPost = pool[Math.floor(Math.random() * pool.length)];

Why: this guarantees the loud voices keep showing up in generic threads too, not just in the threads the dedicated hive engine happens to pick. The "unless the hive already works this thread" check avoids double-posting with Change 1.

7 The tunable knobs

All volume/intensity lives in a handful of constants — dial these to taste.

MAX_HIVE_THREAD_REPLIES_PER_TICK = 5  — how many threads the extremist cell hits per 30s tick.
MAX_ANTAG_THREAD_REPLIES_PER_TICK = 5  — same, for rival brand pages.
requiredReplyGapMinutes / requiredAntagReplyGapMinutes = 0 (hot) / 1 (calm)  — silence floor between passes.
viable cap = 10  — how many candidate threads are considered each pass.
simulateThreadActivity call count = 12/tick today  — raise for even denser generic chatter.

8 Cost, caps & safety

  • Cost: each reply is one OpenAI call. Going from ~1 to up to 5 threads per engine per tick raises calls roughly 5x for those two engines. You confirmed this is acceptable; the per-tick caps keep it bounded even in a very busy session.
  • No runaway loops: the "skip threads where we already hold the last word" guard (kept in both engines) prevents an actor from replying to itself repeatedly.
  • No schema / frontend changes: replies already render and notify through the existing social_post.created broadcast and the @handle[reply_id] nesting format.
  • Violence guardrail retained: extremist content flags still force incites_violence: false.

9 How we verify it worked

  1. Run a social_media session and watch the server logs: confirm runHiveThreadReplies and runAntagonistThreadReplies emit multiple extremist_reply / antagonist_reply events per tick, across different thread IDs.
  2. In the feed, confirm extremist/antagonist replies appear nested in several active threads at once, are longer and more aggressive, and pile onto both sides of arguments.
  3. Confirm no actor replies to a thread where it already had the last word.
  4. Watch tick duration and OpenAI call volume; if too heavy, lower the two cap constants.
