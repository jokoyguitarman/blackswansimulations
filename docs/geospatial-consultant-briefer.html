01

What the system is supposed to be

Black Swan Simulations is an AI-enabled crisis-simulation platform for adaptive tabletop exercises, war-room scenarios and scenario-based decision-making. The geospatial layer is the physical stage on which every exercise plays out: a real-world location is turned into a usable 2-D exercise environment that downstream evacuation, traffic, hazard and responder modules act upon.

🗺️ Real-world ground truth

Take any location (lat/long or place name), pull OpenStreetMap building footprints and roads within a radius, and convert them into clean 2-D layers the simulator can stage on.

🏢 Inside the buildings

Ingest real blueprints & floor plans, organise them by floor/level/zone, georeference them onto the building outline, and digitise interior features — walls, exits, stairs, lifts, rooms.

💥 Live the crisis

Populate the map with threats, hazards and incidents; run bombing, blast-cordon and pedestrian-evacuation simulations; and score participant decisions in the war room.

The strategic point of this briefer

A surprising amount of this is already built and battle-tested in the codebase. The consultant's time is expensive — this document separates the redundant work (cut it) from the genuinely missing work (the reason to hire), so the engagement targets value, not duplication.

03

System architecture, as built

Everything is anchored in WGS84 latitude/longitude and stored as JSONB in Supabase/Postgres (no PostGIS). The map renders with react-leaflet over OpenStreetMap tiles. There are two rendering surfaces and a trainer authoring tool.

Rendering surfaces

MapView.tsx — pure geographic COP map (incidents, hazards, pins, blast zones, evac zones). Fallback when no scene config exists.

SceneCanvasView.tsx — primary hybrid: Leaflet basemap + an HTML5 canvas overlay re-projected every frame from the live map. Draws building footprint, interior walls, hazard zones, exits, stud grid, casualties, pedestrians.

SceneEditor.tsx — trainer authoring tool; hand-draws interior walls, exits, stairwells, hazards into rts_scene_configs.

Coordinate spaces

Two spaces, with conversions:

WGS84 lat/lngsource of truth
All DB tables, API and markers persist here.
Sim-space (local metres)transient
Origin = building-polygon centroid. Used for canvas render + evac engine; interior geometry (walls/exits/blast) stored as {x,y} metres, not lat/lng.

⚠️ The #1 integration hazard: coordinate ordering is inconsistent internally

building_polygon, roads & hazard zones use [lat, lng]; placed_assets uses true GeoJSON [lng, lat]; scenario_locations uses a {lat, lng} object. Any import shim must normalise this. We solve it by mandating one standard GeoJSON contract at the boundary (see §07).

export async function queryMsBuildingFootprints(
  lat: number, lng: number, radiusMeters: number,
): Promise<BuildingFootprint[]> {
  const latDelta = radiusMeters / 111320;
  const lngDelta = radiusMeters / (111320 * Math.cos((lat * Math.PI) / 180));
  const bbox = [lng - lngDelta, lat - latDelta, lng + lngDelta, lat + latDelta];
  const resp = await fetch(STAC_SEARCH_URL, {
    method: 'POST',
    body: JSON.stringify({ collections: ['ms-buildings'], bbox, limit: 10 }),
    signal: AbortSignal.timeout(15000),
  });
  // ... returns better polygons when OSM data is a crude rectangle
}

07

The delivery contract we hand the consultant

There is no GeoJSON/KML/shapefile importer or exporter anywhere in the product today — that shim is our side, net-new. To eliminate the coordinate-ordering ambiguity at the boundary, the consultant delivers one standard format and we fan it out internally.

The one instruction that matters most

Deliver a single GeoJSON FeatureCollection in EPSG:4326 using the GeoJSON standard [lng, lat] ordering. Discriminate layers with properties.layer and tag floors with properties.floor_level. Never deliver in pixel/scene coordinates — the platform derives metres itself from lat/lng.

// Target delivery format — one FeatureCollection per site, WGS84 / EPSG:4326
{
  "type": "FeatureCollection",
  "metadata": {
    "source": "OpenStreetMap + MS Building Footprints",
    "crs": "EPSG:4326",
    "input_location": { "lat": 1.2931, "lng": 103.8558 },
    "extraction_radius_m": 5000,
    "extraction_date": "2026-06-21T00:00:00Z",
    "attribution": "© OpenStreetMap contributors (ODbL)"
  },
  "features": [
    {
      "type": "Feature",
      "properties": { "layer": "building_outline", "osm_id": 123, "levels": 4 },
      "geometry": { "type": "Polygon", "coordinates": [ /* [lng,lat] rings */ ] }
    },
    {
      "type": "Feature",
      "properties": { "layer": "road", "highway": "primary" },
      "geometry": { "type": "LineString", "coordinates": [ /* [lng,lat] */ ] }
    },
    {
      "type": "Feature",
      "properties": { "layer": "floor_plan", "floor_level": "L2", "feature_type": "emergency_exit" },
      "geometry": { "type": "Point", "coordinates": [ /* [lng,lat] */ ] }
    },
    {
      "type": "Feature",
      "properties": { "layer": "threat_zone", "hazard_type": "explosion" },
      "geometry": { "type": "Polygon", "coordinates": [ /* [lng,lat] */ ] }
    }
  ]
}

Recommended trimmed scope

Re-shape the engagement around four moves, so the consultant's hours flow to genuine value instead of duplicated effort.

Cut from scope Already done

Remove the OSM/Overpass + MS-footprint extraction engine, geocoding, the threat/hazard object schema, blast/cordon rendering, the bomb-squad sweep mechanic, and the pedestrian evacuation engine. Hand the consultant these files as the baseline rather than commissioning rebuilds.

Primary deliverable Greenfield — the reason to hire

Workstream 2: the blueprint ingestion + rigorous georeferencing + discrepancy report + cross-floor vertical-connectivity pipeline. This is ~70% greenfield and the genuine value of the engagement.

Repackage, don't rebuild Cheap glue

Workstream 1: wrap the existing functions in a standalone script, standardise to a 5 km radius + GeoJSON output, and add the metadata/attribution envelope, config template and data-quality note.

Own the contract Our side

Give the consultant the GeoJSON FeatureCollection spec (§07) so his outputs import cleanly. The import shim and layer-toggle UI are our work, not his — and they double as the WS3 "unified layer framework" gap.
