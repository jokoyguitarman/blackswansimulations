01 · Why

What actually limits concurrent players

The React frontend is static and effectively unlimited. The ceiling comes from the realtime backend. The harness is designed to stress exactly these four choke points.

Socket.io server

One Node process. Every broadcast fans out to every connected client in the session — cost grows with users × message rate. Almost always the first bottleneck.

Supabase limits

Plan-tier caps on Postgres connections and Realtime channels. Heavy RLS policies make every query more expensive under load.

Hosting tier

The CPU/memory of wherever the Node server runs matters more than the code. Node pegging one core is the classic failure mode.

OpenAI rate limits

Social-mode feed engine and AI injects call OpenAI continuously. Many simultaneous sessions can hit rate limits before sockets do.

02 · Process

The test, end to end

Fully automated: you run one command, wait a few minutes, read the report. No manual session creation, no team assignment, no clicking around.

1

Provision simulated players (one-time, reused after)

Using the Supabase service-role key, the harness creates a pool of test users (loadtest-* emails) plus one trainer account, then signs each in to mint real JWTs. Every simulated player authenticates exactly like a real browser would — the server's websocket auth middleware validates each token.

2

Spin up a live session

Field ops: via POST /api/demo/start (optionally with the existing demo AI bots generating decisions, placements and chat). Social crisis: via the normal sessions API so sim_mode = 'social_media' is set and the feed engine, NPC repliers and watchdog actually run. Test users are registered as session participants.

3

Ramp up connections in stages

The harness opens Socket.io connections in waves (default 25 → 50 → 100 → 200), each joining the session room like a real player. Each stage holds for ~60 seconds while measurements accumulate before the next wave connects.

4

Measure fan-out latency with probes

Every 5 seconds the trainer account posts a chat message via the REST API, which the server broadcasts to the whole session room. The harness timestamps the send and every receive on the same clock, so latency numbers are exact — no server clock-skew. In social mode it also counts the feed engine's own broadcasts (posts, comments, likes) as delivered-vs-expected.

5

Report and tear down

A per-stage table prints to the console and a loadtest-report.json is written. The session is stopped, sockets disconnected. Test users stay behind so the next run skips provisioning.

03 · Game types

Field ops vs. social crisis: different stress profiles

Both game types ride the same Socket.io session rooms, so one harness covers both — but the traffic shape is very different, which is why the harness reports a separate capacity number for each.

Field Operations / EvacuationSocial Media Crisis
Who generates traffic Players and trainers (decisions, placements, pins, injects). Optional demo AI bots simulate this. The server itself. The feed engine continuously produces NPC posts, comments, likes and virality updates — regardless of player count.
Broadcast volume Moderate, bursty — follows human action pace. High, continuous — every feed event fans out to every connected player.
Server-side AI load Only if AI injects / demo bots are active. Constant — feed engine, NPC repliers, statement watchdog, chat surveillance all call OpenAI while the session runs.
Expected ceiling Higher player count before strain. Likely hits the ceiling sooner — both fan-out and CPU/AI pressure.
Session creation path POST /api/demo/start POST /api/sessions + start (so sim_mode and the social engines activate)

Also worth testing separately: one big session (100 players, every message fans out to 100 clients) vs. many small sessions (10 × 10 players). The harness's default is one big session — the harsher of the two.

04 · Metrics

What we measure at every stage

Four numbers tell the whole story. Latency is what players feel; the rest confirm why.

Fan-out latency

p50 / p95 / p99 time from probe sent to each socket receiving it. Healthy: well under 500ms. The knee is where p95 jumps from ~150ms to seconds.

Delivery rate

Messages received ÷ expected (sockets × probes). Anything under 100% means broadcasts are being dropped — players missing injects and chat.

Connection health

Failed connects, auth rejections, and unexpected mid-stage disconnects. Rising disconnects = the server shedding load.

HTTP errors

Probe REST calls that fail or time out (5xx/429). Signals the API side is saturating, not just the socket layer.

06 · Running it

One command, a few flags

After a one-time .env setup (the same Supabase URL and keys the server already uses), it's fire-and-forget.

# Test both game types against a local server, up to 200 players
npm run loadtest -- --users 200 --url http://localhost:3001 --gametype both

# The real capacity number: point at the deployed staging/production server
npm run loadtest -- --users 300 --url https://your-server.example.com --gametype social

# Field ops with AI bots generating realistic player activity on top
npm run loadtest -- --users 150 --gametype field --demo ai
FlagDefaultMeaning
--users100Maximum simulated players (caps the ramp)
--urlhttp://localhost:3001Server to test. Localhost = smoke test; deployed = real capacity
--gametypebothfield, social, or both (back-to-back, reported separately)
--stages25,50,100,200Ramp stages (player counts)
--stage-duration60Seconds to hold and measure at each stage
--demooffField only: scripted or ai starts the existing demo bots for realistic background activity
--scenarioautoSpecific scenario ID; otherwise auto-picked per game type
  • Prerequisites: server running (or a staging URL), .env with Supabase URL + anon key + service-role key, at least one seeded scenario per game type being tested.
  • Costs: a running social session makes OpenAI calls (feed engine); --demo ai does too. Pure field runs with --demo off cost nothing.
  • Cleanup: sessions are closed automatically; loadtest-* test users are kept and reused on the next run.
07 · Interpretation

Reading the verdict & what to do next

healthy p95 < 500 ms, 100% delivery

Players get every update essentially instantly. This stage's player count is safely within capacity.

strained p95 0.5–2 s, occasional drops

Playable but degrading — injects and chat visibly lag. Treat the previous stage as your operating ceiling.

knee p95 > 2 s, drops & disconnects

The server is shedding load. Real players here would see frozen feeds and missed events.

  • Plan events with headroom: run real sessions at ~60–70% of the measured knee.
  • If the knee is too low: in rough order of payoff — bigger hosting tier for the Node server, trimming per-broadcast DB work, Supabase plan upgrade, then horizontal scaling of the socket layer (requires a Socket.io adapter, e.g. Redis).
  • Re-run after changes: the harness is repeatable — run it after every infra or performance change and before big training events.
