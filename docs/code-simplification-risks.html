What Will Break If You Make These Changes

A companion to the Code Simplification Guide. Each refactor in that guide is safe when done correctly — but each also has a "naive way" that silently breaks something. This document is the list of traps, in plain English, with the exact fix for each.

The one-paragraph truth

A refactor is supposed to change how the code is written, not what it does. Nothing in the guide breaks the app if you (a) keep the special-case behaviors the originals had, and (b) get past the project's automated gatekeepers (TypeScript, ESLint, and the pre-commit hook). The breakage almost always comes from over-simplifying — deleting a behavior you didn't notice — or from tooling rules rejecting the new code before it ever runs. Below, "What breaks" = the symptom, "Why" = the cause, "Safe version" = exactly what to do instead.

Global gotchas — these hit every refactor

Three things in your project's setup will bite any new file or edit, regardless of which refactor you're doing. Learn these once.

G1 · New imports must end in .js will crash on start

What breaks

If you create a helper and import it as from '../lib/teams' (no extension), the server throws ERR_MODULE_NOT_FOUND on startup and the whole backend fails to boot.

Why

Your package.json has "type": "module", so the project uses real ES Modules. In that mode, import paths to your own files must include the file extension — and counter-intuitively it's .js, not .ts, even though your files are .ts. (The TypeScript compiler rewrites .ts→.js, so you write the name of the output file.) Every existing import in server/index.ts already does this, e.g. from './lib/logger.js'.

Safe version
BREAKS
import { getTeamNameMap }
  from '../lib/teams';     // no extension
import { getMembers }
  from '../lib/teams.ts';  // wrong extension
WORKS
import { getTeamNameMap }
  from '../lib/teams.js';  // .js, always

G2 · The any type will block your commit can't commit

What breaks

My example helpers in the guide used any for brevity (e.g. (c: any), fn: any, err: any). If you copy those verbatim, git commit will be rejected by the pre-commit hook, and npm run lint will fail.

Why

Your eslint.config.mjs extends typescript-eslint's recommended rules, which flag @typescript-eslint/no-explicit-any. And package.json runs lint-staged with eslint --max-warnings=0 via a Husky pre-commit hook — so even a warning stops the commit. Same goes for leaving an unused import or variable behind (the no-unused-vars rule).

Safe version — use real types instead of any
BLOCKED BY ESLINT
export const getMembers =
  (c: any): string[] =>
    Array.isArray(c?.members)
      ? c.members : [];
PASSES
// "unknown" = honest "we don't know yet",
// then we narrow it safely.
export const getMembers =
  (c: { members?: unknown }): string[] =>
    Array.isArray(c?.members)
      ? (c.members as string[]) : [];

Rule of thumb: replace any with unknown (then check the shape) or with a small inline type like { members?: unknown }. The asyncHandler wrapper is the one place a single, contained any is hard to avoid — see refactor 4 for a typed version and an eslint-disable line if needed.

G3 · A refactor that changes the response shape is a "silent" break frontend may choke

What breaks

If the JSON your backend returns changes shape (extra fields, different error text, different status code), the frontend or any saved test can break — even though the backend "works."

Why

The frontend reads specific fields (e.g. res.data.data, res.error). A refactor must return the same shape. This matters most for refactor 2 (which can add a details field) and refactor 4 (which can change a 500's body entirely). Both are covered below.

1 · Access check helper — what breaks behavior change risk

Refactor: replace inline session checks with assertSessionAccess.

Break A — You forget to capture access.session, then use session later

What breaks

Two routes in channels.ts use the session row after the access check. If you delete the inline fetch but don't keep a session variable, you get ReferenceError: session is not defined (or a TypeScript "Cannot find name 'session'" error).

Where exactly
  • GET /session/:sessionId — line 101 uses session.trainer_id to self-heal default channels: createDefaultChannels(sessionId, session.trainer_id ?? user.id).
  • GET /session/:sessionId/participants — line 358 uses session.trainer_id to add the trainer to the list.
Safe version
BREAKS — session is gone
const access =
  await assertSessionAccess(sessionId, user);
if (!access.ok)
  return res.status(access.status)
    .json({ error: access.error });

// ...later...
createDefaultChannels(sessionId,
  session.trainer_id ?? user.id); // ❌ session?
WORKS — keep the row
const access =
  await assertSessionAccess(sessionId, user);
if (!access.ok)
  return res.status(access.status)
    .json({ error: access.error });
const session = access.session!; // ✅ reuse

createDefaultChannels(sessionId,
  session.trainer_id ?? user.id);

The ! after access.session! tells TypeScript "I know this exists here" — it's safe because ok: true always comes with a session. If you need a column beyond id, trainer_id, pass it: assertSessionAccess(sessionId, user, 'id, trainer_id, status').

Break B — You lose the detailed 500 errors in the DMs route debuggability loss

What breaks

The GET /session/:sessionId/dms route (lines 124–180) currently does something special: when the database itself errors (not just "session missing"), it returns HTTP 500 with the DB message for debugging. The helper collapses every failure into a plain 404 "Session not found."

Why it matters

This isn't an app crash — it's a behavior change. If that endpoint ever fails due to a real DB problem, you'll now see a misleading "404 not found" instead of the actual error. Functionally fine for users; annoying for you when debugging.

Safe version

Acceptable trade-off for most routes — the helper's behavior is cleaner. Just know you're giving up that diagnostic. If you specifically want to keep rich errors on this one route, leave it inline, or extend the helper later to optionally return the underlying error. Don't silently assume behavior is identical.

Break C — Rolling out to all 13 files at once blast radius

The same pattern lives in 13 route files, but the variants differ (some .single(), some .maybeSingle(), some return 500). If you "find & replace" everywhere in one commit and one slips through, you could change auth behavior on a route you didn't test. Do one route, test, commit. Repeat.

2 · dbFail logger — what breaks response shape

Break A — It adds a details field where there wasn't one

What breaks

Some current 500s return only { error: 'Failed to fetch channels' } (e.g. lines 91–94, 339–341) — no details. The dbFail helper always adds details: err?.message. That (1) changes the response shape, and (2) leaks raw database messages to the client, which is a mild security smell.

Why

Your central error handler in server/index.ts deliberately hides DB messages in production (env.nodeEnv === 'production' ? 'Internal Server Error' : error.message). A blanket details field works against that intent.

Safe version — make details opt-in / dev-only
export function dbFail(
  res: Response, err: unknown, message: string,
  context: Record<string, unknown> = {},
) {
  const e = err as { code?: string; message?: string; details?: string; hint?: string };
  logger.error({ error: err, errorCode: e?.code, errorMessage: e?.message,
    errorDetails: e?.details, errorHint: e?.hint, ...context }, message);
  const body: Record<string, unknown> = { error: message };
  if (process.env.NODE_ENV !== 'production') body.details = e?.message; // dev only
  return res.status(500).json(body);
}

Note this also drops the any (uses unknown + a narrow cast) so it passes ESLint per global gotcha G2.

Break B — Blindly replacing a 500 that had special logic feature loss

What breaks

In POST /session/:sessionId/dm (lines 491–518), the error block isn't generic — it inspects error.code === '23514' (a database constraint) to return a helpful "run the migration" message. If you replace the whole block with dbFail(...), that guidance disappears and a real setup problem becomes a confusing generic 500.

server/routes/channels.ts · lines 507–513 (must NOT be lost)
if (error.code === '23514' || error.message?.includes('check constraint')) {
  return res.status(500).json({
    error: 'Direct messaging not enabled. Please run migration 017_add_direct_messaging.sql',
    details: error.message,
  });
}
Safe version

Only swap in dbFail for the plain "log and 500" blocks. Where a block has extra logic (special-case codes, custom status, custom message), keep that logic — call dbFail only as the final fallback, or leave the block alone. Read each error block before replacing it.

3 · getMembers helper — what breaks low risk

Break A — The any in the example (see G2)

Already covered: use { members?: unknown } instead of any so ESLint doesn't block the commit. Functionally this refactor is the safest of the six.

Break B — A tiny behavior difference on weird data edge case

Subtle but harmless

Old code (channel.members as string[]) || [] would pass through a non-array truthy value (e.g. if members were somehow an object). New getMembers returns [] for anything that isn't a real array. This is more correct, not less — but it's technically a behavior change, so if any code elsewhere relied on the old loose behavior (none does in channels.ts), be aware.

Don't over-apply it

One spot that looks like a member cast isn't one: the DM-existence check builds [user.id, recipient_id].sort() (line 447). That's a brand-new array, not channel.members — leave it alone.

4 · asyncHandler wrapper — what breaks highest risk

This is the refactor most likely to break things. Here's the big one first.

Break A — Your error handler is silently mis-wired pre-existing bug

What breaks

If you wrap routes in asyncHandler (which forwards errors via next(err)), the errors will not become your clean JSON 500. Instead Express falls back to its default handler, which returns an HTML error page — and the frontend, expecting JSON, breaks.

Why — the real cause is already in your code

Express only treats a middleware as an error handler if it has exactly four parameters (err, req, res, next). Your handler in server/index.ts (lines 227–237) has only three:

// server/index.ts lines 227–237 — only 3 params!
app.use((err: unknown, _req: express.Request, res: express.Response) => {
  const error = err as Error;
  logger.error({ error: error.message, stack: error.stack }, 'Request error');
  res.status(500).json({ error: /* ... */ });
});

Because it has 3 params, Express thinks it's a normal middleware, not an error handler — so next(err) skips right past it. (Today this is dormant because every route catches its own errors. The moment you adopt asyncHandler, it matters.)

Safe version — add the missing next parameter first
CURRENT (mis-registered)
app.use((err: unknown,
  _req: express.Request,
  res: express.Response) => {
  // never runs on next(err)
});
FIXED (4 params)
app.use((err: unknown,
  _req: express.Request,
  res: express.Response,
  _next: express.NextFunction) => {
  // now Express calls this
});

Even though _next is unused, it must be declared for Express to recognize the handler. The leading underscore tells ESLint "intentionally unused" so it won't complain.

Break B — You lose the per-route log message

What breaks

Each route's catch logs a specific label like 'Error in GET /channels/session/:sessionId'. A central handler logs one generic 'Request error'. You don't lose error capture, but you lose the "which route" breadcrumb unless you add it back (e.g. log req.method + req.originalUrl in the central handler).

Break C — TypeScript loses req.user type errors

What breaks

Your routes use req.user! via the custom AuthenticatedRequest type. A generic asyncHandler typed with plain Express Request can make TypeScript forget about .user, producing compile errors across many routes at once.

Safe version — type the wrapper to allow your request type
import type { Request, Response, NextFunction, RequestHandler } from 'express';

// Accepts a handler that may be async; keeps Express's typing.
export const asyncHandler =
  (fn: (req: Request, res: Response, next: NextFunction) => Promise<unknown>): RequestHandler =>
  (req, res, next) => { fn(req, res, next).catch(next); };

Inside each route you keep casting as you do today (async (req: AuthenticatedRequest, res) => ...) — that still works because AuthenticatedRequest extends Request. Because of this typing complexity, the guide rates this refactor "do last."

Break D — Routes that don't actually return their res calls

Most handlers return res.status(...) on early exits, which is good. If any route does work after sending a response (rare), removing its try/catch could expose a "headers already sent" warning. Test each wrapped route once.

5 · getTeamNameMap — what breaks low risk

Break A — Passing a non-string session_id

What breaks

In the messages route, channel.session_id is typed as unknown (it came from a loosely-typed query). If the helper's parameter is sessionId: string, TypeScript will error: "unknown is not assignable to string."

Safe version
// cast at the call site, like the existing code already does on line 583:
const userTeamMap = await getTeamNameMap(
  channel.session_id as string,
  senderIds,
);

Break B — Don't drop the surrounding guards prematurely

Why it's fine

Both call sites currently wrap the logic in if (ids.length > 0). The helper already returns an empty Map for an empty list, so you can remove those guards — but the behavior is identical either way. No breakage; just don't accidentally remove the use of the map (the loop that applies team_name onto each sender/participant).

6 · loadSession extraction — what breaks low risk

Break A — Forgetting that assertSessionOwner still needs the session row

What breaks

Both assertSessionAccess and assertSessionOwner return { ok: true, session } on success. If your extracted loadSession returns the row but you forget to pass it back out in the success object, callers that rely on access.session (refactor 1, Break A) suddenly get undefined.

Safe version — preserve the public return shape exactly
const session = await loadSession(sessionId, select);
if (!session) return { ok: false, status: 404, error: 'Session not found' };
// ...checks...
return { ok: true, session }; // ✅ keep returning the row

Break B — Error vs missing are merged

Equivalent, but know it

The original returns 404 for both a DB error and a missing row (if (error || !session)). The extracted version returns null for both, and the caller maps null→404. Same outcome. This is the lowest-risk refactor because it's one self-contained file and doesn't change the exported function signatures.

Pre-flight checklist — run before every commit

If all of these pass, you've avoided essentially every breakage in this document:

  • Did every new import end in .js? (G1)
  • Did I avoid any (used unknown + a narrow type instead)? (G2)
  • Did I delete any imports/variables that are now unused? (G2)
  • Does the response still have the same fields, status codes, and error text as before? (G3)
  • Did I keep a session variable anywhere the route uses it later? (Refactor 1)
  • Did I preserve special-case error blocks (like the 23514 migration message)? (Refactor 2)
  • If I touched asyncHandler: did I fix the 4-parameter error handler in index.ts first? (Refactor 4)
  • Did I run npm run typecheck and see zero errors?
  • Did I run npm run lint and see zero warnings?
  • Did I start the app (npm run dev) and click through the one feature I changed?
  • Did I change exactly one refactor in this commit, so a problem is easy to trace?
The two commands that catch most problems before they reach the browser
npm run typecheck   # TypeScript: catches missing vars, wrong types, .js issues
npm run lint        # ESLint: catches any, unused vars (the commit blockers)
