Editable Blueprint Review + Option A fields

Two changes that only make sense together: add five recurring concepts as first-class blueprint fields (so the things that used to land in "Unmapped" have a home), wire each to a real generation consumer (so they actually change the scenario), and make the Blueprint Review step editable (type a value or accept an AI suggestion) so the trainer can correct/extend the blueprint before it drives generation.

Option A (5 fields) Editable review No new flag · no migration No-blueprint = unchanged

1. Why these two go together

An edit only matters if something consumes the field. Today only factions (persona generation) and timeline (the live Director) are consumed; the concepts in your "Unmapped" panels (misconduct types, cross-cutting constraints, cross-stakeholder dynamics, global tone, example vignettes) have no field and no consumer. So this feature does both halves: it gives those concepts fields, routes each to a generator, and exposes them as editable. Editing without consumers would be a "I changed it and nothing happened" trap; consumers without editing would leave the trainer unable to fix a bad extraction. Together they close the loop.

5. Backend changes

5a. Fields survive extraction (merge + coverage + extraction prompt)

A field added to the schema but not to the merge gets dropped at merge time (the merge starts from an empty blueprint and only copies known fields). So three coordinated edits:

// blueprintMerge.ts -- mergeBlueprints: array fields follow the unionStrings template
merged.incident_types = unionStrings(...parts.map((p) => p.incident_types));
merged.cross_stakeholder_dynamics = unionStrings(...parts.map((p) => p.cross_stakeholder_dynamics));
merged.example_vignettes = unionStrings(...parts.map((p) => p.example_vignettes));
merged.global_tone_guidance = parts.map((p) => p.global_tone_guidance).find((v) => v.trim()) || '';

5b. Consumers (thread blueprint into the two generators that lack it)

Today only generateNPCsAndFactSheet receives the blueprint. generateUnifiedStoryline and generateConvergenceLayer need it threaded through (function param + route + wizard body). Each consumer uses a guidance helper that mirrors buildFactionGuidance -- returns an empty string when the field is empty (this is the no-regression hinge):

function buildStorylineGuidance(blueprint?: ScenarioBlueprint | null): string {
  if (!blueprint) return '';
  const parts: string[] = [];
  if (blueprint.example_vignettes.length) parts.push(`EXAMPLE DYNAMICS to emulate:\n- ${blueprint.example_vignettes.join('\n- ')}`);
  if (blueprint.global_tone_guidance.trim()) parts.push(`GLOBAL TONE: ${blueprint.global_tone_guidance}`);
  return parts.length ? `\n\n${parts.join('\n\n')}` : '';   // empty -> prompt unchanged
}

6. The editable review UI

Mostly frontend, because the POST path already carries the blueprint. The work:

  1. Widen the frontend blueprint state. Today BlueprintView is a partial display subset; carry the full blueprint so edits round-trip to generation/compile.
  2. Editable inputs bound to setBlueprint for the consumed fields (factions, timeline, the 5 new fields, global tone). Keep unmapped_directives read-only.
  3. Accept-suggestion: each Gap Report warning with a suggested_fix gets an "Apply" button that writes the suggestion into the target field -- the same state mutation as typing.
  4. Live-vs-reference badges: mark each field drives generation or reference only so trainers know what actually changes the scenario.
  5. Send the edits: add blueprint to the generateStoryline + generateConvergence request bodies (NPC + compile already send it); the server re-validates with coerceBlueprint.
Accept-suggestion and typing are the same action -- both just mutate the blueprint state before it's sent. So the AI proposes, the human approves or overrides; the AI never silently rewrites the blueprint.

7. Safety, gating & the traps

No-regression hinge: every guidance helper returns '' when its field/blueprint is empty, so an empty or no-document blueprint produces byte-identical prompts to today. Enforced by a test.
  • No new flag -- rides under enableDocumentBlueprint (already on in prod). No migration (jsonb; whole blueprint persists in initial_state).
  • Trap #1: forgetting to extend mergeBlueprints/scoreCoverage -> per-chunk values silently dropped. Done in the same change.
  • Trap #2: wiring a consumer without threading the blueprint into the storyline/convergence generators (+ routes + wizard) -> edit has no effect. Covered in Phase B.
  • Scoring: constraints are context-only (see caution above).

8. Tests

  • mergeBlueprints unions/merges the 5 new fields; empty -> empty (no-regression).
  • scoreCoverage includes the new fields.
  • Each build*Guidance helper returns '' for an empty blueprint, and the expected text when populated.
  • coerceBlueprint round-trips the 5 fields.
  • Manual: edit a field in review -> proceed -> the generated scenario reflects the edit; a no-document run is unchanged.

9. Worked example: the Post-Terrorism Communications Crisis doc

Concretely, here is how this document (the interfaith/terrorism firestorm) flows through the editable review with the five new fields. Useful contrast: unlike the misconduct doc, this one populates some of the new fields richly and leaves others nearly empty — which is exactly where the editable step earns its keep.

What each new field holds for this document

FieldWhat this doc yieldsPre-filled?Consumer → trainee-facing effect
cross_stakeholder_dynamics "far-right mobilisation provokes left-wing counter-protests; jihadists exploit anti-Muslim backlash; media framing of religion inflames Islamophobic residents; Marxists accuse interfaith leaders of being state-co-opted" rich convergence gates + Director → live inter-faction pile-ons fire as the crisis heats up
global_tone_guidance "focus on emotions, mobilisation, mistrust; include uncertainty, contradiction, evolving info; represent extremist exploitation WITHOUT amplifying it; allow credible voices to de-escalate" rich all prompts → every NPC/agitator post carries that emotional, uncertain, internet-native register

Note the spread: cross_stakeholder_dynamics and global_tone_guidance extract richly here (the doc is explicit about them), incident_types is intentionally empty (the doc defers the specific incident to the trainer), and example_vignettes is thin. That uneven fill is the whole reason the review is editable.

The editable review experience for this doc

  1. Pre-filled, editable: the rich fields (cross-stakeholder dynamics, global tone) come populated; the trainer can tweak wording — e.g. soften the Marxist faction's "sympathetic to extremists" phrasing the Gap Report flagged.
  2. Empty, type it in: incident_types is blank, so the trainer types the specific incident here instead of leaving it to the AI — the one field this doc explicitly hands to the human.
  3. Accept-suggestion on the Gap items: the three warnings for this doc become one-click fixes:
    (1) Government/Media conflicting drivers → "Apply" rewrites them to a coherent "accurate but occasionally over-corrects under pressure." (2) Far-Right 4chan link → "Apply" swaps it for "fringe forums" (the safety layer drops it regardless). (3) Time Progression vs intensity ladder → surfaced for the trainer to reconcile stage timing.
  4. Proceed: the edited blueprint is POSTed to generate-npcs / generate-storyline / generate-convergence / compile — so every edit above actually shapes the generated scenario.
