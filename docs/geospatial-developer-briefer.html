01

Product & platform context (read first)

Black Swan Simulations is an AI-enabled crisis-simulation platform for war-room / tabletop exercises. The geospatial layer is the physical stage a scenario plays out on: a real location becomes a 2-D map that trainers author on and players act on, while evacuation, hazard, bombing and responder modules run on top of it. Three facts shape every gap below.

🧭 Everything is WGS84 lat/long on Leaflet

The map is real geographic react-leaflet over OpenStreetMap tiles (two surfaces: a plain MapView and the primary hybrid SceneCanvasView, which overlays a canvas re-projected from the live map). Persistence is Supabase/Postgres JSONB — no PostGIS. Deliverables must be in WGS84 (EPSG:4326), never pixel/scene coordinates.

📦 There is no importer yet — we define the contract

The product has no GeoJSON/KML/shapefile import or export anywhere. BSS will build the import shim; your job is to deliver into the agreed format below. This keeps the boundary clean and is why we mandate one standard format.

The delivery contract — deliver one GeoJSON FeatureCollection (EPSG:4326)

To avoid an internal coordinate-ordering trap (see below), all extracted/processed layers should be delivered as a single standard GeoJSON FeatureCollection in EPSG:4326 with the GeoJSON-standard [lng, lat] ordering, each feature carrying properties.layer (building_outline / road / floor_plan / threat_zone) and properties.floor_level where relevant. BSS's shim fans these into the internal tables.

⚠️ Why we insist on standard GeoJSON: our internal coordinate ordering is inconsistent

Inside the platform, building_polygon, roads and hazard zones use [lat, lng]; placed_assets uses true GeoJSON [lng, lat]; scenario_locations uses a {lat,lng} object. Delivering in one standard format means the ambiguity is resolved once, in our shim — not silently baked into your output.

One architectural decision BSS owns (not you)

Two un-unified "interior" models exist: scenario_floor_plans.features (geographic, multi-floor, read-only overlay) and rts_scene_configs interior arrays (sim-metre, single-floor, editable — the more "live" path). For the blueprint work, target scenario_floor_plans as the delivery shape unless BSS directs otherwise; choosing the canonical model long-term is our call.

02

The gaps

Ordered by weight. Gaps 1–4 are Workstream 2 (blueprints) and carry the engagement. Gaps 5–7 are Workstream 1 (OSM extraction) — mostly repackaging of code that already exists.

1

Blueprint / floor-plan ingestion

Workstream 2Priority: CriticalGreenfield
🎯 What we want

A trainer building a scenario at a real venue can upload that building's actual floor plans (a PDF from facilities, an image of a fire-evacuation diagram, a CAD export) and have the system turn them into usable per-floor map layers — so the exercise plays out inside the real interior of the building, not a generic box.

🧩 What we have & how it works

The storage and display side already exists. There's a multi-floor table, a floor-switcher UI, and a defined feature taxonomy (exits, stairs, lifts, rooms, corridors). What feeds them today is AI-generated or hand-drawn data — never a real uploaded plan. The plan_image_url column even exists but nothing populates it.

Builtmigrations/126_scenario_floor_plans.sqlL4–20 
CREATE TABLE IF NOT EXISTS scenario_floor_plans (
  scenario_id UUID NOT NULL REFERENCES scenarios(id),
  floor_level TEXT NOT NULL,        -- e.g. B1, G, L2
  floor_label TEXT NOT NULL,
  plan_svg TEXT,
  plan_image_url TEXT,            -- ⚠️ exists but never populated from an upload
  bounds JSONB,                  -- bbox to overlay on Leaflet
  features JSONB NOT NULL DEFAULT '[]'   -- [{id, type, label, geometry}]
);
🚫 What's lacking

The entire ingestion pipeline: there is no upload endpoint for plans, no parsing of PDF/image/CAD, and no path from a source file to populated plan_image_url + features. This is the heart of the engagement.

💡 Why it matters

The realism of every building-based scenario (an active-shooter floor sweep, a fire evacuation, a hostage room-clearance) depends on the interior being the real one. Without ingestion, trainers are stuck with invented layouts that don't match the venue they're rehearsing for — which undermines the core training value the platform sells.

🛠️ How it could work — suggested, for discussion

One possible shape:

  • An upload step that accepts PDF/PNG/JPG (CAD/DXF optionally, later), stores the raw file, and records source metadata.
  • A per-floor extraction step (manual-assisted is fine for v1) that outputs the raster/SVG overlay plus a features[] list using the existing taxonomy — populate the model, don't redefine it.
  • Output as the standard GeoJSON contract with properties.layer = "floor_plan" and properties.floor_level.

Acceptance criteria: a BSS staffer can take a real floor plan, run it through the workflow, and end up with toggleable per-floor layers that render on our existing floor-switcher with exits/stairs/rooms tagged in the existing taxonomy.

2

True georeferencing of plans to the map

Workstream 2Priority: CriticalHighest technical value
🎯 What we want

When a floor plan is placed on the map, it should sit exactly where the real building is — correct position, correct rotation, correct scale — so a player looking at the map sees the building's interior aligned with its real footprint, and a casualty pin "inside room 204" is genuinely inside room 204.

🧩 What we have & how it works

Today "alignment" is a crude bounding-box stretch: the plan SVG is squashed to fit the building's bounding box with preserveAspectRatio="none". There's no rotation, no real scaling, no control points. Feature positions are stored as normalised 0–1 fractions of that box, then linearly mapped — so any building that isn't a perfect axis-aligned rectangle is misaligned.

Crude todayfrontend/src/components/COP/FloorPlanOverlay.tsxL79–100 
// "Alignment" = stretch SVG to a bbox. No rotation / scale solver / control points.
svgEl.setAttribute('preserveAspectRatio', 'none');   // distorts to fill bbox
const overlay = new LeafletSVGOverlay(sanitisedSvg, leafletBounds, {
  opacity: 0.85, interactive: false, zIndex: 100,
});
Crude todayserver/services/floorPlanSvgService.tsL13–24 
export interface FloorFeature {
  type: string;        // emergency_exit | exit | stairs | elevator | room …
  /** ⚠️ Normalised 0-1 position within bbox — not a true coordinate */
  position_x: number;
  position_y: number;
}
🚫 What's lacking

A real georeferencing capability: control points / anchor points, scale solving, rotation and an affine (or similarity) transform that maps plan coordinates to WGS84 — plus storing those transform parameters so alignment is reproducible.

💡 Why it matters

Misalignment quietly corrupts everything downstream: evacuation routes cross walls that aren't really there, hazard/blast overlays sit over the wrong rooms, casualty and asset positions become meaningless. The map stops being a faithful stage. This is the single most valuable piece of the engagement precisely because so much depends on it being correct.

🛠️ How it could work — suggested, for discussion

One possible shape: let the user place a few control points matching plan landmarks to known map points (building corners, a marked entrance), solve a similarity/affine transform (scale + rotation + translation) from those pairs, then emit feature geometry as real lat/long via that transform — and persist the transform so it can be re-applied or refined later.

Acceptance criteria: a placed floor plan aligns to its building outline within an agreed tolerance (e.g. ≤2 m positional, ≤3° rotational), holds up on non-rectangular buildings, and the alignment can be reproduced from stored parameters without redoing it by hand.

3

Discrepancy detection: blueprint vs map outline

Workstream 2Priority: HighGreenfield
🎯 What we want

When a blueprint is ingested, the system should automatically flag where it doesn't match reality — e.g. the plan's footprint is a different scale or orientation than the building outline from the map, or floors/areas are missing — so a trainer trusts the data, or knows exactly what to fix, before running an exercise on it.

🚫 What's lacking

Nothing compares the two today. We extract building outlines from OpenStreetMap/Microsoft, and (after Gap 2) we'll have a blueprint-derived footprint — but there is no code that checks one against the other for scale, orientation, footprint shape or missing floors.

Built (the outline side only)server/services/osmVicinityService.tsL903–928 
// We already produce the map-derived building footprint to compare against:
export async function fetchBuildingFootprints(lat, lng, radiusMeters = 8000) {
  // ... Overpass query for way["building"] ...
  footprints.push({ name: tags.name, polygon: geometry.map((pt) => [pt.lat, pt.lon]) });
}
💡 Why it matters

Real blueprints are messy — wrong scale bars, north arrows off, pages missing. Without a discrepancy check, those errors get silently baked into the scenario and surface mid-exercise as nonsensical geometry. The flag turns "we hope this is right" into "we know what's right and what to review", which is what makes the deliverable trustworthy for training.

🛠️ How it could work — suggested, for discussion

One possible shape: after georeferencing, compare the blueprint-derived footprint to the map outline and produce a short discrepancy report — area ratio, orientation delta, centroid offset, and a list of floors present vs expected. Surface it as a reviewable note, not a hard block.

Acceptance criteria: ingesting a deliberately mis-scaled or rotated plan produces a clear, human-readable flag identifying the mismatch type and magnitude.

4

Cross-floor vertical connectivity

Workstream 2Priority: MediumPartial
🎯 What we want

Staircases, lifts and escalators should be understood as connections between floors, not just dots on each floor — so the system can reason about movement between levels (an evacuation down 3 floors, responders going up to floor 5), and a route can traverse the building vertically.

🧩 What we have & how it works

The data model is partly there: the RTS scene types include a Stairwell with a connectsFloors field, used in the single-floor scene editor. But in the multi-floor overlay, stairs/lifts are just independent per-floor points with no shared identity across levels — floor 2's stairwell and floor 3's stairwell aren't linked.

Built (the field exists)frontend/src/lib/rts/types.tsStairwell type 
export interface Stairwell {
  id: string;
  pos: Vec2;
  connectsFloors: [number, number];   // the linkage concept exists…
}
// …but the multi-floor overlay doesn't enforce shared IDs across floors.
🚫 What's lacking

An enforced connectivity graph: shared identifiers (or explicit links) so that a given stairwell/lift is the same object across the floors it serves, queryable as a vertical edge between levels.

💡 Why it matters

Multi-storey scenarios (evacuating a tower, clearing floor by floor) only make sense if the simulation knows how people move between floors. Disconnected per-floor dots can't support vertical routing, evacuation timing across levels, or "which exits does floor 4 actually feed into" — the exact questions a high-rise exercise asks.

🛠️ How it could work — suggested, for discussion

One possible shape: give each vertical connector a stable connector_id and a list of the floor levels it serves, so the same staircase appears with the same id on every floor it touches; emit these as connectivity records alongside the floor features.

Acceptance criteria: for a multi-floor building, you can query "what connects floor 2 to floor 3" and get the specific staircases/lifts, and a route can be expressed that changes floors through them.

03

Workstream 1 — mostly repackaging

The OSM/Microsoft extraction engine already exists and is battle-tested (Overpass querying, failover, footprint parsing, geocoding). Gaps 5–7 are about packaging that engine for repeatable, standards-compliant delivery — not rebuilding it.

5

Standalone extraction runner (CLI)

Workstream 1Priority: Low (glue)Packaging
🎯 What we want

A BSS staffer can run a single command, give it a location, and get building + road files out — repeatably, without spinning up the whole application. A self-service "extract this place" tool.

🧩 What we have & how it works

The extraction functions already exist and work — but only as internals of the running web server. They're triggered as a side-effect of app flows (e.g. generating a scenario) and write straight into the database. There is no command-line runner to invoke them on demand.

Built (as a server function)server/services/osmVicinityService.tsL495–518 
// The capability exists — it just isn't runnable standalone
export async function fetchRouteGeometries(lat, lng, radiusMeters = 6000) {
  const radius = Math.min(radiusMeters, 10000);
  const query = `[out:json][timeout:90];
    (way["highway"~"^(primary|secondary|...)"](around:${radius},${lat},${lng}););
    out body geom;`;
  const elements = await runRawOverpassQuery(query, 95000);
  // returns { name, highway_type, one_way, coordinates:[lat,lng][] }
}
🚫 What's lacking

A thin standalone script/CLI that reads a location (arg or config), calls the existing functions, and writes the result to a file on disk. The hard work is done; this is the ignition switch.

💡 Why it matters

The SOW explicitly asks for a reusable extraction tool, and BSS needs to extract new locations on demand without a developer running the app. It also makes the output auditable and re-runnable — the basis for the metadata and data-quality deliverables.

🛠️ How it could work — suggested, for discussion

One possible shape: a small Node script, e.g. node extract-osm.js --lat 1.2931 --lng 103.8558 --radius 5000 --out site.geojson, that imports the existing functions and serialises results to the GeoJSON contract.

Acceptance criteria: one command, given a location, produces importable building + road files with no app/database involvement.

6

Standardise the extraction radius to 5 km

Workstream 1Priority: Low (glue)One-line-ish
🎯 What we want

A consistent, predictable extraction footprint: when BSS asks for a location, it gets everything within a 5 km radius — the SOW's agreed standard — every time.

🧩 What we have & how it works

Radius is already a first-class parameter throughout the extractor — but defaults vary (300 m / 6 km / 8 km depending on the call), and the per-building path is hard-capped at 1 km, so a 5 km request silently gets clamped.

Capped todayserver/services/osmVicinityService.tsL571–572 
// ⚠️ a 5 km request is clamped to 1 km here
const radius = Math.min(radiusMeters ?? 300, 1000);
🚫 What's lacking

A standardised 5 km default and lifting the 1 km cap on the building path (with sensible performance limits at the larger radius).

💡 Why it matters

Consistency makes outputs comparable and predictable across sites, and matches the contract BSS agreed to. Without it, "extract this location" quietly returns different coverage depending on which path runs.

🛠️ How it could work — suggested, for discussion

One possible shape: make 5 km the runner's default, raise the building-path cap, and guard against over-fetch (result limits / tiling) since a 5 km building query is much larger than the current 1 km one.

Acceptance criteria: a 5 km request returns buildings and roads across the full 5 km, within acceptable time/size limits.

7

Metadata + attribution envelope

Workstream 1Priority: Medium (correctness + legal)Greenfield
🎯 What we want

Every extracted file should be self-describing and legally clean: drop it into the platform and the buildings land in the right place first time (no coordinate guessing), and we can show that map to clients/government without breaching OpenStreetMap's licence.

🧩 What we have & how it works

The output keeps only the centre point and radius. The coordinates are WGS84, but nothing says so, nothing records when it was pulled, and the OSM/Microsoft attribution is dropped.

Today's shape: { center, radius_meters, buildings[], route_geometries[] } — geometry but no label.

🚫 What's lacking

A small metadata block stating: CRS (EPSG:4326), extraction date, data source (OSM / Microsoft), and the required OpenStreetMap (ODbL) attribution.

💡 Why it matters
  • CRS → lands in the right place. Because our internal coordinate ordering is inconsistent, an undeclared system forces the importer to guess — and a wrong guess puts buildings in the sea or rotates the map. Declaring CRS makes import reliable.
  • Extraction date → is the map still true? OSM changes weekly; a date lets trainers judge freshness and lets us reproduce/audit a layer later.
  • Attribution → we're allowed to show it. OSM's ODbL licence legally requires "© OpenStreetMap contributors" wherever the data is shown. Shipping without it is a compliance exposure the moment we demo to a client.
🛠️ How it could work — suggested, for discussion

One possible shape — wrap the features in a metadata envelope:

{
  "metadata": {
    "source": "OpenStreetMap (Overpass) + Microsoft Building Footprints",
    "crs": "EPSG:4326",
    "input_location": { "lat": 1.2931, "lng": 103.8558 },
    "extraction_radius_m": 5000,
    "extraction_date": "2026-06-24T00:00:00Z",
    "attribution": "© OpenStreetMap contributors (ODbL)"
  },
  "type": "FeatureCollection",
  "features": [ /* … */ ]
}

Acceptance criteria: a delivered file is self-describing enough that an importer never guesses the coordinate system, and the OSM credit travels with the data.

The one-line brief

Don't rebuild the OSM/Microsoft extraction engine or the threat/evacuation/bombing simulations — those already exist and run. Spend the time on turning real blueprints into correctly-aligned, trustworthy, multi-floor map layers (Gaps 1–4), and on packaging the existing extractor into a standards-compliant, self-describing tool (Gaps 5–7). Deliver everything as the standard GeoJSON contract in §01.
