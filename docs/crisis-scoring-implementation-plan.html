Crisis Scoring Overhaul - Implementation Plan

Repair the disconnected scoring steps and add three new scored capabilities - reward fact-usage in all outgoing communications, scored reporting with a reason, and grading against the six-attribute crisis-comms framework via one consolidated standards watchdog - across the backend engine/services, DB migrations, and the trainer/player frontend.

Companion: docs/social-crisis-scoring-demo.html (the scoring reference this plan implements). Every step below cites the exact file and line range it touches.

1. At a glance: phases & risk order

Five phases, ordered so the lowest-risk, highest-value fixes land first and the score-balance-shifting work is batched and validated together.

2. Key facts that shape the plan

  • scenarios.initial_state is JSONB, saved wholesale at socialCrisisPersistenceService.ts:35 - new fields like crisis_cluster need no migration.
  • social_state lives in sessions.current_state (JSONB) - new gauges/scores (standards, speed metrics, report precision) need no migration.
  • post_reported is already allowed by the action-type CHECK (175_facebook_messenger_groups_events.sql:121-129) - but dispute_filed / dispute_upheld / dispute_rejected are NOT, so those recordPlayerAction inserts currently fail silently (the error is logged, not thrown). This directly undermines the fact-check SOP credit and must be fixed in Phase 0.
  • Latest migration is 190_*, so new migrations start at 191.
  • The persistence layer writes scenario.initial_state in one shot (socialCrisisPersistenceService.ts:25-39), so generator-side additions auto-persist.

3. Target data flow

After the overhaul, four input streams feed the deterministic engine, which still owns the headline gauges:

Player post / reply / email / dispute → contentGrader (per item, +clarity) → social_posts / sim_emails .sop_compliance_score
statementWatchdog (consolidated standards pass, ~2 min) → sessions.current_state.crisis_standards
player_actions (incl. post_reported + report validity) → computeSocialState (~30s) → social_state → Trainer dashboard meters

The watchdog writes crisis_standards every scan (decoupled from its punitive-inject cooldown); computeSocialState reads them and folds bounded contributions into the four dimensions.

2Scored reporting with a reason

Add a real "report" path that asks why, rewards correct reports of genuinely harmful content, and penalises frivolous flagging of legitimate speech. Ground truth = the post's existing content_flags.

2a. Backend - new endpoint

server/routes/socialMedia.ts - add POST /posts/:postId/report, mirroring the flag handler at 803-864:

// body: { session_id, violation_category, reason_text }
const post = await load social_posts (id, session_id, content_flags)
const valid = hasHarmfulFlags(post)         // reuse the engine's predicate
insert social_post_flags { post_id, player_id, violation_category, reason_text, is_valid_report: valid }
await recordPlayerAction(session_id, user.id, 'post_reported', postId, reason_text,
                         { violation_category, valid }, valid ? 'monitor' : null)
if (valid) markPostResponded / surfacePostToSession   // counts as "addressed"
broadcast social_post.flagged
  • Valid report (post has a harmful flag): records post_reported with valid:true, marks the post addressed (so it leaves the unattended penalty set), and via Phase 1's line 386 lowers Escalation Risk.
  • Invalid report (no harmful flag): records post_reported with valid:false, does not mark addressed, and feeds the penalty below.
  • Also extend the existing flag handler (838-848) to accept the optional reason fields.

2b. Engine - precision & penalty

server/services/socialStateUpdaterService.ts:

  • Read player_actions.metadata.valid on post_reported rows. Compute report_precision = validReports / totalReports; add both to the SocialState object (451-589).
  • Ensure invalid reports do not enter the "addressed" set used at 238-240 (only valid reports/flags should clear the unattended-hostile penalty).
  • Apply a modest moderation-accuracy penalty (e.g. small Narrative Control deduction scaled by invalid-report count, capped) so spam-flagging hurts without discouraging legitimate moderation.
  • Optional: in evaluateConsequenceTriggers (611+) fire a one-shot backlash inject ("users accuse the brand of censoring critics") when precision drops below a threshold with >N reports.

2c. Frontend - category picker

  • SocialFeedApp.tsx - replace the bare handleFlag (701-722) flow with a small modal offering categories (Hate speech, Incitement to violence, Misinformation, Organized harassment, Harmful narrative) + optional free text, then POST to /report. The dispute modal nearby (728-739) is a good structural template.
  • FacebookFeedApp.tsx - same change at the flag call (~897); category constants can sit beside the existing FORMAT options (44-56).
  • TrainerSimDashboard.tsx - add a "Report precision" tile to the Strategic Actions panel (around the existing tiers block).

4Best-practice framework watchdog

Implement the six-attribute framework by extending the existing watchdog into one consolidated standards pass (Transparency, Consistency, RDAP, Victim-centring), keeping Speed deterministic and Clarity inside the per-post grader. This is the highest-risk phase - apply and validate last.

  • Add meters - RDAP posture, Transparency, Consistency, Speed - near the Sentiment Gauges block (954-986), reusing the existing Gauge component and colour helpers.

Where NOT to over-consolidate

Keep RDAP splittable: if the single multi-task call degrades posture quality, RDAP is the one aspect to break into its own call (whole-session scope differs from the watchdog's delta-scoped intent). Do not collapse Speed into the LLM (timestamps are deterministic) or Clarity out of the per-post grader.

6. Validation & rollout

  • Phase 1 smoke test: run a session, post an official_statement, confirm the dashboard checklist flips to checked, the +10 Narrative bonus appears, and the "no statement at 20/35 min" negatives stop.
  • Phase 2: report a genuinely harmful post (Escalation Risk drops, post leaves the unattended list) and a benign post (no de-escalation, precision tile falls, optional backlash inject).
  • Phases 3-4 together: these shift strategic_ratio and Public Trust, so re-run a full sample session and sanity-check each meter, then update the worked numbers in docs/social-crisis-scoring-demo.html (sections 6b and 7) for the regained +10 and new credits.
  • Cost guardrails: the watchdog keeps its existing ~2-min interval, cooldown, and per-session post cap; only the (cheap) score write is un-throttled.
