The one idea that explains almost everything

Nearly every problem below comes from a single design choice. Understand this and the rest is easy.

The app stores its data in a database (Supabase). That database has a built-in bouncer called Row Level Security — rules like “a player can only see their own session.” But our backend server talks to the database using a master key (the “service-role key”). The master key tells the bouncer: “Let me do anything, skip the checks.”

Analogy. Think of the database as a hotel. Normally each guest’s keycard only opens their own room — that’s the bouncer (Row Level Security). Our server is staff carrying a master keycard that opens every room. That’s fine… as long as the staff checks your name against the room before opening a door for you. The bugs below are all places where the staff opened a door without checking whether it was actually your room.

Because the server holds the master key, the database will not stop a bad request on its own. The only thing standing between a user and everyone else’s data is the code in each route checking “wait, are you actually allowed to touch this?” In a lot of places, that check is missing. The attacker doesn’t need anything fancy — just a normal logged-in account and the ability to change an ID number in a request.

What “logged in” proves today

Only who you are (“yes, you’re a real user named Bob”). The middleware that runs on every request is called requireAuth — it checks your identity and nothing else.

What it should also prove

What you’re allowed to touch (“…and Bob is a member of this session, so he may read this chat”). This second check is the fix for most findings on this page.

Plain-English takeaway: “Are you logged in?” and “Are you allowed to do this specific thing?” are two different questions. The backend reliably asks the first, but often forgets the second.
01 · Broken Access Control (IDOR) High

Reading — and posting into — any session’s private chat

File: server/routes/channels.ts · The chat-message endpoints only verify membership for one-on-one DMs. For every other channel type, anyone can walk in.

Each chat channel has a number (an ID). When the app wants the messages in a channel, it asks the server: “give me the messages for channel abc-123.” The server checks “are you logged in?” — yes — and then, for a normal team/command channel, just hands over the messages. It never checks whether you belong to that session.

Analogy. It’s like a mailroom that hands you any mailbox’s contents as long as you can read the number off the front. You don’t need a key to your box — you just say a box number and they empty it onto the counter for you. Worse, they’ll also post letters into any box you name.

How the attack works

  • The attacker is a normal participant in their own session — a completely legitimate, low-privilege account.
  • Channel IDs are sequential-ish/guessable, or leak through other responses. The attacker grabs one belonging to a different session (e.g. a rival team’s command channel).
  • They change the channel ID in the request and ask for the messages. The server returns them.
  • They can also POST a message into that channel — impersonating chatter inside an exercise they were never part of.

What you’re preventing: a single curious or malicious participant from reading every team’s private comms and planting fake orders mid-exercise. The fix is one extra line per endpoint: “is this person actually in this session?”
02 · Privilege Escalation High

Joining any session as “whoever you want to be”

File: server/routes/sessions.ts · The “join a session” endpoint lets you pick your own role, and trusts whatever you type.

When you join a session, the request includes a role field — like participant. The problem: the server accepts any text you put there and saves it as your role. Nothing forces it to be “participant.” So you can declare yourself incident_commander, trainer, or legal_oversight — roles that may unlock powers and information meant for senior staff.

Analogy. Imagine signing into a building’s visitor book where you write your own job title, and security hands out badges based purely on what you wrote. Write “Fire Chief” and you get the Fire Chief’s badge — no one checks.

The validation rule literally accepts any string:

server/routes/sessions.ts — the rule for joining
const joinSessionSchema = z.object({
  params: z.object({ id: z.string().uuid() }),
  body: z.object({
    role: z.string(),   // ← ANY text is accepted and saved as your role
  }),
});
■ Attack — before the fix
attacker promotes themselves while joining
$ curl -X POST https://api.blackswan.app/api/sessions/<any-session-id>/join \
     -H "Authorization: Bearer eyJ...<my-normal-token>" \
     -d '{"role":"incident_commander"}'

HTTP/1.1 201 Created
{ "data": { "user_id": "me", "session_id": "...", "role": "incident_commander" } }
^ I'm now seated as the commander in a session I just walked into.
■ After the fix — role is restricted to a safe list
request is rejected

the fix, in code
role: z.enum(['participant', 'observer'])
       .default('participant'),

// Roles like commander/trainer are
// assigned by a trainer, never self-
// picked at join time.

Note: the app’s real join flow (the invite-link sign-up in join.ts) already does this correctly — it forces role: 'participant'. The vulnerable endpoint shown here isn’t actually called by the app anymore, so tightening it is both safe and a sensible belt-and-suspenders fix against anyone calling the API directly.

What you’re preventing: a regular trainee from giving themselves a command role (and the visibility/authority that comes with it) in any exercise.
03 · Missing Authorization (Denial of Service) High

Ending anyone’s live exercise with one request

File: server/routes/demo.ts · The “stop” endpoint forces a session to completed — with no check that you own it or are even staff.

There’s a companion to the “start demo” feature: /stop. “Start” correctly requires a trainer/admin. “Stop” forgot to. So any logged-in user can send a session ID and the server marks that session completed — freezing a live, in-progress training exercise for everyone in it.

Analogy. The fire alarm has a glass cover and a key for the “test” switch — but the “evacuate the whole building” switch is just a bare button anyone walking by can slap. Same panel, one switch protected, the other wide open.
server/routes/demo.ts — notice no role / ownership check
router.post('/stop', requireAuth, validate(stopDemoSchema), async (req, res) => {
  const { sessionId } = req.body;       // ← whatever the caller sends
  getDemoPlaybackService().stop(sessionId);
  await supabaseAdmin.from('sessions')
    .update({ status: 'completed', end_time: new Date().toISOString() })
    .eq('id', sessionId);   // ← no "are you the trainer who owns this?" check
});
■ Attack — before the fix
one request ends a live exercise
$ curl -X POST https://api.blackswan.app/api/demo/stop \
     -H "Authorization: Bearer eyJ...<any-logged-in-user>" \
     -d '{"sessionId":"the-live-exercise-everyone-is-in"}'

HTTP/1.1 200 OK   { "success": true }
^ Session is now "completed". 30 trainees just got kicked out mid-scenario.
■ After the fix — only the owning trainer (or admin) can stop it
attacker is blocked
$ curl -X POST .../api/demo/stop \
     -d '{"sessionId":"..."}'

HTTP/1.1 403 Forbidden
{ "error": "Only the session's trainer can stop it" }
the fix, in code
if (user.role !== 'admin') {
  const { data: s } = await supabaseAdmin
    .from('sessions').select('trainer_id')
    .eq('id', sessionId).single();
  if (!s || s.trainer_id !== user.id)
    return res.status(403).json({ error: ... });
}
What you’re preventing: a griefer (or a bored trainee) from shutting down everyone’s exercises at will. This is a classic “denial of service” — not stealing data, just breaking the service for everyone.
04 · Broken Access Control (IDOR) High

Editing other people’s incidents, casualties, and map assets

Files: incidents.ts, placements.ts, casualties.ts, decisions.ts · Same root cause as #1: change an ID, edit data that isn’t yours.

Several “edit” endpoints take the ID of a thing (an incident, a casualty, a map placement) and update it — without ever checking whether that thing belongs to a session you’re part of. The server finds the record by ID and writes your changes. Many of these also accept a team_name straight from the request, so you can act as another team.

Analogy. A shared spreadsheet where every row has a number, and the “edit row” button works on any number you type — including rows in other people’s private copies. You can flip a casualty’s triage status, move a rival team’s roadblock, or rewrite an incident’s severity, all from your own account.

Here’s the incident editor — it fetches by ID and updates, with no ownership check in between:

server/routes/incidents.ts — PATCH /:id
const { id } = req.params;
const { data: incident } = await supabaseAdmin
  .from('incidents').select('*').eq('id', id).single();
if (!incident) return res.status(404)...;
// <-- nothing here checks the incident belongs to YOUR session -->
await supabaseAdmin.from('incidents').update(updateData).eq('id', id);
■ Attack — before the fix
rewriting a rival session’s incident
$ curl -X PATCH https://api.blackswan.app/api/incidents/SOME-OTHER-INCIDENT-ID \
     -H "Authorization: Bearer eyJ...<my-normal-token>" \
     -d '{"severity":"low","status":"resolved","casualty_count":0}'

HTTP/1.1 200 OK
^ I just marked someone else's active mass-casualty incident as "resolved, 0 hurt".
  Their scoreboard, AI injects, and after-action report are now corrupted.
■ After the fix — check the parent session, then check the team you claim
the fix, in code — shared helpers, reused everywhere
const { data: incident } = await supabaseAdmin
  .from('incidents').select('*').eq('id', id).single();
if (!incident) return res.status(404)...;

// helpers RETURN a result — no thrown errors that a try/catch could turn into a 500
const access = await assertSessionAccess(incident.session_id, user);
if (!access.ok) return res.status(access.status).json({ error: access.error });

// VALIDATE the team you claim — don't trust it, but don't blindly override it either
const team = await assertTeamMembership(incident.session_id, user, req.body.team_name);
if (!team.ok) return res.status(403).json({ error: 'Not on that team' });

await supabaseAdmin.from('incidents').update(updateData)
  .eq('id', id).eq('session_id', incident.session_id);  // scoped to the right session

The same pattern (“load the parent session → check access → verify the team you’re claiming is really yours”) fixes the matching holes in placements (move/delete any team’s map assets), casualties (re-triage anyone’s patients), and decision execution.

Two important refinements (so we don’t break real users).
• Validate, don’t overwrite. A player can be on more than one team, and trainers are on none. So instead of forcing a team onto the request, we simply confirm the team you claim is one you actually belong to. Trainers and admins are exempt — they may legitimately act on any team.
• Social-media-crisis sessions don’t use “teams.” There, your identity is the org page / account you were assigned (manually or automatically). For those actions we check “do you control this page?” using the same idea — the page-controller records — instead of the team list.
What you’re preventing: one participant quietly sabotaging other teams’ exercises and corrupting the data the scoring and after-action reports are built on. The cure is the same everywhere — check the parent session, and verify the team (or page) you claim to act as.
05 · Filter / Query Injection High

Tricking the database by smuggling code into a name field

File: server/routes/socialMessenger.ts · This is the “SQL-injection-style” bug you asked about — the one place user text is glued directly into a database query.

Most of the database calls in this app are safe, because values are passed as separate, clearly-labelled parameters (the database treats them as data, never as commands). But in one spot, the code builds a database filter by pasting the user’s text directly into the query string. When you mix “instructions” and “user data” in the same string, a clever user can make their data look like instructions. That’s injection.

Analogy. Imagine telling an assistant out loud: “Find the message where recipient is <whatever the user typed>.” If the user’s “name” is actually “Bob” OR open every drawer in the building, and you read it aloud word-for-word, the assistant hears a brand-new instruction. The fix is to treat the whole thing as a literal name in quotes — no matter how sneaky it looks.

The vulnerable line builds the filter by string-pasting the recipient handle the user sent:

server/routes/socialMessenger.ts — POST /send
const { recipient_handle } = req.body;   // ← raw text from the user
await supabaseAdmin.from('sim_direct_messages')
  .or(`and(sender_handle.eq."${senderHandle}",recipient_handle.eq."${recipient_handle}"),...`)
  // the user's text becomes part of the FILTER GRAMMAR, not just a value

In this query language, characters like ", ,, ( and ) and words like .eq. are structure. By including them in the “handle,” an attacker can close out the intended condition and bolt on their own — for example, broadening the search to match other people’s private messages in the session.

■ Attack — before the fix
a “recipient handle” that is really a smuggled filter
$ curl -X POST https://api.blackswan.app/api/social/messenger/send \
     -H "Authorization: Bearer eyJ...<my-normal-token>" \
     -d '{"session_id":"...","content":"hi",
          "recipient_handle":"x",recipient_handle.neq."__never__"}'

# The server pastes that in, and the database now sees an extra condition
# it was never meant to: "...OR recipient_handle is not '__never__'" — i.e. MATCH ALMOST EVERYTHING.
Result: the thread lookup latches onto messages between OTHER participants.
■ After the fix — treat the handle as plain data
malformed handle is rejected
$ curl ... -d '{"recipient_handle":
   "x\",recipient_handle.neq.\"y"}'

HTTP/1.1 400 Bad Request
{ "error": "Invalid recipient handle" }
the fix, in code
// The value sits inside double-quotes in the
// filter, so the ONLY breakout characters are
// the double-quote and the backslash. Reject those.
if (/["\\]/.test(recipient_handle))
  return res.status(400)...;
Why a “block-list,” not an “allow-list.” An earlier idea was to allow only letters/numbers/dots. But real recipients include AI-generated character handles (like @angry_citizen_42) and org-page handles from the database, whose exact characters we don’t fully control — a strict allow-list could reject perfectly valid recipients. Since the handle is wrapped in quotes inside the query, blocking just the quote and backslash shuts the injection completely without turning away legitimate names.
Good news on the rest

We specifically hunted for classic SQL injection across the whole backend and the database functions. Everywhere else, values are passed safely (as parameters), and the database’s stored functions either use parameters or quote identifiers correctly. We also found no command-injection, no eval, no server-side template injection, and no attacker-controllable “server-fetches-a-URL” (SSRF) holes — every outbound request goes to a fixed, hard-coded address. This one messenger filter is the only true injection point.

What you’re preventing: a user from using a “name” field to quietly rewrite a database query and read messages that aren’t theirs. The fix: reject the two characters that can break out of the query (" and \), which closes the hole while still accepting every legitimate handle.
06 · Secrets & API Keys Clean

The secrets check: what we looked for, and why it passed

You specifically asked about exposed tokens and secret keys. We searched the entire repository — backend, frontend, config, database scripts, and docs. Here’s the result in plain English.

A “secret” is anything that proves the app is who it says it is — API keys (OpenAI, Google Maps), the email password, and especially the database master key. If any of those leak into a public place (like the code shipped to browsers, or committed into git history), an outsider could use them directly. The good news: none are exposed.

✓ The master key stays on the server

The database master key (service-role key) is only ever used in backend code. The browser app uses a separate, limited “public” key that’s safe to share — exactly as intended.

✓ No .env file is committed

The file that holds the real secrets is correctly ignored by git, so it never gets uploaded. We confirmed git is not tracking any such file.

✓ Secrets are scrubbed from logs

The logging system automatically redacts passwords, tokens, API keys and auth headers, so they don’t end up sitting in log files.

✓ No secrets hard-coded in the code

Searches for key patterns (sk-…, JWTs, bearer tokens, private keys, passworded connection strings) found only placeholders in documentation — no live values.

verification — is any secret file tracked by git?
$ git ls-files | grep -i "\.env"
(no results)     # nothing secret is committed

$ cat .gitignore
.env
.env.*
!.env.example     # real secrets ignored; only the blank template is allowed
✓ Secrets are configured via environment variables, never stored in the repo.

Three tiny housekeeping items (all Low, none are live secrets): a hard-coded password for internal demo-bot accounts in a database script, some real email addresses left in a debug script, and a development-only fallback password that’s correctly required to be set in production. Worth tidying, but none expose the system today.

Bottom line on secrets: the project follows good practice here. Nothing critical is leaking. The real work is the access-control fixes in sections 1–5.

The shared helpers that fix most of this

Rather than 30 different patches, almost every High/Medium item is solved by a small set of shared functions, called wherever a request carries an ID. They return a result (rather than throwing), so each route can cleanly turn a denial into a 403 — without an existing try/catch accidentally masking it as a 500.

the reusable guardrails (server/lib/access.ts)
assertSessionAccess(sessionId, user)   // ok if trainer/admin OR a participant; else 403/404
assertSessionOwner(sessionId, user)    // ok only for the owning trainer or an admin
assertTeamMembership(sessionId, user, teamName)  // validate the team you CLAIM (trainer/admin = any)
assertControlsOrgPage(sessionId, user, handle)   // social-crisis: do you control this page?

// Recipe for any endpoint that takes a record ID:
//   1. load the record, read its session_id
//   2. const a = await assertSessionAccess(session_id, user)
//      if (!a.ok) return res.status(a.status).json({ error: a.error })
//   3. if acting as a team/page, validate it (don't trust the body)
//   4. scope every update/delete by session_id
Designed so nothing breaks

Before writing any code we traced the real usage to make sure the new checks won’t lock out legitimate users:

  • The self-join endpoint we’re hardening isn’t even used by the app — players join through a separate invite-link flow that already assigns the correct role.
  • Team checks validate the team you claim instead of overwriting it, so multi-team players and team-less trainers keep working.
  • Social-crisis sessions use the org page you control as your identity, not a team list.
  • Scenario editing is locked to its creator (admins exempt), but viewing/listing stays open to all trainers — exactly as requested.
  • The handle check is a block-list of two characters, so AI-generated and org-page handles still go through.
