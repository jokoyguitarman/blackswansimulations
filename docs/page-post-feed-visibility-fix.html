Page-Post Feed Visibility & Notification Deep-Link Fix

Two players co-controlled one organization page. When a player posted as the page, that post never appeared in their own newsfeed (only on the page profile), and tapping a reply notification did nothing for them — yet it worked perfectly for the other page controller. This explainer walks through why, and the exact code change that fixed it.

1 The symptoms

Player 1 and Player 2 both control the same org page. Player 1 posts as the page.

BehaviorPlayer 1 (the author)Player 2 (other controller)
Page-post shows in own newsfeed No (even sorted by "most recent") Yes
Page-post shows on the org page profile Yes Yes
Clicking a reply notification scrolls to the post & thread No (click appears dead) Yes

2 Why it happened (root cause)

The feed listens for new posts over WebSocket and deliberately skips any post the current user authored, on the assumption that the user already inserted their own post optimistically the moment they composed it. "Authored by me" was detected purely by user_id === me.

The assumption is correct for personal posts composed in the feed, but breaks for page posts because:

  • The server always stamps user_id with the human who clicked — even for page posts (where author_type === 'official_account'). So a page post Player 1 makes is "owned" by Player 1.
  • Page posts are composed in a different component, OrgPageView.tsx, which has its own separate posts state. So the page-post is optimistically inserted into the page profile list — never into the newsfeed list.
The gap: For Player 1, the page-post is (a) never optimistically added to the newsfeed, and (b) skipped by the live WebSocket handler because user_id === me. So it simply never enters the newsfeed's array. Player 2 isn't the author, so the live event is not skipped for him — his feed gets the post normally.

4 The code, before the change

Identical logic lived in both feeds: frontend/src/components/SimDevice/FacebookFeedApp.tsx (Facebook) and frontend/src/components/SimDevice/SocialFeedApp.tsx (X/Twitter).

Inside the social_post.created WebSocket handler:

const newPost = evtData.post;
if (newPost.platform !== 'facebook') return;

// "Own" = anything I authored, by user_id alone.
const isOwnPost =
  currentUserIdRef.current &&
  newPost.user_id === currentUserIdRef.current;

if (newPost.reply_to_post_id) {
  // ...add reply to the thread, bump reply_count if !isOwnPost...
} else {
  if (!isOwnPost) {                          // <-- page posts by me are excluded here
    setPosts((prev) => {
      if (prev.some((p) => p.id === newPost.id)) return prev;
      return [newPost, ...prev];
    });
  }
}

Explanation: when a page post authored by the current user arrived, isOwnPost was true, so the else branch's if (!isOwnPost) was false and the post was never inserted into that user's feed. Because page posts are composed in OrgPageView (separate state), there was also no optimistic insert to fall back on. Net effect: the author's feed never contained their own page post.

5 The code, after the change

Same edit applied to both feed components.

const isOwnPost =
  currentUserIdRef.current &&
  newPost.user_id === currentUserIdRef.current;
// Posts composed as the org page live in OrgPageView's separate state, so they are
// NOT optimistically inserted into this feed. They must come through the live event
// (dedup by id guards against double inserts if composed from within this feed).
const isOwnPagePost = isOwnPost && newPost.author_type === 'official_account';

if (newPost.reply_to_post_id) {
  // ...reply branch UNCHANGED — still uses isOwnPost for reply_count dedup...
} else {
  if (!isOwnPost || isOwnPagePost) {        // <-- page posts by me now flow in
    setPosts((prev) => {
      if (prev.some((p) => p.id === newPost.id)) return prev;  // dedup safety
      return [newPost, ...prev];
    });
  }
}

How the new condition reads, case by case:

PostisOwnPostisOwnPagePost!isOwnPost || isOwnPagePostResult
My personal posttruefalsefalseSkipped (already added optimistically) — unchanged
My page posttruetruetrueInserted live (the fix)
Someone else's postfalsefalsetrueInserted — unchanged
The prev.some((p) => p.id === newPost.id) guard means that even if a page post is composed from within the feed (where it would also be optimistically inserted), the live event can never create a duplicate.

6 The regression we deliberately avoided

A simpler "fix" would have been to redefine isOwnPost itself to exclude page posts (... && author_type !== 'official_account'). That was the first attempt — but it would have broken comment counting.

The reply branch also uses isOwnPost, to avoid double-incrementing reply_count for a comment you already added optimistically: 

if (newPost.reply_to_post_id) {
  setPostReplies(/* dedup by id */);
  if (!isOwnPost) {
    setPosts((prev) => prev.map((p) =>
      p.id === pid ? { ...p, reply_count: (p.reply_count || 0) + 1 } : p));
  }
}
If isOwnPost had been made false for page comments, then commenting as the page from inside the feed would increment reply_count twice — once optimistically and once from the live event. To prevent that, the reply branch keeps the original isOwnPost semantics; only the top-level insertion gained the isOwnPagePost exception.

7 Outcome & verification

Both original symptoms are resolved by this one targeted change:

  • Newsfeed: a player's own page post now appears in their feed live, no reload needed.
  • Notifications: because the post is now rendered (with its fb-post-<id> element), the reply-notification deep-link finds it and scrolls to the comment thread — matching the behavior the co-controller always had.
  • No double counting: reply counts are unaffected; the dedup guard prevents duplicate inserts.
