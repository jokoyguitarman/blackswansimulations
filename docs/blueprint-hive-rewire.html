Blueprint Hive Persona Rewire

The extremist hive already pushes the document's grievance wedge, but it still posts as a fixed 6-bot cast and the scoring only recognizes those fixed handles. This change makes the hive post as synthetic blueprint-faction personas and teaches the live scoring to recognize them — by replacing one static handle set with a single per-session "roster" read everywhere. No new flag; a no-blueprint session stays byte-identical to today.

1. Why this is needed

After Phase 5, three things drive the live feed: timed injects, the Scenario Director (posts as document factions), and the extremist hive (a recurring agitator cell that lurks and strikes when it detects an opening). Today the hive borrows the document's grievance wedge via resolveFrame, but the accounts doing the posting are still the fixed six (@unfiltered_truth, @frontline_wire, ...). So in an Interfaith-Firestorm run, the hive speaks the right grievance but under generic handles instead of the document's Far-Right / Jihadist factions. This change closes that gap.

2. The core coupling problem

One line couples everything:

// extremistDoctrine.ts
export const EXTREMIST_HANDLES = new Set(EXTREMIST_CELL.map((p) => p.handle));

That static set is read in three places. If the hive simply posted under new handles, all three would silently break:

ReaderWhat it uses the handle set forBreakage if handles change
extremistHiveService.ts (~540, 616, 619)Avoid replying to itself, filter non-hive replies, score "keep heat on own post"Hive talks over itself / mis-scores threads
socialStateUpdaterService.ts (251, 294)Weights agitator posts 3x / counts them as designed NPCs when computing escalation & narrative controlHive posts lose their scoring weight — the firestorm stops registering
ambientContentService.ts (66)Static registration of the cellAmbient layer doesn't recognize the agitators
The real work isn't "make the hive use new personas" — it's "make all three readers agree on who the agitators are this session." That is the whole rewire.

4. The design: one roster, read everywhere

Introduce a single helper that every reader calls. It returns both the personas to post as and the handle set to score by.

flowchart LR bp[(initial_state.blueprint)] --> roster["hiveRosterFor(blueprint) -> { cell, handles }"] none[no blueprint] --> roster roster --> hive["hive: rotation + self-reply + scoring"] roster --> css["computeSocialState: 3x weighting"] roster --> amb["ambient registration"] roster -.->|no blueprint| fixed["= EXTREMIST_CELL / EXTREMIST_HANDLES (identical)"]
NEW server/services/extremistDoctrine.ts — the single source of truth
export function hiveRosterFor(blueprint?: ScenarioBlueprint | null) {
  const cell = resolveCell(blueprint);              // already falls back to EXTREMIST_CELL
  const handles = new Set(cell.map((p) => p.handle));
  return { cell, handles };
}
The safety hinge: resolveCell(null) returns EXTREMIST_CELL, so hiveRosterFor(null).handles is value-equal to today's EXTREMIST_HANDLES. That single fact is what makes the no-blueprint path identical — and it's exactly what the no-regression test asserts.

5. The code changes, file by file

A. extremistHiveService.ts — post as the roster

Both passes already load scenario.initial_state, so the blueprint is in hand. Compute the roster once, then swap the static references.

  const roster = hiveRosterFor((initialState.blueprint as ScenarioBlueprint | undefined) ?? null);
  // rotation / selection:
  const persona = EXTREMIST_CELL.find(...) || EXTREMIST_CELL[rand];
  const persona = roster.cell.find(...) || roster.cell[rand];
  // self-reply avoidance + non-hive filter + thread scoring:
  if (EXTREMIST_HANDLES.has(handle)) ...
  if (roster.handles.has(handle)) ...

The chosen persona's fields (handle, name, author_type, role, voice, primary_moves) come straight from the ExtremistPersona shape resolveCell already produces — the prompt builders and move logic are untouched.

B. socialStateUpdaterService.ts — score the roster (the critical change)

// it already loads initial_state; derive the same roster handles
  const agitatorHandles = hiveRosterFor((initialState.blueprint as ...) ?? null).handles;
  // lines 251 + 294: designed-NPC test / 3x weighting
  if (EXTREMIST_HANDLES.has(post.author_handle)) weight = 3;
  if (agitatorHandles.has(post.author_handle)) weight = 3;
This is what makes the synthetic agitators actually move the needle: without it, a Far-Right hive post would land in the feed but the escalation/narrative scores would ignore it. No-blueprint -> identical set -> scoring unchanged.

C. ambientContentService.ts — recognize the roster

Make the line-66 cell registration roster-aware so the ambient layer treats the synthetic agitators consistently. Low risk; included only if it changes observable behavior, otherwise a fast follow-up.

7. Why a no-blueprint session is provably unchanged

Chain of equality: no blueprint → resolveCell(null) returns the exact EXTREMIST_CELL → hiveRosterFor(null).cell === EXTREMIST_CELL and .handles equals the contents of EXTREMIST_HANDLES. Every reader (hive, scoring, ambient) therefore sees identical data to today. The behavior only diverges when a blueprint actually contributes hostile/amplifier factions.
Containment is unchanged: the hive's hard safety rules (no real orgs/ideologies/slogans, no calls to violence, paraphrase-only) still apply to every post, whatever persona it comes from. The synthetic handles are slugs, not real accounts.

8. Will this break anything?

Short answer: done as planned, no net breakage, and the blast radius is exactly four files (extremistDoctrine, extremistHiveService, ambientContentService, socialStateUpdaterService) plus the test. Nothing in AAR, reporting, session lifecycle, or the frontend reads EXTREMIST_HANDLES / EXTREMIST_CELL, so there is no hidden consumer to break. But there is one coupling that makes a part of the plan non-optional.

The one real risk: the scoring swap is NOT optional

The scoring "designed NPC" test decides both the 3x weighting AND whether harmful in-thread replies count at all. It is an OR of four conditions: 

// socialStateUpdaterService.ts (245-253)
const isDesignedNPCPost = (p) => (
  !!p.inject_id ||
  npcHandles.has(handle) ||           // generated personas (incl. blueprint factions)
  antagonistHandles.has(handle) ||
  EXTREMIST_HANDLES.has(handle)        // <-- must become roster.handles
);
If you swapped the hive to synthetic @<faction-slug> handles but left this on EXTREMIST_HANDLES, the hive's posts would match none of these sets — they'd drop to weight 1, and harmful in-thread hive replies would stop counting entirely (they're gated by isDesignedNPCPost). The firestorm would under-register. So Phase B (persona swap) and Phase C (scoring swap, lines 251 + 294) must ship together. The plan ties them; they cannot be split.

Good news: the Scenario Director is unaffected

That same check already includes npcHandles — the generated npc_personas handles. The Director posts as those generated faction personas, so Director posts already score correctly today and this rewire doesn't touch that path.

One cosmetic gap (not breakage): ambient registration

ambientContentService (L66) registers the fixed cell so thread continuations address them in-character. If left on the fixed cell, a blueprint session's synthetic agitator handles won't be registered, so ambient continuations addressing them could fall back to plain handles instead of in-character names. That's the Phase D polish item — a cosmetic gap, never a failure.

Small edges (handled by the plan)

  • EXTREMIST_CELL.length is used as query/slice limits (286, 500, 595...) -> becomes roster.cell.length; a 1-2 faction roster just makes the slices smaller. Harmless.
  • Slug-handle collisions (two factions producing the same slug, or colliding with a generated handle) are very unlikely and at worst share a handle — not a crash.
Bottom line: as scoped, no net breakage and no-blueprint is provably identical. The only thing that WOULD break it is shipping Phase B without Phase C — keep them atomic. Ambient (Phase D) is a cosmetic follow-up, not a risk.

9. Tests & rollout

  • No-regression (the key test): hiveRosterFor(null) / hiveRosterFor(emptyBlueprint()) — cell deep-equals EXTREMIST_CELL and handles equals EXTREMIST_HANDLES.
  • Blueprint with hostile/amplifier factions → handles are the faction slugs; count matches resolveCell.
  • Determinism — same blueprint yields the same handle set across calls (handles are deterministic slugs).
  • Manual: run one blueprint session; confirm hive posts now appear under @<faction-slug> handles AND still move escalation_risk/narrative_control (proving the scoring picked up the new handles).

Rollout: no new flag (blueprint-gated, provable no-regression); separate branch + PR; typecheck/lint/tests green; squash-merge to master. The one thing to watch is the scoring weighting source change — the equivalence test is what guards it.
