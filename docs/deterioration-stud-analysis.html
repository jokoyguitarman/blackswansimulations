Architecture Analysis

Deterioration Data Doesn't Reach the Studs

The codebase has a sophisticated fire-spread physics engine that computes stud-by-stud propagation, but it's disconnected from live gameplay. Here's the full anatomy of the gap.

The Core Problem

Two Parallel Worlds, No Bridge

The system computes fire spread across studs at time intervals (0, 1, 3, 5, 10, 15, 20, 30, 45, 60 min) — but this data is trapped as a preview artifact. Live sessions use a completely different, text-based deterioration model that spawns pins by zone, not by stud.

🐛 Format Mismatch — warroomAiService.ts:7905

Trainer-designed scene hazards get enrichment events (array of spread events) assigned to deterioration_timeline — but the runtime service expects an object with at_10min/at_20min/at_30min keys.

// What gets stored (enrichment events format): deterioration_timeline: matchedEnrichment?.events // → [{ triggerTimeSec: 0, spreadType: "fire", spreadRadius: 5, ... }] // What hazardDeteriorationService.ts expects: // → { at_10min: "string", at_20min: "string", at_30min: "string" } if (timeline.at_30min) { ... } // ← never matches events array

Consequence: session deterioration silently does nothing for trainer scene hazards
