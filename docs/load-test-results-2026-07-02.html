01 · TL;DR

The two capacity numbers

Field Operations no knee found up to 100

Latency stayed essentially flat as players quadrupled — p95 went 1.34s → 1.63s → 1.67s from 25 to 100 players, with 100% delivery and zero drops throughout. The socket fan-out layer isn't close to its limit. Safe to run 100-player field exercises today; the real ceiling is somewhere above, unmeasured.

Social Media Crisis degraded at 100

Healthy at 25 and 50 players, then p95 jumped from 1.6s to 7.3s at 100 players — a 4.5× spike. Nothing was dropped and no one disconnected, but updates would feel visibly laggy. Keep social sessions at ~50 players (or ~35 with comfortable headroom) until the bottleneck below is addressed.

Why they differ: in social mode the feed engine runs continuously on the server — AI-generated NPC posts, comments and likes, each broadcast to every player (3,508 passive events reached spectators during the 100-player stage vs 2,100 in field mode). All of that shares one CPU core with the probe broadcasts, and at 100 players the queue starts backing up.
UPDATE, 23:13 same night: the Round 1 fixes were deployed and the test re-run. The social knee at 100 players is gone — p95 dropped from 7.3s to 1.4s. See the before/after section below. The numbers above are preserved as the baseline.

One honest footnote: the warm-up blip

In the re-test, the first stage of each run (25 players) briefly breached the 2s threshold (field p95 2.03s, social p95 3.0s with 2% undelivered in one probe) and then everything got faster as load increased — the opposite of a capacity problem. This run started minutes after Render finished deploying, so the first stage absorbed cold caches and instance warm-up. Degradation that disappears as load grows is warm-up, not a knee; the report tooling now only flags degradation that persists through the highest stage.

Updated operating guidance: both game types are healthy at 100 concurrent players on the current Render Standard instance. The true ceiling for both is now above the tested range — a follow-up run with --stages 100,150,200 would locate it. Raw data: loadtest-report-prod-round1.json.

Verdict: the platform handles 1,000 concurrent players across 10 simultaneous 100-player sessions on the current Render Standard instance. Field ops: effortless, ceiling not found. Social: passes at 10 sessions with 100% delivery, but the worst-session tail (2.9s) says this is approaching the comfortable limit — plan up to ~8 concurrent 100-player social sessions for headroom, or upgrade the instance for events bigger than that. Session-count load costs roughly nothing in field mode; in social mode each extra session adds a feed engine's worth of AI and CPU work regardless of how many players are in it.

Method notes: session ramp created sessions cumulatively (2 → 5 → 10) with probes staggered across each 5s interval (one probe per session per cycle); per-session latency tracked separately so a single sick session can't hide in the average. ~120 test identities backed the 1,000 sockets (Supabase sign-in rate limits; connection count and room membership unaffected). 04 · How to read these numbers

Methodology & caveats

What a probe measures

Every 5s the trainer account posts a chat message via the production REST API; the harness timestamps the send and each of the N spectators' receive on the same clock. Latency = send → receive, so it includes ~1.2s of fixed REST overhead (two Supabase round-trips before the broadcast). Growth above that baseline is the real signal.

Real connections

Every spectator is a genuine authenticated Socket.io connection joining the session and channel rooms exactly like the browser client. Due to a Supabase sign-in rate limit, 24 distinct JWTs were minted and shared round-robin across the 100 sockets — connection count, room membership and fan-out cost are unaffected.

What wasn't measured

Write load from many players acting simultaneously (decisions, placements, posts). Spectators are passive receivers; server-generated activity (feed engine) provided the background traffic. Also: one big session was tested — ten sessions of 10 players would spread fan-out differently.

05 · Recommendations

What to do with these numbers

  • Plan events within today's envelope: field exercises up to ~100 players are safe now; keep social sessions to ~35–50 players.
  • Cheapest code win — the notification loop: every chat message inserts a notification row per participant sequentially (server/routes/channels.ts). At 100 participants that's 100 serial DB writes per message, which is why probe HTTP p95 sat at 7–12s. Batching into a single bulk insert would cut most of that and free CPU for broadcasts.
  • Social-mode headroom: the feed engine competes with broadcast fan-out for the single core. Batching its per-event broadcasts, or upgrading the Render instance (Pro, 2 CPU / 4 GB — note Node uses one core unless clustered, but the faster core and scheduler headroom still help), should push the social knee past 100. Re-run the harness after either change to confirm what it bought.
  • Find the field ceiling: when convenient, run npm run loadtest -- --users 200 --stages 100,150,200 --gametype field --url https://blackswansimulations-backend.onrender.com off-hours.
  • Re-test ritual: one command, ~10 minutes, zero manual steps — run it before every large training event and after every infra change.
