Proposed Feature · Design Spec

Echo-Chamber Post Surfacing

When a player engages with a post that is exclusive to their personal feed (a demographic echo-chamber or NPC-bubble post), the post should be promoted so the whole team can see it — enabling the team to band together, fact-check, and respond collectively. This document explains the current behavior, the proposed change, the implementation touchpoints, and the expected outcomes.

1 How feeds work today

The crisis simulation already personalizes each player's feed across three layers.

Layer Targeting field Who sees it
Shared / universal
Major breaking-news posts		 none Everyone in the session
Per-player bubble
Personalized NPC-bubble posts		 target_player_ids Only the listed player(s)
Demographic echo chamber
Content tailored to race/age/etc.		 target_demographics Only players matching the demographics

The feed query in server/routes/socialMedia.ts is the single gate that decides visibility — a post is shown only if it has no targeting, or the player matches the targeting: 

const filteredData = (data || []).filter((post) => {
  if (isTrainerOrAdmin) return true;
  if (post.target_player_ids) {
    return (post.target_player_ids).includes(user.id);
  }
  if (post.target_demographics) {
    // ...player must match every demographic key...
  }
  return true; // untargeted = visible to all
});
Key insight: visibility is driven entirely by the target_player_ids / target_demographics columns. If we clear (or override) those on a post, it instantly becomes visible to the whole team — no other code needs to change to make it appear.

2 The problem this solves

Today, when a player reacts to or comments on an echo-chamber-exclusive post, nothing changes its targeting. The post stays invisible to teammates, so:

  • A teammate cannot help fact-check a piece of misinformation only one player can see.
  • The team cannot coordinate a unified response to a damaging post circulating in one demographic bubble.
  • A dangerous narrative can slip past the team because only one member ever saw it.
The closest workaround today is the repost action, which creates a brand-new player-authored copy of the content. That works, but it's a manual "quote & share" — it does not surface the original post (with its original engagement, flags, and reply thread) to the team.

3 The proposed change

When a player performs a meaningful engagement on a targeted post, the backend promotes that post to session-wide visibility and notifies all connected clients in real time.

Before

Post has target_demographics: { race: "X" }. Only matching players ever load it. Engagement (likes, flags, replies) is invisible to everyone else.

After

On a qualifying engagement, the post's targeting is cleared and a surfaced_by / surfaced_at marker is set. A WebSocket event pushes it into every teammate's feed instantly, tagged "Surfaced by teammate".

Two implementation styles are possible:

ApproachHowTrade-off
A. Clear targeting Set target_player_ids = null and target_demographics = null on the post. Simplest — works with the existing filter as-is. Loses the record of who it originally targeted.
B. Add a surfaced flag Add a is_surfaced_to_session column; the feed filter returns true when it's set. Preserves original targeting for analytics/after-action review. Requires a migration + one filter tweak.
Recommendation: Approach B. Keeping the original targeting intact is valuable for the trainer's after-action review ("this misinformation started in the X bubble and was surfaced by Player A at T+14"), and the filter change is a single extra condition.

4 What triggers surfacing

Not every interaction should surface a post. Recommended trigger set:

ActionEndpointSurface?Why
Flag POST /posts/:id/flag Yes Deliberate "this needs attention" signal — perfect for team fact-checking.
Comment / reply POST /posts (with reply_to_post_id) Yes Player is actively engaging the content; the team needs the parent for context.
Repost POST /posts/:id/repost Yes Explicit amplification intent.
Like / react POST /posts/:id/like Optional Low-effort; could surface spoilers accidentally. Suggest leaving OFF by default.
Decision needed: should a simple like surface a post? Defaulting to flag / comment / repost (intentional acts) avoids accidental spoilers and keeps the training pressure realistic.

STEP 3

STEP 4

STEP 5

Every client inserts the post into its feed, tagged "Surfaced by teammate".

7 Implementation touchpoints

Approach B (surfaced flag). Roughly four small edits + one migration.

DB  Migration — add columns

  ADD COLUMN is_surfaced_to_session boolean DEFAULT false,
  ADD COLUMN surfaced_by uuid,
  ADD COLUMN surfaced_at timestamptz;

Backend  Feed filter — respect the flag

server/routes/socialMedia.ts  (~line 114)

const filteredData = (data || []).filter((post) => {
  if (isTrainerOrAdmin) return true;
  if (post.is_surfaced_to_session) return true; // promoted to whole team
  if (post.target_player_ids) {
    return (post.target_player_ids).includes(user.id);
  }
  if (post.target_demographics) { /* ...unchanged... */ }
  return true;
});

Backend  Shared helper — surface + broadcast

server/routes/socialMedia.ts  (new helper, reused by flag / reply / repost)

async function surfacePostToSession(postId, sessionId, userId) {
  const { data: post } = await supabaseAdmin
    .from('social_posts')
    .select('target_player_ids, target_demographics, is_surfaced_to_session')
    .eq('id', postId).single();

  // Only surface posts that were actually targeted and not already shared
  const wasTargeted = post?.target_player_ids || post?.target_demographics;
  if (!wasTargeted || post.is_surfaced_to_session) return;

  const { data: updated } = await supabaseAdmin
    .from('social_posts')
    .update({ is_surfaced_to_session: true, surfaced_by: userId, surfaced_at: new Date().toISOString() })
    .eq('id', postId).select().single();

  getWebSocketService().broadcastToSession(sessionId, {
    type: 'social_post.surfaced',
    data: { post: updated, surfaced_by: userId },
    timestamp: new Date().toISOString(),
  });
}

Then call await surfacePostToSession(postId, sessionId, user.id) inside the flag handler (~line 759), the reply branch of POST /posts (~line 310), and optionally the repost handler (~line 795).

Frontend  Handle the new event

frontend/src/components/SimDevice/SocialFeedApp.tsx & FacebookFeedApp.tsx

// In the WebSocket message handler, alongside 'social_post.created':
case 'social_post.surfaced':
  setPosts((prev) => prev.some((p) => p.id === msg.data.post.id)
    ? prev.map((p) => p.id === msg.data.post.id ? msg.data.post : p)
    : [msg.data.post, ...prev]); // insert if not already present
  break;

Optionally render a small "Surfaced by teammate" badge when post.is_surfaced_to_session is true.

8 Expected outcomes

  • Team can rally on one post. A bubble-exclusive post becomes a shared focal point the moment a player engages it.
  • Collaborative fact-checking. Misinformation that only one demographic saw can now be verified and countered by the whole team.
  • Live propagation. Teammates see the post appear without refreshing, via the existing WebSocket session broadcast.
  • Engagement carries over. Because it's the same post (not a repost copy), its flags, reply thread, and reaction counts come with it.
  • Trainer visibility preserved. With Approach B, the original targeting and a surfaced_by / surfaced_at trail remain for after-action review.
  • Minimal footprint. One migration, one filter line, one shared helper, one frontend event handler.
Edge cases to confirm: (a) a post should only surface once (the helper short-circuits if already surfaced); (b) only originally-targeted posts surface — untargeted posts are already global; (c) trainers/admins are unaffected since they already see everything.
