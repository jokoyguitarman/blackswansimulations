01 · Recap

What the load test told us

On 2 July we simulated 100 players against the live server. Field operations sailed through. The social crisis game was fine at 50 players but at 100, updates that normally arrive in about 1.5 seconds took up to 7–9 seconds. Nothing crashed and nothing was lost — things just queued up. Why?

The picture to keep in mind Your server is one worker at one desk (the Render instance has a single CPU). Every job — delivering messages to players, writing to the database, running the AI social feed — goes through that one desk. The worker is fast, but when too many jobs pile up at once, everything behind them waits in line. At 100 players in social mode, the line got long. These fixes remove the biggest unnecessary jobs from the desk.

While building the test we also discovered a bug that has been silently misbehaving in production for a while. It costs nothing extra to fix it in the same round — and fixing it makes the system more honest about failures in the future.

02 · Change 1 of 3

Stop sending 100 letters one at a time the big performance win

File: server/routes/channels.ts and server/services/notificationService.ts

Analogy When anyone sends a chat message, every other player in the session should get a little notification (the red bell dot). Today the code does this like someone mailing 100 letters by driving to the post office, mailing one letter, driving home, driving back, mailing the next letter... 100 round trips. Each "trip" is a call to our database, which takes a fraction of a second — but 100 of them, one after another, adds up to several seconds of the worker's time per chat message. The fix: write all 100 letters, drive to the post office once, and hand over the whole stack.

The evidence this matters: during the load test, the chat-message API call took 7–12 seconds to respond at 100 players — and most of that time is this exact loop. Every real chat message in a big session pays this cost today.

A bonus bug gets fixed for free

The codebase already has a "send a stack of letters at once" function — it's used elsewhere. But it has a flaw: the copy it shows players in real time is missing the notification's ID number. The notification bell in the app uses that ID to mark things as read and to avoid showing duplicates — so right now, notifications delivered this way are quietly ignored by the app until the next page refresh. Our fix upgrades this function to include the ID, which fixes that too.

03 · Change 2 of 3

The announcer who refuses to speak live bug, fixed

File: server/services/eventService.ts

Analogy When something happens in a session, the server does two jobs: announce it to all connected players (instant, what players actually feel) and write it in the record book (the database, for the after-action review timeline). The code currently does them in a strange order: the announcer refuses to say anything until the record book entry is accepted. Some time ago, a database rule change made the record book start rejecting entries of the type "chat message" — and since then, the announcer has been silently skipping those announcements in production. No error, no crash. Just… silence on that particular channel.

Why players haven't complained: chat luckily has a second announcement path (to the chat-room channel) that still works — it's how our load test measured latency. But the session-wide announcement, which other parts of the app can listen to, has been dead. The deeper value of this fix: a bookkeeping problem can never again silence the real-time layer. If the record book breaks in the future, we'll see a loud error in the logs instead of mystery silence.

04 · Change 3 of 3

Teach the record book to accept chat messages small migration

New file: migrations/190_allow_message_event_type.sql

Analogy The record book (the session_events database table) has a strict list of entry types it accepts — "decision", "inject", "incident", and so on. "Chat message" isn't on the list, which is what set off the announcer problem above. This change simply adds "message" to the approved list. It's one small database script, run the same way as the 189 migration scripts before it.

With this in place, chat messages will start appearing in the session timeline again — which the timeline code was already written to display (it has been expecting them all along). Change 2 protects us even if this list ever drifts again; change 3 restores the record-keeping itself.

05 · Honest risk assessment

What could go wrong, and the safety nets

Batched letters: one bad address managed

With 100 letters in one stack, a single invalid recipient (say, a deleted user still listed in a session) could make the post office reject the whole stack — today only that one letter would fail. Safety net: notifications are already wrapped so a failure never blocks the chat message itself; worst case is missing bell-dots for one message, visible in server logs. We'll verify with real sessions before and after deploy.

Announcer reorder: double news? low

Announcing before logging means players can briefly know about an event that failed to reach the record book. For a training simulation this is the correct trade — players seeing the event matters more than the archive row. The function's callers don't change at all; they already treat it as fire-and-forget.

The migration lowest

Adding one value to an allowed-types list can't break existing rows or code — it only permits something that was previously rejected. It follows the exact same pattern as five earlier migrations that extended this same list (044, 052, 155, 156…).

Overall confidence: ~95% nothing breaks (the batching function already exists and is used in production; we're upgrading, not inventing). The main honest uncertainty is how much faster the social game gets: the chat API time will collapse for near-certain, but the feed engine also contributes to the 100-player slowdown. That's exactly why the last step below exists — we measure instead of guess. If social mode still degrades at 100, Round 2 (batching the AI feed's announcements) is the prepared next move.
06 · Verification

How we prove it worked

1

Local check

Run the type checker, start the server locally, send chat messages in a test session, and confirm notifications arrive with working "mark as read".

2

Deploy + run the migration

Push to git (Render redeploys automatically) and run migrations/190 in the Supabase SQL editor — same routine as every previous migration.

3

Re-run the exact same load test

Same command, same stages (25 → 50 → 100 players), same time of night, both game types. Takes about 10 minutes and cleans up after itself.

4

Compare before / after

The 2 July results are the baseline. Success looks like: chat API time drops from 7–12s to ~1–2s, and the social game's 100-player latency spike (7.3s) shrinks — ideally under the 2s health threshold. The results page gets a before/after section either way.
